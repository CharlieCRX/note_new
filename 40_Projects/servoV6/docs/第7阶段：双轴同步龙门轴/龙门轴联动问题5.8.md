# 龙门轴联动问题5.8

## 问题背景

5月8号在复盘龙门轴逻辑的时候，发现了很多问题：

1. QML结构不可控了
2. 不知道如何调整才能让龙门轴联动

今天就弄懂一个点，`servoV6`进程如何才能让X1、X2联动。

## TDD测试看如何联动1

根据《双轴龙门系统（Gantry）Domain 层 TDD 架构设计文档》，可以阅读联动建立约束测试 (test_gantry_coupling.cpp)来确定如何建立约束测试。

根据`CouplingRequestedEventEmitted`提供的联动成功案例，我们可以确定联动满足的条件为：

1. X1X2均使能
2. X1X2均未报警
3. X1X2均未超限
4. X1X2镜像同步

只要满足这四个条件，就有联动的充分条件。

区别在于，TDD测试是脱离FakePLC的，是直接使用了X1X2物理轴的“便捷状态字段更新”方法

```C++
    void setEnabled(bool v) { m_state.enabled = v; }
    void setAlarmed(bool v) { m_state.alarmed = v; }
    void setPosLimitActive(bool v) { m_state.posLimitActive = v; }
    void setNegLimitActive(bool v) { m_state.negLimitActive = v; }
    void setPosition(double v) { m_state.position = v; }
```

## TDD测试看如何联动2

但是主程序是使用FakePLC进行轴状态更新的，那么是否是轴状态更新出了问题？我们看下龙门轴使用FakePLC进行状态更新的TDD测试案例 `CoupledJogThenDeviationFault`（位于 test_gantry_full_integration.cpp）。

```C++
struct ScanResult {
    AxisState aggregatedState;
    std::vector<GantryEvents::Event> events;
};

static ScanResult doScanCycle(GantrySystem& gantry,
                               FakeGantryFeedbackPort& feedback) {
    // Step 1: 从 "HAL" 读取反馈 → 同步到 GantrySystem
    gantry.x1().syncState(feedback.getX1Feedback());
    gantry.x2().syncState(feedback.getX2Feedback());

    // Step 2: 状态聚合
    gantry.aggregateState();

    // Step 3: 排空事件
    ScanResult r;
    r.aggregatedState = gantry.aggregatedState();
    r.events = gantry.drainEvents();
    return r;
}
```

`doScanCycle()`的逻辑已经按照注释写的很清晰了：

1. X1X2从 HAL 层中读取到对应轴的状态信息

2. 龙门系统聚合根`GantrySystem`根据获取的信息，统一更新聚合状态（`aggregateState()`）

   系统反应是**即时且自动的**：

   ```
   aggregateState()
   ├── Coupled 模式?
   │   ├── 是 → 检查 |X1 + X2| <= epsilon?
   │   │   ├── 是 → 保持 Coupled
   │   │   └── 否 → 强制 Decoupled
   │   │       └── 发布 DeviationFault + Decoupled 事件
   │   └── 否 → 跳过同步检查
   └── 聚合物理状态到逻辑轴
   ```

3. 提取聚合时候产生的事件并清理事件

理解了`doScanCycle()`的逻辑后，我们查看`FullScanCycle_EventEmission`提供的语义约束案例：

```C++
TEST(GantryFullIntegration, FullScanCycle_EventEmission) {
    FakePLC plc;
    FakeGantryFeedbackPort feedback(plc);
    FakeGantryCommandPort command(plc);
    FakeGantryEventBus eventBus;

    PhysicalAxis x1(AxisId::X1);
    PhysicalAxis x2(AxisId::X2);
    GantrySystem gantry(x1, x2);

    plc.forceState(AxisId::X1, AxisState::Idle);
    plc.forceState(AxisId::X2, AxisState::Idle);
    plc.setPosition(AxisId::X1, 0.0);
    plc.setPosition(AxisId::X2, 0.0);

    // 首周期: 无事件
    auto r1 = doScanCycle(gantry, feedback);
    // 初始状态不应产生异常事件

    // 建立联动
    gantry.requestCoupling();
    bool hasCoupled = false;
    for (auto& e : gantry.drainEvents()) {
        if (e.type == GantryEvents::Type::Coupled) hasCoupled = true;
    }
    EXPECT_TRUE(hasCoupled);
}
```

可以看出当X1X2状态满足建立联动条件后：

```C++
    plc.forceState(AxisId::X1, AxisState::Idle);
    plc.forceState(AxisId::X2, AxisState::Idle);
    plc.setPosition(AxisId::X1, 0.0);
    plc.setPosition(AxisId::X2, 0.0);
```

因为聚合根的`GantryMode`默认为`Decoupled`，此时进行`doScanCycle()`扫描会跳过同步检查，只会更新一些状态信息。

因为聚合根已经更新了当前状态，所以此时执行联动命令，会执行成功并且相继发送两个信息：

- couplingRequested()
- coupled()

所以此时就会获取到联动成功的状态。

## servoV6当前情况

可以看出，只要X1X2状态满足`requestCoupling()`的要求，就能联动。

在当前的 main.cpp 中：

1. 首先初始化龙门物理轴状态为可联动的状态。

   ```c++
   // 初始化龙门物理轴状态
   plc.forceState(AxisId::X1, AxisState::Idle);
   plc.forceState(AxisId::X2, AxisState::Idle);
   plc.setSimulatedJogVelocity(AxisId::X1, 15.0);
   plc.setSimulatedJogVelocity(AxisId::X2, 15.0);
   plc.setSimulatedMoveVelocity(AxisId::X1, 40.0);
   plc.setSimulatedMoveVelocity(AxisId::X2, 40.0);
   plc.setLimits(AxisId::X1, 800.0, -800.0);
   plc.setLimits(AxisId::X2, 800.0, -800.0);
   plc.setPosition(AxisId::X1, 0);
   plc.setPosition(AxisId::X2, 0);
   ```

2. 系统物理时钟中，从 FakePLC 获取X1X2状态：

   ```C++
   auto x1Fb = gantryFbPort.getX1Feedback();
   auto x2Fb = gantryFbPort.getX2Feedback();
   physX1.syncState(x1Fb);
   physX2.syncState(x2Fb);
   ```

3. `qtGantryAVM.tick()`函数会最终调用`GantryViewModelCore::tick()`执行状态聚合

那么理论上，应该也跟测试`FullScanCycle_EventEmission`类似，接收到联动后，能够正常联动。但是为什么没有成功呢？

### A. 状态更新的“时序断层”

在 `main.cpp` 的 `physX1.syncState(x1Fb)` 之后，虽然物理轴对象更新了，但 `GantrySystem` 聚合根是否执行了 `aggregateState()`？

这个就可以通过单步调试来确认了。单步运行到以下代码处：

```C++
auto x1Fb = gantryFbPort.getX1Feedback();
auto x2Fb = gantryFbPort.getX2Feedback();
physX1.syncState(x1Fb);
physX2.syncState(x2Fb);
```

此时查看`physX1`和`physX2`的物理轴状态，发现其中的变量`enabled`均为`true`。

所以可以得知，我们从FakePLC设置的X1X2初始数据，是能够获取到的。

并且跟着断点向前执行，可以看到执行了：

```C++
qtGantryAVM.tick();
```

拆开后查看

```C++
void QtGantryViewModel::tick() {
    m_core->tick();

    // 节流发射信号：检测变更
    bool coupled = m_core->isCoupled();
    if (coupled != m_lastCoupled) {
        m_lastCoupled = coupled;
        emit coupledChanged();
    }
...
```

其中`m_core->tick()`代表的就是：

```C++
void GantryViewModelCore::tick() {
    // 1. 执行状态聚合
    m_gantry.aggregateState();

    // 2. 检测状态变更并写日志
    detectAndLogStateChanges();
    ...
```

也可以看到确实执行了`aggregateState()`的，并且执行了

```c++
aggState = AxisState::Idle;
```

所以我能保证：在 `main.cpp` 循环中，确实执行了聚合逻辑。

### A反. 更新错对象了

因为确认了状态没问题后，我转向了 GantryControlBlock.qml ，查看 UI 到 ViewModel 的管线是否是通的。

加入了一个新的按钮：

```html
IndustrialButton {
    text: root.gantryVM && root.gantryVM.coupled ? "解除联动" : "建立联动"
    onClicked: {
        console.log("联动按钮被点击！当前 gantryVM 对象是:", root.gantryVM)
        if (root.gantryVM) {
            root.gantryVM.requestCoupling()
        } else {
            console.error("糟了！gantryVM 是空的，无法调用 C++！")
        }
    }
}
```

观察按下后输出的日志：

```bash
qml: 联动按钮被点击！当前 gantryVM 对象是: QtGantryViewModel(0x777affea40)
[09:06:47.277][WARN][UI][GantryVM][N/A][N/A][N/A] requestCoupling 鈫� REJECTED: X1 is not enabled
```

可以得到一个结论：

- 调试时看到 `enabled` 是 `true`，但 `GantrySystem` 却大喊 `X1 is not enabled`！

这是为什么呢？

我们再回头看下 TDD 测试函数 doScanCycle() 中，如何更新物理轴X1X2的：

```C++
gantry.x1().syncState(feedback.getX1Feedback());
gantry.x2().syncState(feedback.getX2Feedback());
```

可以看出，TDD 里是直接拿到 `gantry` 内部的 `x1` 引用去更新的。

再对比 main.cpp 中的写法：

```C++
auto x1Fb = gantryFbPort.getX1Feedback();
auto x2Fb = gantryFbPort.getX2Feedback();
physX1.syncState(x1Fb);
physX2.syncState(x2Fb);
```

当拿到了X1X2的最新数据后，这里更新的是 main.cpp 里定义的局部变量 physX1 和 physX2。

这就导致了：

1. `main.cpp` 里的 `physX1` 确实被同步成了 `Idle` (Enabled = true)。
2. 但是 `gantryA` 内部的 `m_x1` 永远停留在初始化时的默认状态（通常是 `Disabled`）。
3. 当我点击联动，`GantrySystem` 检查自己肚子里的 `m_x1`，发现它还是 Disabled，于是就无情地抛出 `REJECTED: X1 is not enabled`。

修改方案很简单，就是删除无效的更新轴状态，改为更新龙门聚合根中的轴状态即可：

```C++
auto x1Fb = gantryFbPort.getX1Feedback();
auto x2Fb = gantryFbPort.getX2Feedback();
gantryA.x1().syncState(x1Fb);
gantryA.x2().syncState(x2Fb);
```

最终解决了这个问题。
