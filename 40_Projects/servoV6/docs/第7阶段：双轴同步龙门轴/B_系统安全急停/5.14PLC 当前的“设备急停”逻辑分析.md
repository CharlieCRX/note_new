# 一、先理解 PLC 当前的“设备急停”逻辑

你现在 PLC 的：

```text
设备急停
```

本质上不是：

```text
普通业务按钮
```

而是：

# “全系统安全状态”

它具备：

```text
1. 全局作用域
2. 强制覆盖所有运动
3. 强制掉电
4. 强制清空执行动作
5. 优先级最高
```

这是非常关键的。

------

# 二、当前 PLC 实际做了什么

从你给的 PLC 逻辑看：

------

# 1️⃣ 急停控制逻辑

PLC 中：

```pascal
轴X1使能 := 使能轴X电机 AND NOT 设备急停;
轴X2使能 := 使能轴X电机 AND NOT 设备急停;

轴Y使能 := 使能轴Y电机 AND NOT 设备急停;
轴Z使能 := 使能轴Z电机 AND NOT 设备急停;
轴R使能 := 使能轴R电机 AND NOT 设备急停;
```

意味着：

------

## 当：

```text
设备急停 = TRUE
```

时：

所有轴：

```text
立即掉电
```

也就是说：

```text
设备急停
```

拥有：

# “最终电源裁决权”

------

# 2️⃣ 急停动作清空逻辑

你 PLC 里：

```pascal
轴X1点动正转 := FALSE;
轴X1点动反转 := FALSE;
...
轴RMoveDo := FALSE;
```

本质是：

# “强制清空所有执行动作”

包括：

- 点动
- 定位
- 回原点
- MoveDo
- JogDo

也就是说：

```text
设备急停
```

不仅仅是：

```text
停止
```

而是：

# “全系统动作熔断”

------

# 3️⃣ 急停状态寄存器

你还有：

```text
设备急停中
```

状态位。

这意味着：

PLC 内部已经形成：

# 安全状态机

也就是：

| 状态   | 含义         |
| ------ | ------------ |
| 非急停 | 系统允许执行 |
| 急停中 | 系统强制停机 |

------

# 三、这个逻辑本质属于什么？

非常重要。

它：

# 不是 Axis 逻辑

因为：

```text
它控制整个系统
```

------

# 不是 Gantry 逻辑

因为：

```text
它不只作用于 X
```

------

# 不是 Motion 逻辑

因为：

```text
它优先级高于所有运动
```

------

# 它真正属于：

# “Safety Domain（安全域）”

这是工业控制里的一个独立领域。

------

# 四、当前项目缺少什么？

你现在的项目：

已经有：

```text
Axis Domain
Gantry Domain
```

但：

```text
没有 Safety Domain
```

------

# 五、急停不应该放在哪里？

------

# ❌ 不应该放 Axis

例如：

```cpp
Axis::emergencyStop()
```

这是错误的。

因为：

```text
急停不是单轴行为
```

------

# ❌ 不应该放 Gantry

例如：

```cpp
GantryEmergencyStopController
```

也是错误的。

因为：

```text
急停不是龙门行为
```

------

# ❌ 不应该放 application/policy

因为：

```text
Policy 是业务编排
```

而：

```text
急停是系统级安全真相
```

------

# ❌ 不应该放 infrastructure

因为：

```text
PLC 只是物理执行者
```

而：

```text
安全状态
```

本身属于：

# Domain 真相

------

# 六、正确的位置

我建议你：

# 新增：

```text
domain/safety
```

------

# 七、推荐结构

------

domain/

------

# 八、为什么必须独立成 Safety Domain

因为：

```text
安全系统
```

在工业控制里：

# 永远是最高优先级

它不属于：

- Motion
- Axis
- Gantry

它是：

# 横切系统（Cross-cutting Domain）

------

# 九、你现在 PLC 已经证明了这一点

因为：

```pascal
AND NOT 设备急停
```

已经说明：

```text
急停
```

位于：

# 所有控制逻辑之上

这其实就是：

# “安全裁决层”

------

# 十、推荐的领域模型

------

# 10.1 EmergencyStopController

职责：

```text
管理急停状态机
```

例如：

```cpp
class EmergencyStopController {
public:
    SafetyRejection requestEmergencyStop(bool active);

    void applyFeedback(const EmergencyStopFeedback& feedback);

    bool isEmergencyStopped() const;
};
```

------

# 10.2 EmergencyStopCommand

```cpp
struct EmergencyStopCommand {
    bool active;
};
```

------

# 10.3 EmergencyStopFeedback

```cpp
struct EmergencyStopFeedback {
    bool emergencyStopped;
};
```

------

# 十一、然后进入统一命令总线

现在：

```cpp
using SystemCommand = std::variant<
    AxisCommandWithId,
    GantryCouplingCommand,
    ServoPowerCommand,
    EmergencyStopCommand
>;
```

这会非常自然。

------

# 十二、真正关键的一点

# 急停不只是“按钮”

而是：

# “系统状态”

很多人会犯一个错误：

```text
把急停当 UI 行为
```

实际上：

```text
急停是系统进入安全模式
```

这是两个完全不同的东西。

------

# 十三、SystemContext 是否需要感知急停？

# 需要。

因为：

```text
急停会影响整个系统的访问语义
```

例如：

```text
急停时：
- 禁止运动
- 禁止定位
- 禁止联动
- 禁止点动
```

这已经是：

# 系统上下文真相

------

# 十四、推荐未来结构

------

# SystemContext

持有：

```cpp
EmergencyStopController
GantryCouplingController
GantryPowerController
Axis
```

------

# Context 负责：

```text
系统级状态裁决
```

例如：

```text
if emergency stop:
    reject all motion
```

------

# 十五、Application 层应该做什么？

Application：

```text
只负责编排
```

例如：

```cpp
EnableUseCase
```

执行前：

```cpp
if (ctx.emergencyStopController().isEmergencyStopped())
    return UseCaseError::EmergencyStopped;
```

------

# 十六、PLC 与 Domain 的职责边界

这是最关键的。

------

# PLC：

负责：

```text
物理安全执行
```

例如：

```text
真正断使能
真正清空动作
```

------

# Domain：

负责：

```text
系统安全语义
```

例如：

```text
当前系统是否处于 EmergencyStop
```

------

# 独立成为系统级安全聚合
