# 重新理解 AI VibeCoding 下的龙门轴架构

## 一、“守旧派”的困境

之前我是一个严格的代码守旧派。一般开发一个逻辑的时候，我通常就是这个步骤：

1. AI生成初步需求。写代码前，需要将口头描述以及现实思考，输入给AI，将其整理为现实业务需求；
2. 手工校正需求说明。拿到AI输出的业务需求后，我会经过反复斟酌和考量，进行增加、删减或润色，以达到满意的业务需求说明；
3. 生成较为严格的业务语义约束。拿到改良版的业务需求后，让AI协助我生成对应的代码业务语义约束；
4. 基于TDD实现语义约束。基于 TDD 的最小实现步骤：针对一个功能点，进行测试案例的“红灯” - “绿灯” 的开发步骤，逐步增加和完善功能点；。

但是随着愈加复杂的代码逻辑以及文件的增多，基于网页对话框的 AI “闻讯” 式的对话开发，让我逐渐地感到其限制性——无法感知整体架构:partly_sunny: ：

- 每一个对话框能容纳的上传文件数量是有限的。即使强如Gemini，上传文件数量和大小依旧有限制。
- Gemini的代码仓库上传固然好用，但是实测下来，Gemini好像并没有理解仓库代码的整体逻辑。导致写的新需求很多都跟实际架构不符合。
- 对话框的记忆限制太小。对话框每次内容较多后，就会卡顿（特别是长对话的ChatGPT，非常卡顿）。
- 架构理解能力较弱。例如我实现了Domain领域层的逻辑后，上一层的Application代码逻辑不仅要符合实际的业务语义，也要兼顾Domain层的代码逻辑。但是AI对话框的架构依赖能力较弱，经常无法有效地生成一个业务需求横跨多个层的逻辑。

所以在这样的开发条件下，随着代码数量上升和层级结构的复杂上升，AI 给我生成的代码结构就会变得跟实际需求相去甚远。

感到力不从心的我，开始研究如何才能让开发效率更高的办法:book: 。

此时正值`DeepseekV4`模型发布，抱着试一试的心态，将其 API 导入了编辑器中。从此就打开了代码编辑的新世界:key:  。

## 二、初遇API的喜悦和烦恼

将`DeepseekV4`的API导入到编辑器后，之前基于网页对话框式的开发困难，就基本没有了：

- 文件上传*基本*无限。这里说的`基本`的意思，就是截至当前项目的69个文件，API每次对话都能较好地分析和理解每一次提问涉及到的文件。
- 上下文对话超长。反正基本没遇到过记忆不够导致的幻觉。每次回答都很精准。
- 代码生成效率遥遥领先。这个就是基本上把修改文件和执行本地命令的权限交给它后，只要需求够明确，每次对话都能生成对应的一整套基于 TDD 的代码逻辑。

这对于之前每次新开对话框，还要头疼如何组织 prompt 来让其拥有之前开发记忆的我来说，API开发彻底解放了我的焦虑。

而尝到甜头的我，相继开发了新项目的：

- 分组模块
- 多轴控制模块
- 龙门轴模块（正在进行）

但是在5月1号放假空闲的时候，我记得公司的电脑为了后台跑AI代码生成还没关机，于是远程连接到电脑上，想看看最近生成的代码逻辑。

此时就遇到了API开发带给我的一份“恐惧”炸弹：

- 我看不懂双轴龙门结构的设计逻辑

所有的代码我都能看懂，但是我并不知道为什么这么设计以及到底想要干什么。（~~其实代码有些我也看不懂，类似工厂设计以及C++的特殊语法等~~）

我问了下ChatGPT关于为什么看不懂的原因，如下是截取的部分回答：

> 你把「构建过程」外包给了 AI，但没有建立自己的“架构控制权”。
>
> 你现在实际上同时在学习理解：
>
> 1. 工业控制
> 2. DDD设计思想
> 3. Clean Architecture
> 4. TDD开发约束
> 5. 现代 C++语法
>
> ------
>
> 这是非常恐怖的信息量。
>

是啊，代码生成的越来越快，越来越好，而我无法理解这份美好:cry: 。

无法掌控自己的代码，这种痛苦对于一个软件工程师来说，是真的挺难受！

> 不久的将来，我依旧会记得：在5月1号的那个阳光灿烂的下午，我拉取代码后首先震撼于 API 给我生成的完整的架构文档和测试代码是如此的完备，而又痛苦于读不懂的痛苦感受。

是时候做出改变了。

## 三、“主动阅读”的破局之道

我开始采用了“通读全文”的逻辑，就是一个一个的代码文件进行通读。

但是这样的阅读的效率较低，无法通过代码将背后蕴含的业务逻辑串联起来。从理论的角度来看，这个应该算是低效率的被动思考。:stop_sign:

毕竟复杂系统不是“线性结构”，无法通过从上往下顺着读所有代码来弄懂所有逻辑。

AI给我的解决方案是：做为“约束驱动系统”，我应该带着“问题”阅读。而开始，就从

```bash
tests/domain/gantry/
```

目录下的测试开始。（截至5月7号，借助API生成的测试文件已经有19个）

> 测试描述的是：“系统行为”。而代码描述的是：“系统实现”。
>
> 第一阶段不要研究“怎么实现”，只研究：系统到底想保证什么行为。

那就慢一点，通过阅读tdd测试，来倒逼自己认识这个系统吧。并且为了检验自己的认知，我后续的阅读要配合主动笔记。

唉，真的是：

> AI 可以帮你生成“局部实现”。
>
> 但系统最终的长期形态，
>
> 只能由你自己决定啊。🌱
