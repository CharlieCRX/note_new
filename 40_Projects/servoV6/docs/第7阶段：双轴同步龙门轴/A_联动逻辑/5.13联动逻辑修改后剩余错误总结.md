## 架构分析结论：`SystemContext` 是否剥夺了 `GantryGroup` 的功能？

### 核心结论：**否，当前代码不存在该问题。**

当前 `SystemContext::tryGetAxis()` 中的龙门门控代码完全委托给 `GantryGroup`：

```cpp
if (m_gantry->isCoupled()) {   // 直接查询 GantryGroup 的四态状态机
    if (id == AxisId::X1 || id == AxisId::X2) { ... }
} else {
    if (id == AxisId::X) { ... }
}
```

耦合状态的**唯一真相来源**是 `GantryGroup` 内部维护的 `GantryCouplingState` 四态机（`Decoupled → CouplingRequested → Coupled → DecouplingRequested`），而该状态机又完全由来自 PLC 的 `applyFeedback(const GantryFeedback& feedback)` 驱动。不存在任何名为 `m_isGantryCoupled` 的冗余全局标志。`SystemContext` 只做单向只读查询，不存在职责剥夺。

## 多组+多轴状态下的联动/解耦设计说明

### 1. 联动状态声明

- **每个 `SystemContext` 实例持有一个独立的 `GantryGroup`**（含独立的 `GantryCouplingState` 四态机），各组之间完全物理隔离。
- 联动状态的**唯一写入路径**：`GantryGroup::applyFeedback(const GantryFeedback&)` 来自 PLC 的真实物理反馈。
- 联动状态的**唯一读取路径**：`GantryGroup::isCoupled()` / `isCouplingRequested()` / `isDecouplingRequested()`。

### 2. 联动流程控制

```
用户层 → GantryOrchestrator::startCoupling()
         ↓
         tick() → EnsuringEnabled   (EnableUseCase → X 轴使能)
         ↓
         tick() → WaitingEnabled    (等待 X 轴 Idle)
         ↓
         tick() → Coupling          (gantry.requestCouple(true) → 产生 GantryCommand)
         ↓
         tick() → WaitingCoupled    (等待 PLC 反馈 isCoupled=true)
         ↓
         tick() → Done              (tryGetAxis(X1/X2) 被拒，tryGetAxis(X) 可用)
```

### 3. 解耦流程控制

```
用户层 → GantryOrchestrator::startDecoupling()
         ↓
         tick() → Decoupling        (gantry.requestCouple(false) → 产生解耦指令)
         ↓
         tick() → WaitingDecoupled  (等待 PLC 反馈 isCoupled=false)
         ↓
         tick() → Done              (tryGetAxis(X) 被拒，tryGetAxis(X1/X2) 可用)
```

## 解决方案清单

| #    | 问题                                                | 解决方案                                                     |
| ---- | --------------------------------------------------- | ------------------------------------------------------------ |
| 1    | `GantryOrchestrator` SetUp 耦联污染（影响测试 1-3） | 从 `SetUp()` 中**移除** `applyFeedback({isCoupled=true})`，改为在需要耦联前置条件的测试（`full_decoupling_flow`、`error_when_gantry_rejects_coupling`、`error_when_plc_reports_position_error`）中内联设置，或在各自测试的 Given 块中显式设置 |
| 2    | `tick()` 入口无条件获取 X 轴（影响测试 4）          | 将 `tryGetAxis(AxisId::X)` 从 `tick()` 的公共前置步骤**下沉**到 `EnsuringEnabled` / `WaitingEnabled` / `Coupling` 步骤内部；在 `Decoupling` / `WaitingDecoupled` 步骤中不应获取 X 轴，或让 X 成为一个 `std::optional<Axis*>` |
| 3    | `SystemIntegrationTest` 默认解耦（影响测试 5）      | 在 `ShouldRejectPhysicalAxisWhenGantryCoupled` 内增加前置 `ctxA->gantry().applyFeedback({.isCoupled = true, .errorCode = 0})` |

**所有问题本质均为测试编写偏差或极小的编排器入口逻辑不合理，不涉及架构层面的职责混乱。**
