## 问题确认：是的，`m_isGantryCoupled` 确实剥夺了 `GantryGroup` 的控制权

### 核心矛盾：两个独立的耦合状态真相源并存

```
SystemContext
├── m_isGantryCoupled (bool)          ← 手动设置的简单布尔值
│   └── tryGetAxis() 使用此值做轴访问控制
│
└── m_gantry (GantryGroup)
    └── GantryCouplingState           ← 4态状态机，由 PLC 反馈驱动
        └── isCoupled() 通过 applyFeedback() 获得真实物理状态
```

**两个状态之间没有任何同步机制。**

### 具体问题（4个）

#### 问题 1：`tryGetAxis()` 依赖的是过时的 `m_isGantryCoupled`

```cpp
// SystemContext.h lines 34-41
if (m_isGantryCoupled) {
    if (id == AxisId::X1 || id == AxisId::X2) {
        reason = ContextRejection::PhysicalAxisLockedByGantry;
        return false;
    }
} else {
    if (id == AxisId::X) {
        reason = ContextRejection::LogicalAxisUnavailableWhenDecoupled;
        return false;
    }
}
```

`tryGetAxis()` 查询的是 `m_isGantryCoupled`，**而不是** `m_gantry->isCoupled()`。

#### 问题 2：`setCoupledState()` 直接修改布尔值，完全绕过 GantryGroup

```cpp
void setCoupledState(bool coupled) { m_isGantryCoupled = coupled; }
```

这个接口可以直接把 `m_isGantryCoupled` 设为任意值，即使 PLC 反馈说状态不同。这是一个**危险的旁路**。

#### 问题 3：`GantryOrchestrator` 完成后不通知 `SystemContext`

编排器等待 `gantry.isCoupled()` 完成后，**从不调用** `group->setCoupledState()` 来同步布尔值。

```cpp
// GantryOrchestrator.h line 134-138
case Step::WaitingCoupled:
    if (gantry.isCoupled()) {          // ← 仅检查 GantryGroup
        m_step = Step::Done;           // ← 完成后不再同步 SystemContext
    }
```

#### 问题 4：集成测试通过 `setCoupledState(false)` 掩盖了架构缺陷

```cpp
// test_system_integration.cpp line 375
ctxB->setCoupledState(false);  // 手动绕过 GantryGroup 状态机
```

### 典型的数据流断裂场景

| 步骤 | 操作                                     |   `m_isGantryCoupled`    | `GantryGroup::isCoupled()` |
| ---- | ---------------------------------------- | :----------------------: | :------------------------: |
| 1    | 初始状态                                 |          `true`          |  `false` (初始 Decoupled)  |
| 2    | 用户发起解耦，Orchestrator 下发指令      |          `true`          |   `DecouplingRequested`    |
| 3    | PLC 反馈解耦完成，`applyFeedback()` 调用 |          `true`          |     `false` ✅ 真实解耦     |
| 4    | 用户想操作 X1，调用 `tryGetAxis(X1)`     | ❌ **拒绝** (还在 `true`) |     `false` (但没用上)     |

**结论：步骤 4 时，PLC 已经解耦，GantryGroup 也知道解耦了，但 `tryGetAxis` 依然阻止访问 X1。这是实实在在的 bug。**

---

## 解决方案方案（核心思路）

### 方向：移除 `m_isGantryCoupled`，让 `tryGetAxis()` 委托给 `GantryGroup`

```cpp
// 现在的做法（问题）：
if (m_isGantryCoupled) { ... }

// 应该改为（方案）：
if (m_gantry->isCoupled()) { ... }
```

### 带来的改变

| 项目             | 当前                    | 建议                                                        |
| ---------------- | ----------------------- | ----------------------------------------------------------- |
| 状态真相源       | 两个（布尔值 + 状态机） | 一个（`GantryCouplingState`）                               |
| 轴访问控制依据   | `m_isGantryCoupled`     | `m_gantry->isCoupled()`                                     |
| 外部设置状态接口 | `setCoupledState(bool)` | **移除**，只通过 `requestCouple()` + `applyFeedback()` 闭环 |
| 同步问题         | 编排器完成后需手动同步  | 无需同步，`tryGetAxis()` 自动反映真实状态                   |

## 多组 + 多轴状态下的策略

### 1. 一个组 X1/X2 的联动状态声明方式

**组的耦合状态应由 `GantryGroup` 内部的状态机唯一声明。**

```
SystemManager
├── "Machine_A" → SystemContext_A
│   ├── Axis Y, Z, R
│   └── GantryGroup_A (X1_A, X2_A)  ← GantryCouplingState 声明是否联动
│
├── "Machine_B" → SystemContext_B
│   └── GantryGroup_B (X1_B, X2_B)  ← GantryCouplingState 声明是否联动
│
└── "Machine_C" → SystemContext_C
    └── GantryGroup_C (X1_C, X2_C)  ← GantryCouplingState 声明是否联动
```

每组之间**完全独立**。

### 2. 联动/解耦完整控制流程

```
应用层 (Application):
  GantryOrchestrator
    ├── startCoupling()   → Step::EnsuringEnabled
    ├── startDecoupling() → Step::Decoupling
    └── tick()            → 逐帧驱动
         │
         ▼
领域层 (Domain):
  SystemContext
    ├── tryGetAxis()       ← 查询 GantryGroup::isCoupled()
    └── gantry()           → GantryGroup
  GantryGroup
    ├── requestCouple(bool) → 意图生成（幂等/冲突检查）
    ├── popPendingCommand() → 弹出 GantryCommand
    └── applyFeedback()     ← PLC 物理状态驱动状态机
         │
         ▼
基础设施层 (Infrastructure):
  ISystemDriver
    ├── sendGantry(cmd)    → 下发耦合/解耦命令到 PLC
    └── 轮询反馈            → GantryGroup::applyFeedback()
```

### 3. 设计原则清单

| 原则               | 说明                                                         |
| ------------------ | ------------------------------------------------------------ |
| **单一真相源**     | 耦合状态只有 `GantryCouplingState` 一处，`SystemContext` 不另存布尔值 |
| **反馈驱动**       | `tryGetAxis()` 的行为由 PLC 反馈更新后的 `GantryGroup` 决定  |
| **意图与反馈分离** | `requestCouple()` 只产生意图，`applyFeedback()` 才真正改变状态 |
| **以组为单位**     | 每组独立的 `GantryGroup`，互不干扰                           |
| **无旁路接口**     | 不提供 `setCoupledState()` 这种绕过状态机的直接赋值接口      |

### 4. 多组隔离示意

```
Machine_A 组:
  GantryGroup_A: isCoupled() → true  (X1_A/X2_A 联动中)
  tryGetAxis(X1_A) → PhysicalAxisLockedByGantry  ✅ 正确拒绝
  tryGetAxis(X)    → OK                          ✅ 逻辑轴可用

Machine_B 组:
  GantryGroup_B: isCoupled() → false (X1_B/X2_B 独立控制中)
  tryGetAxis(X1_B) → OK                          ✅ 正确放行
  tryGetAxis(X)    → LogicalAxisUnavailableWhenDecoupled ✅ 正确拒绝
```

## 修正清单（总结）

### 已确认的问题
- [x] `SystemContext::m_isGantryCoupled` 作为独立布尔值与 `GantryGroup` 状态机脱节
- [x] `tryGetAxis()` 依赖过时布尔值而非 `GantryGroup::isCoupled()`
- [x] `setCoupledState()` 提供了绕过完整反馈闭环的旁路
- [x] `GantryOrchestrator` 完成后不同步 `SystemContext`
- [x] 集成测试通过直接调用 `setCoupledState()` 掩盖了架构缺陷

### 建议的修正项
- [ ] 移除 `SystemContext::m_isGantryCoupled` 私有变量
- [ ] 移除 `SystemContext::setCoupledState()` / `isGantryCoupled()` 接口
- [ ] `tryGetAxis()` 改为委托 `m_gantry->isCoupled()` 决定访问策略
- [ ] `GantryOrchestrator` 在完成 `WaitingCoupled`/`WaitingDecoupled` 后不需要额外同步——因为 `tryGetAxis()` 已在查询真实状态机
- [ ] 集成测试中移除对 `setCoupledState()` 的直接调用，改为通过 `GantryOrchestrator` 走完整流程
