# 龙门轴「使能-联动」悖论分析与修正方案

---

## 一、悖论复原

当前 `GantryOrchestrator::tick()` 每帧开头：

```cpp
SystemContext* group = ...;                              // ① 获取分组
Axis* x = nullptr;
group->tryGetAxis(AxisId::X, x, ctxReason);              // ② 获取逻辑轴 X
GantryGroup& gantry = group->gantry();                   // ③ 获取龙门聚合根
```

`SystemContext::tryGetAxis` 的裁决逻辑：

```cpp
if (m_gantry->isCoupled()) {
    // 允许 X，锁定 X1/X2
} else {
    if (id == AxisId::X) {
        // 拒绝 X：LogicalAxisUnavailableWhenDecoupled  ← 关键
    }
}
```

**悖论链**：

```
联动流程启动
  → Step::EnsuringEnabled
  → 调用 EnableUseCase::execute(mgr, group, AxisId::X, true)
  → EnableUseCase 内部调用 group->tryGetAxis(AxisId::X, ...)
  → GantryCouplingState 处于 Decoupled → AxisId::X 被拒绝
  → 错误：LogicalAxisUnavailableWhenDecoupled

结论：想联动就必须先使能 X，想访问 X 就必须先联动。死锁。
```

更深层的问题：**即使是 `WaitingEnabled` 步骤中读取 `x->state()`，也需要先通过 `tryGetAxis(AxisId::X)` 拿到 `Axis*`，同样被阻断。**

## 二、根因分析

### 2.1 PLC 寄存器全景

| 类型 | 寄存器名          | 含义                            |
| ---- | ----------------- | ------------------------------- |
| 控制 | 使能轴X电机       | 同时使能 X1/X2 电机（单寄存器） |
| 控制 | 轴X联动使能       | 命令 X1/X2 进入联动同步         |
| 状态 | 轴X状态显示       | 电机是否已使能（0/1）           |
| 状态 | 轴X联动状态       | 是否已联动（0/1）               |
| 状态 | Gantry_Error_Code | 联动失败原因                    |

### 2.2 过载的语义

`AxisId::X` 同时承载了两个不同含义：

| 含义                                                         | 场景                | tryGetAxis 约束              |
| ------------------------------------------------------------ | ------------------- | ---------------------------- |
| **硬件寄存器代理**：控制「使能轴X电机」、反馈「轴X状态显示」 | 使能/掉电、状态查询 | **永远需要**（无论是否联动） |
| **逻辑联动轴**：联动后的虚拟运动轴                           | Jog/Move/Stop       | **仅联动时可用**             |

当前 `tryGetAxis(AxisId::X)` 只按「逻辑联动轴」语义裁决，导致使能/掉电路径被误杀。

### 2.3 三类操作场景

| 场景         | 使能轴X电机 | 轴X联动使能 | 可用轴             | 说明         |
| ------------ | ----------- | ----------- | ------------------ | ------------ |
| **使能联动** | ON          | ON          | X（X1/X2 锁定）    | 龙门联动运动 |
| **使能解耦** | ON          | OFF         | X1、X2（X 不可用） | 两轴独立控制 |
| **不使用**   | OFF         | OFF         | 无                 | 全部掉电     |

## 三、设计建议：新增 `GantryMotorController` 类

### 3.1 职责定位

```
┌─────────────────────────────────────────────────────┐
│ GantryMotorController                                │
│   唯一职责：控制 PLC 寄存器「使能轴X电机」             │
│                                                     │
│   不关心联动状态（Couple/Decouple）                   │
│   不关心 X1/X2 是否锁定                              │
│   只需知道：电机是否上电？                             │
└─────────────────────────────────────────────────────┘
```

**与现有组件的关系**：

```
                  ┌──────────────────────┐
                  │  GantryOrchestrator   │  ← 只负责场景1（使能联动）
                  └──────────┬───────────┘
                             │ 使用
                  ┌──────────▼───────────┐
                  │  GantryMotorController│  ← 控制「使能轴X电机」
                  └──────────┬───────────┘
                             │ 使用（反馈路径）
                  ┌──────────▼───────────┐
                  │  GantryGroup           │  ← 控制「轴X联动使能」+ 状态机
                  └──────────────────────┘
```

- **GantryMotorController**：控制「使能轴X电机」寄存器。联动/解耦状态下均可访问。
- **GantryGroup**（现有）：控制「轴X联动使能」寄存器 + 维护 `GantryCouplingState` 五态状态机。
- **GantryOrchestrator**（修正后）：编排场景 1（使能→联动）流程，使用上述两者。
- **场景 2&3**：由外部直接操作 `GantryMotorController` + 独立的 `AxisId::X1`/`AxisId::X2`，无需编排器。

### 3.2 接口设计

```cpp
struct MotorCommand {
    bool enable;  // true = 使能, false = 掉电
};

class GantryMotorController {
public:
    // ========== 意图生成 ==========

    /**
     * @brief 请求使能/掉电龙门电机（控制「使能轴X电机」寄存器）
     * @return None 成功（包括幂等）；NotSynchronized 尚未与PLC同步
     *
     * 约束：
     *   - NotSynchronized → 拒绝（与 GantryGroup 入口守卫一致）
     *   - 已处于目标状态 → 幂等，不生成命令
     *   - 无论联动/解耦均可调用
     */
    GantryRejection requestEnable(bool active) {
        if (!m_synchronized) return GantryRejection::NotSynchronized;
        if (m_enabled == active) return GantryRejection::None;  // 幂等
        m_enabled = active;
        m_pending_cmd = MotorCommand{active};
        return GantryRejection::None;
    }

    // ========== 状态查询 ==========
    bool isEnabled() const { return m_enabled; }
    bool isSynchronized() const { return m_synchronized; }

    // ========== 命令弹出 ==========
    bool hasPendingCommand() const { return m_pending_cmd.has_value(); }
    MotorCommand popPendingCommand() {
        auto cmd = *m_pending_cmd;
        m_pending_cmd.reset();
        return cmd;
    }

    // ========== 反馈接收（由驱动/PLC 轮询线程调用） ==========

    /**
     * @brief 接收 PLC 寄存器「轴X状态显示」的反馈
     *        首次反馈自动标记 m_synchronized = true（退出未同步态）
     */
    void applyMotorFeedback(bool enabled) {
        m_synchronized = true;
        m_enabled = enabled;
    }

private:
    bool m_enabled = false;
    bool m_synchronized = false;  // 首次收到 PLC 反馈前为 false
    std::optional<MotorCommand> m_pending_cmd;
};
```

### 3.3 关键设计决策

| 决策                                                         | 理由                                                         |
| ------------------------------------------------------------ | ------------------------------------------------------------ |
| 独立类而非集成到 `GantryGroup`                               | 「使能轴X电机」和「轴X联动使能」是两个独立 PLC 寄存器，分属不同硬件操作。独立类拆分职责，场景 2 只需使能电机不碰联动状态机 |
| `m_synchronized` 独立于 `GantryCouplingState::NotSynchronized` | 电机控制器只需知道是否收到过反馈，不需要五态状态机。首次 `applyMotorFeedback` 即同步 |
| 幂等由自身保证                                               | `m_enabled == active` 时直接返回 None，外部无需额外判断      |
| `requestEnable` 在任何联动状态下均可调用                     | 调用方自行保证语义正确性（场景 2 解耦时使能是合理的）        |

### 3.4 反馈路径

```
PLC 轮询线程
  │
  ├─→ GantryMotorController::applyMotorFeedback(轴X状态显示)
  │     （同时同步 Axis::X 实体的 AxisState）
  │
  └─→ GantryGroup::applyFeedback(GantryFeedback{isCoupled, errorCode})
        （驱动 GantryCouplingState 状态机）
```

## 四、修正后的 GantryOrchestrator 状态机流转约束

> **GantryOrchestrator 只负责场景 1「使能联动」**。场景 2（使能解耦）和场景 3（不使用）由调用方直接操作 `GantryMotorController` 和独立轴。

### 4.1 前置依赖

`GantryOrchestrator` 需要同时持有 `GantryMotorController` 和 `GantryGroup` 的引用：

```cpp
class GantryOrchestrator {
    // ...
private:
    GantryMotorController& m_motor;  // 新增
    GantryGroup& m_gantry;           // （通过 SystemContext 间接获取）
};
```

### 4.2 tick() 获取引用部分修正

```cpp
void tick() {
    SystemContext* group = ...;
    // 移除 tryGetAxis(AxisId::X) — 不再需要
    GantryGroup& gantry = group->gantry();
    GantryMotorController& motor = group->motorController(); // 新增

    switch (m_step) { ... }
}
```

### 4.3 入口守卫（不变）

```
startCoupling():
  IF tryGetGroup 失败           → Error(ContextRejection)
  IF gantry.isNotSynchronized() → Error(GantryRejection::NotSynchronized)
  ELSE                          → EnsuringEnabled
```

### 4.4 步骤流转

---

### 4.4.1 Initial（初始态）

```
触发: 构造后默认
tick(): 无操作，等待 startCoupling() / startDecoupling() 驱动
```

---

### 4.4.2 EnsuringEnabled → WaitingEnabled / Error（仅 startCoupling 路径）

```
触发: startCoupling() 通过入口守卫后

tick():
  调用 m_motor.requestEnable(true):
    IF GantryRejection::None               → WaitingEnabled
       （若有 pendingCommand，下发到 driver）
    IF GantryRejection::NotSynchronized    → Error(GantryRejection::NotSynchronized)

说明:
  MotorController 在任何联动状态下均可访问，无论 Decoupled/Coupled/NotSynchronized。
  requestEnable 内部幂等：已使能再调用 → 返回 None 且无新命令。
```

**关键变化**：不再经过 `tryGetAxis(AxisId::X)` → `EnableUseCase` 链路。原链路在 `Decoupled` 状态下被 `LogicalAxisUnavailableWhenDecoupled` 扼杀。新路径直接操作 `GantryMotorController`，绕开 `SystemContext` 的龙门锁定。

---

### 4.4.3 WaitingEnabled → Coupling / Error（仅 startCoupling 路径）

```
tick():
  IF m_motor.isEnabled()       → Coupling
  ELSE                          → 保持 WaitingEnabled
  // 可选超时保护：若轮询次数超限 → Error(Timeout)

说明:
  "轴X状态显示" 寄存器反馈 → GantryMotorController::applyMotorFeedback(enabled)
  → m_enabled 更新。不依赖 Axis::X 实体（Decoupled 状态下不可访问）。
```

**关键变化**：不再读取 `x->state()`（原需 `tryGetAxis(AxisId::X)` 获取轴指针）。电机状态由 `GantryMotorController` 独立追踪。

---

### 4.4.4 Coupling → WaitingCoupled / Error（不变）

```
tick():
  调用 gantry.requestCouple(true):
    IF GantryRejection::None    → 下发 pendingCommand（若有）→ WaitingCoupled
    IF 其他拒绝                 → Error(GantryRejection)

幂等场景:
  电机已使能 + 已联动 → requestCouple(true) 内部幂等返回 None → WaitingCoupled
  → 下帧 isCoupled()=true → Done（静默成功，无副作用）
```

---

### 4.4.5 WaitingCoupled → Done / Error（不变）

```
tick():
  优先级 1: IF gantry.isCoupled()                                    → Done
  优先级 2: IF gantry.hasError()                                     → Error(lastError)
  优先级 3: IF !gantry.isCouplingRequested() && !gantry.isCoupled()  → Error(UnexpectedFeedback)
             // PLC errorCode=0 但 isCoupled=false，状态机从 CouplingRequested 退回 Decoupled
  优先级 4: ELSE                                                      → 保持 WaitingCoupled
```

---

### 4.4.6 Decoupling → WaitingDecoupled / Error（仅 startDecoupling 路径）

```
触发: startDecoupling() 通过入口守卫

tick():
  调用 gantry.requestCouple(false):
    IF GantryRejection::None    → 下发 pendingCommand（若有）→ WaitingDecoupled
    IF 其他拒绝                 → Error(GantryRejection)

幂等场景:
  已解耦 → requestCouple(false) 内部幂等返回 None（不生成命令）
  → WaitingDecoupled → 下帧 !isCoupled() && !isCouplingReq() && !isDecouplingReq() → Done

注意:
  解耦不操作电机（电机保持使能 → 场景2「使能解耦」）。
  掉电由外部通过 GantryMotorController::requestEnable(false) 独立执行。
```

---

### 4.4.7 WaitingDecoupled → Done / Error（不变）

```
tick():
  优先级 1: IF !gantry.isCoupled()
            && !gantry.isCouplingRequested()
            && !gantry.isDecouplingRequested()                        → Done
  优先级 2: IF gantry.hasError()                                     → Error(lastError)
  优先级 3: IF gantry.isCoupled() && !gantry.isDecouplingRequested() → Error(UnexpectedFeedback)
             // PLC errorCode=0 但 isCoupled=true，状态机从 DecouplingRequested 退回 Coupled
  优先级 4: ELSE                                                      → 保持 WaitingDecoupled
```

---

### 4.4.8 Done / Error（终态不变）

```
Done:
  tick() → 保持 Done（幂等）
  可重新调用 startCoupling()/startDecoupling() 发起新流程

Error:
  tick() → 保持 Error（幂等）
  m_lastError 保留最后错误
  调用方通过 startCoupling()/startDecoupling() 重新发起
```

**与旧版流转图的关键差异**：

|                 | 旧版                                              | 新版                                                 |
| --------------- | ------------------------------------------------- | ---------------------------------------------------- |
| EnsuringEnabled | `EnableUseCase(AxisId::X)` → 被 `tryGetAxis` 扼杀 | `motor.requestEnable(true)` → 不经过 `SystemContext` |
| WaitingEnabled  | 读取 `x->state()` → 需要 `tryGetAxis(AxisId::X)`  | 读取 `motor.isEnabled()` → 独立于龙门锁定            |
| `Axis::X` 实体  | 在联动流程中扮演双重角色                          | 在联动流程中**完全不再被 GantryOrchestrator 引用**   |

## 八、分层职责与反馈路径

### 8.1 完整组件关系图

```

       │           │  tick()                                      │
       │           │    ├─ motor.requestEnable()                  │
       │           │    ├─ motor.isEnabled()                      │
       │           │    └─ gantry.requestCouple()                 │
       │           │                                              │
       │           │  场景 1: 使能联动（本编排器）                   │
       │           └──────────────────────────────────────────────┘
       │
       │  场景 2: 使能解耦 — 外部直接操作
       │           motor.requestEnable(true)
       │           独立控制 Axis::X1 / Axis::X2

       │           motor.requestEnable(false)
       │

### 8.2 反馈路径

```
PLC 轮询线程每一帧:

  1. 读「轴X状态显示」→ motor.applyMotorFeedback(enabled)
     └─ 首次调用自动设置 m_synchronized = true

  2. 读「轴X联动状态」「Gantry_Error_Code」
     → GantryFeedback{isCoupled, errorCode}
     → gantry.applyFeedback(feedback)
     └─ GantryGroup::applyFeedback 内部:
        1. m_last_error = translatePlcError(errorCode)
        2. if (isCoupled) m_state.applyCoupledFeedback()
           else           m_state.applyDecoupledFeedback()
        └─ NotSynchronized → Coupled/Decoupled（退出初始态）

  3. 读 Axis::X1 / Axis::X2 各自的状态寄存器
     → 各自 Axis 实体的 applyFeedback()
```

### 8.3 时序：成功联动流程

```
时刻  PLC状态                                          编排器                         MotorController    GantryGroup
───── ──────────────────────────────────────────────── ───────────────────────────── ───────────────── ────────────────
 T0   使能轴X电机=OFF  轴X联动使能=OFF  轴X状态显示=0   initial                       m_enabled=false    NotSynchronized
                                                       m_synchronized=false

## 九、三种龙门场景的控制方案

| 场景            | 描述             | 控制方式                                                     | 可用轴                                 |
| --------------- | ---------------- | ------------------------------------------------------------ | -------------------------------------- |
| **1. 使能联动** | 电机使能 + 联动  | `GantryOrchestrator::startCoupling()`                        | `AxisId::X`（X1/X2 锁定）              |
| **2. 使能解耦** | 电机使能，不联动 | `GantryMotorController::requestEnable(true)` + 直接控制 `AxisId::X1`/`AxisId::X2` | `AxisId::X1`、`AxisId::X2`（X 不可用） |
| **3. 不使用**   | 电机掉电         | `GantryMotorController::requestEnable(false)`                | 无（全部不可用）                       |

**场景 2&3 不经过 GantryOrchestrator**，由 UI/UseCase 层直接调用 `GantryMotorController` 和独立轴实体。
