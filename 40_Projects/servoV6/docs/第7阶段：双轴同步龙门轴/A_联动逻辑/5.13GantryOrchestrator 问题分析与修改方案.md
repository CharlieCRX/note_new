# GantryOrchestrator 问题分析与修改方案

## 一、当前架构上下文

```
GantryOrchestrator (编排器)
  → tick() 每帧: tryGetGroup → tryGetAxis(X) → gantry() → switch(m_step)
  → 读取 GantryGroup 状态，不负责 applyFeedback
GantryGroup (聚合根)
  → requestCouple(active): NotSynchronized 守卫 → 拒绝
  → applyFeedback(fb): translatePlcError + 驱动 m_state
GantryCouplingState (五态状态机)
  → 初始态 NotSynchronized，只能通过 feedback 退出
```

## 二、识别的 7 个问题

### 问题 1：startCoupling() 在 NotSynchronized 状态产生副作用残留
- `startCoupling()` → `EnsuringEnabled` → 下发 X 轴使能命令成功
- 到 `Coupling` 步骤才被 `NotSynchronized` 守卫拒绝
- **X 轴已被使能但联动失败**，留下孤立的使能态副作用

### 问题 2：startDecoupling() 缺少 NotSynchronized 前置守卫
- 直接跳入 `Decoupling` 步骤，等 tick() 才暴露错误
- UI 层无法提前判断不可操作（无法禁钮）

### 问题 3：WaitingCoupled 死锁
- PLC 返回 `{isCoupled=false, errorCode=0}` 时
- `isCoupled()=false` 且 `hasError()=false` → 两个条件都不满足
- **编排器永久卡在 WaitingCoupled**

### 问题 4：WaitingDecoupled 对称死锁
- PLC 返回 `{isCoupled=true, errorCode=0}` 而本地处于 `DecouplingRequested`
- `applyCoupledFeedback()` 将状态回到 `Coupled`
- 退出条件 `!isCoupled()` 永远不满足 → **永久卡死**

### 问题 5：startDecoupling() 缺乏调用时机守卫
- 允许在 Initial/EnsuringEnabled/WaitingEnabled 等不合理状态下调用
- 应在 `m_step==Done` 且 `gantry.isCoupled()==true` 时才允许

### 问题 6：编排器不负责 applyFeedback，职责边界不清晰
- tick() 只读取 GantryGroup，从不写入反馈
- 调用者必须理解「先 applyFeedback 再 tick」的隐式时序契约
- 测试与生产代码路径不一致

### 问题 7：startCoupling() 在非初始态调用缺乏守卫（次要）
- 在 Done/Error 等状态下调用虽然最终可能幂等通过，但语义不清晰

## 三、修改方案

### 3.1 新增 GantryRejection::UnexpectedFeedback = 103
解决问题 3、4 死锁——PLC 反馈 errorCode=0 但物理状态与期望矛盾。

### 3.2 startCoupling() 入口增加 NotSynchronized 前置拦截
```cpp
void startCoupling() {
    // 快速失败: 获取龙门状态，NotSynchronized → Error
    if (gantry.isNotSynchronized()) {
        m_step = Step::Error;
        m_lastError = GantryRejection::NotSynchronized;
        return;
    }
    m_step = Step::EnsuringEnabled;
}
```
**效果**: 在使能 X 轴之前拦截，彻底消除副作用残留。

### 3.3 startDecoupling() 入口增加双重守卫
```cpp
void startDecoupling() {
    if (gantry.isNotSynchronized()) → Error(NotSynchronized)
    if (!gantry.isCoupled()) → Error(StateConflict)
    m_step = Step::Decoupling;
}
```
**效果**: 确保只在已联动状态下才能发起解耦。

### 3.4 WaitingCoupled 增加「反馈与期望矛盾」检测
```cpp
case Step::WaitingCoupled:
    if (gantry.isCoupled()) → Done
    else if (gantry.hasError()) → Error
    else if (!gantry.isCouplingRequested() && !gantry.isCoupled()) → Error(UnexpectedFeedback)
    // PLC 静默否决联动（errorCode=0 但状态退回到 Decoupled）
```

### 3.5 WaitingDecoupled 增加对称检测
```cpp
case Step::WaitingDecoupled:
    if (!isCoupled() && !isCouplingRequested() && !isDecouplingRequested()) → Done
    else if (hasError()) → Error
    else if (isCoupled() && !isDecouplingRequested()) → Error(UnexpectedFeedback)
    // PLC 静默否决解耦（errorCode=0 但状态回到 Coupled）
```

### 3.6（可选）增加 applyFeedback 透传
```cpp
void applyFeedback(const GantryFeedback& feedback) {
    group->gantry().applyFeedback(feedback);
}
```
统一调用接口，明确「先反馈再 tick」的时序契约。
