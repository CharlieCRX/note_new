# 龙门联动状态机精细化 — 问题分析与解决方案汇总

---

## 一、问题背景

> 系统是否处于联动状态，应该**完全由 PLC 寄存器决定**，而非由本地默认值假设。

PLC 硬件逻辑：上位机写「联动使能X寄存器」→ ON，PLC 内部自动完成 X1/X2 联动建立，无误后返回 `Gantry_Error_Code = 0`；若失败，通过 `Gantry_Error_Code` 返回具体错误原因。整个过程的本地位元状态由 `GantryFeedback` 快照驱动。

---

## 二、识别的具体问题

### 问题 1：默认初始态假设物理状态

**位置**：`domain/gantry/GantryCouplingState.h`

```cpp
Status m_status = Status::Decoupled;  // ← 未经 PLC 确认的假设
```

系统启动时 PLC 寄存器「轴X联动状态」可能是 ON（物理已联动），但本地状态机武断宣称 `Decoupled`，导致**状态幻觉**：

| 危害场景                     | 后果                                                         |
| ---------------------------- | ------------------------------------------------------------ |
| PLC 保持联动，本地认为已解耦 | 用户发起联动请求时，`GantryGroup::requestCouple(true)` 通过幂等检查，会生成并下发重复联动指令到 PLC，可能引起不可预期的硬件行为 |
| PLC 已解耦，本地认为已联动   | 同理，反之亦然                                               |

### 问题 2：`SystemContext` 与 `GantryCouplingState` 没有同步机制

**位置**：`domain/entity/SystemContext.h`

```cpp
bool m_isGantryCoupled = true;  // ← 硬编码真值，从未与状态机同步
```

`SystemContext::tryGetAxis` 依赖 `m_isGantryCoupled` 裁决 X/X1/X2 轴暴露，此值与 `GantryCouplingState` 的实际物理状态完全脱钩。两者不一致时出现轴访问逻辑错误。

### 问题 3：`GantryGroup` 初始化时状态残缺

`GantryGroup` 构造时直接继承 `GantryCouplingState` 的默认状态，而该默认状态是一个未经物理确认的假设。

---

## 三、最终确定的解决方案

### 3.1 新增 `NotSynchronized` 状态

在 `GantryCouplingState::Status` 枚举中新增第五个状态，使状态机从四态升级为**五态**：

```cpp
enum class Status {
    NotSynchronized,       // ← 新增：系统启动后尚未收到任何 PLC 快照，物理真相未知
    Decoupled,
    CouplingRequested,
    Coupled,
    DecouplingRequested
};
```

默认初始值改为 `NotSynchronized`：

```cpp
Status m_status = Status::NotSynchronized;
```

新增查询接口：

```cpp
bool isNotSynchronized() const { return m_status == Status::NotSynchronized; }
```

### 3.2 状态转换图

```
                    ┌──────────────────────────────────────────┐
                    │         NotSynchronized (初始态)          │
                    │   系统启动后，尚未收到任何 PLC 反馈快照     │
                    └──────┬───────────────────┬───────────────┘
                           │                   │
             applyCoupledFeedback()   applyDecoupledFeedback()
             (PLC: isCoupled=true)   (PLC: isCoupled=false)
                           │                   │
                           ▼                   ▼
                    ┌─────────────┐     ┌──────────────┐
                    │   Coupled   │     │  Decoupled   │
                    └──────┬──────┘     └──────┬───────┘
                           │                   │
                    requestDecouple()    requestCouple()
                           │                   │
                           ▼                   ▼
                  ┌────────────────┐   ┌────────────────────┐
                  │DecouplingReq.  │   │ CouplingRequested  │
                  └───────┬────────┘   └─────────┬──────────┘
                          │                      │
              applyDecoupledFeedback()  applyCoupledFeedback()
                          │                      │
                          ▼                      ▼
                    ┌──────────────┐      ┌─────────────┐
                    │  Decoupled   │      │   Coupled   │
                    └──────────────┘      └─────────────┘
```

### 3.3 核心约束规则

| 规则                                 | 说明                                                         |
| ------------------------------------ | ------------------------------------------------------------ |
| **初始态唯一**                       | 默认 `NotSynchronized` — 不预设任何物理状态                  |
| **退出初始态方式唯一**               | 只能通过 `applyCoupledFeedback()` / `applyDecoupledFeedback()` 退出 — **只有 PLC 物理反馈能打破未知** |
| **意图操作拒绝**                     | `NotSynchronized` 下，`requestCouple()` 和 `requestDecouple()` 均被拒绝，返回 `GantryRejection::NotSynchronized` |
| **GantryCouplingState 底层方法不变** | `applyCoupledFeedback()` 和 `applyDecoupledFeedback()` 保持独立，分别映射 `isCoupled = true/false` |

### 3.4 分层职责（不变）

| 层级                  | 方法                                                  | 职责                                                         |
| --------------------- | ----------------------------------------------------- | ------------------------------------------------------------ |
| `GantryCouplingState` | `applyCoupledFeedback()` / `applyDecoupledFeedback()` | 纯状态机驱动：根据物理 `isCoupled` 转换状态                  |
| `GantryGroup`         | `applyFeedback(const GantryFeedback&)`                | **1.** 翻译 `errorCode` → `m_last_error`；**2.** 根据 `feedback.isCoupled` 分发到底层方法 |

`GantryGroup::applyFeedback` 保持现有模式不变：

```cpp
void applyFeedback(const GantryFeedback& feedback) {
    // 1. 翻译并记录 PLC 错误码
    m_last_error = translatePlcError(feedback.errorCode);

    // 2. 根据 PLC 实际联动状态驱动本地状态机
    if (feedback.isCoupled) {
        m_state.applyCoupledFeedback();
    } else {
        m_state.applyDecoupledFeedback();
    }
}
```

### 3.5 新增拒绝原因

在 `GantryRejection` 枚举中新增：

```cpp
NotSynchronized = 102,  // 状态机尚未与 PLC 物理状态同步，拒绝操作
```

### 3.6 `GantryGroup::requestCouple` 守卫增强

在现有 Axis Error 检查之前，增加 `NotSynchronized` 前置拦截：

```
if (m_state.isNotSynchronized()) {
    return GantryRejection::NotSynchronized;
}
```

这样在系统尚未收到第一帧 PLC 快照之前，任何联动/解耦意图都会被拒绝。

### 3.7 PLC 联动逻辑与本地位元关系

```
┌──────────────────────────────────────────────────────────────┐
│ PLC 硬件侧                                                    │
│   写 联动使能X寄存器 ON                                       │
│        │                                                      │
│        ├── 成功 ──▶ Gantry_Error_Code = 0                     │
│        │            X1/X2 自动进入联动状态                     │
│        │            GantryFeedback{isCoupled=true, errorCode=0}│
│        │                                                      │
│        └── 失败 ──▶ Gantry_Error_Code = 1/2/3/4/5             │
│                     GantryFeedback{isCoupled=false, errorCode=X}│
│                     本地位 m_last_error 记录具体原因            │
└──────────────────────────────────────────────────────────────┘
```
