# GantryOrchestrator 正确状态机流转约束（修正版）

---

## 一、状态枚举

```
Initial           // 构造后默认，等待入口驱动
EnsuringEnabled   // 联动路径：下发使能 X 轴
WaitingEnabled    // 联动路径：等待 X 轴进入 Idle
Coupling          // 联动路径：请求 GantryGroup 联动
WaitingCoupled    // 联动路径：等待 PLC 反馈联动结果
Decoupling        // 解耦路径：请求 GantryGroup 解耦
WaitingDecoupled  // 解耦路径：等待 PLC 反馈解耦结果
Done             // 终态：流程完成
Error            // 终态：流程失败，保留 lastError
```

---

## 二、入口守卫

启动函数只拦截 **NotSynchronized**（唯一会产生外部副作用的危险状态），不重复领域层已有的幂等/冲突判断。

### 2.1 startCoupling() 入口

```
IF tryGetGroup() 失败              → Error(ContextRejection::GroupNotFound)
IF gantry.isNotSynchronized()      → Error(GantryRejection::NotSynchronized)
ELSE                               → EnsuringEnabled
```

**设计意图**：
- `NotSynchronized` 下 `EnsuringEnabled` 会调用 `EnableUseCase` 下发使能命令到硬件，产生不可逆副作用。必须在此拦截。
- 已联动时重复调用：`EnableUseCase` 幂等 → `Coupling` 中 `requestCouple(true)` 幂等返回 `None` → `WaitingCoupled` 中 `isCoupled()=true` → `Done`。领域层静默处理，编排器无需拦截。

### 2.2 startDecoupling() 入口

```
IF tryGetGroup() 失败              → Error(ContextRejection::GroupNotFound)
IF gantry.isNotSynchronized()      → Error(GantryRejection::NotSynchronized)
ELSE                               → Decoupling
```

**设计意图**：
- `NotSynchronized` 下发起解耦命令到硬件是危险的（物理状态未知）。
- 已解耦时重复调用：`Decoupling` 中 `requestCouple(false)` 幂等返回 `None`（不生成命令）→ `WaitingDecoupled` 中 `!isCoupled() && !isCouplingRequested() && !isDecouplingRequested()` → `Done`。领域层静默处理，编排器无需拦截。**撤销此前 `!isCoupled()` 的冗余守卫。**

## 三、逐步骤流转约束

### 3.1 Initial
```
触发: 构造后默认
流转: tick() → 无操作（等待 startCoupling/startDecoupling 驱动）
```

### 3.2 EnsuringEnabled → WaitingEnabled / Error
```
tick():
  调用 EnableUseCase::execute(mgr, group, AxisId::X, true)
    IF 成功               → WaitingEnabled
    IF 失败               → Error(UseCaseError)
```

### 3.3 WaitingEnabled → Coupling / Error
```
tick():
  IF x->state() == Idle   → Coupling
  IF x->state() == Error  → Error(x->lastRejection())
  ELSE                    → 保持 WaitingEnabled
```

### 3.4 Coupling → WaitingCoupled / Error
```
tick():
  调用 gantry.requestCouple(true):
    IF 返回 None          → 下发 pendingCommand（若有）→ WaitingCoupled
    IF 返回其他拒绝        → Error(GantryRejection)
```

### 3.5 WaitingCoupled → Done / Error

```
tick():
  优先级 1: IF gantry.isCoupled()
            → Done

  优先级 2: IF gantry.hasError()
            → Error(gantry.getLastError())

  优先级 3: IF !gantry.isCouplingRequested() && !gantry.isCoupled()
            → Error(GantryRejection::UnexpectedFeedback)
            // PLC 静默否决联动: errorCode=0 但 isCoupled=false,
            //           状态机已从 CouplingRequested 退回到 Decoupled

  优先级 4: ELSE → 保持 WaitingCoupled
```

**死锁修复说明**：原代码只有条件 1 和 2。当 PLC 返回 `{isCoupled=false, errorCode=0}` 时，`isCoupled()=false` 且 `hasError()=false`，两个条件均不满足 → 永久卡死。条件 3 检测「状态机已退出 CouplingRequested 但未到 Coupled」这一异常模式。

### 3.6 Decoupling → WaitingDecoupled / Error
```
tick():
  调用 gantry.requestCouple(false):
    IF 返回 None          → 下发 pendingCommand（若有）→ WaitingDecoupled
    IF 返回其他拒绝        → Error(GantryRejection)
```

> **幂等场景**：已解耦时 `requestCouple(false)` 返回 `None` 但不生成 `pendingCommand`，直接进入 `WaitingDecoupled`，下帧立刻退出到 `Done`。

### 3.7 WaitingDecoupled → Done / Error

```
tick():
  优先级 1: IF !gantry.isCoupled()
            && !gantry.isCouplingRequested()
            && !gantry.isDecouplingRequested()
            → Done

  优先级 2: IF gantry.hasError()
            → Error(gantry.getLastError())

  优先级 3: IF gantry.isCoupled()
            && !gantry.isDecouplingRequested()
            → Error(GantryRejection::UnexpectedFeedback)
            // PLC 静默否决解耦: errorCode=0 但 isCoupled=true,
            //           状态机已从 DecouplingRequested 退回到 Coupled

  优先级 4: ELSE → 保持 WaitingDecoupled
```

**死锁修复说明**：原代码只有条件 1 和 2。当 PLC 返回 `{isCoupled=true, errorCode=0}` 时，`isCoupled()=true` → 条件 1 不满足，`hasError()=false` → 条件 2 不满足 → 永久卡死。条件 3 检测「状态机已退出 DecouplingRequested 但未到 Decoupled」这一异常模式。

### 3.8 Done / Error（终态）
```
Done:
  tick() → 保持 Done（幂等）
  可重新调用 startCoupling()/startDecoupling() 发起新流程

Error:
  tick() → 保持 Error（幂等）
  m_lastError 保留最后错误
  调用方通过 startCoupling()/startDecoupling() 重新发起
```

## 五、流转约束总表

| 当前状态             | tick() 驱动条件                                      | 目标状态         | 错误类型               |
| -------------------- | ---------------------------------------------------- | ---------------- | ---------------------- |
| Initial              | —                                                    | Initial          | —                      |
| EnsuringEnabled      | EnableUseCase 成功                                   | WaitingEnabled   | —                      |
| EnsuringEnabled      | EnableUseCase 失败                                   | Error            | UseCaseError           |
| WaitingEnabled       | X 轴 Idle                                            | Coupling         | —                      |
| WaitingEnabled       | X 轴 Error                                           | Error            | 轴 Rejection           |
| WaitingEnabled       | 其他                                                 | WaitingEnabled   | —                      |
| Coupling             | requestCouple(true) = None                           | WaitingCoupled   | —                      |
| Coupling             | requestCouple(true) ≠ None                           | Error            | GantryRejection        |
| **WaitingCoupled**   | **isCoupled() = true**                               | **Done**         | —                      |
| **WaitingCoupled**   | **hasError() = true**                                | **Error**        | **GantryRejection**    |
| **WaitingCoupled**   | **!isCouplingReq() ∧ !isCoupled()**                  | **Error**        | **UnexpectedFeedback** |
| Decoupling           | requestCouple(false) = None                          | WaitingDecoupled | —                      |
| Decoupling           | requestCouple(false) ≠ None                          | Error            | GantryRejection        |
| WaitingDecoupled     | !isCoupled() ∧ !isCouplingReq() ∧ !isDecouplingReq() | Done             | —                      |
| **WaitingDecoupled** | **hasError() = true**                                | **Error**        | **GantryRejection**    |
| **WaitingDecoupled** | **isCoupled() ∧ !isDecouplingReq()**                 | **Error**        | **UnexpectedFeedback** |
| Done                 | —                                                    | Done             | —                      |
| Error                | —                                                    | Error            | —                      |

**粗体行**为相对于原始代码的修正。

## 六、入口拦截总表

| 入口函数            | 守卫条件                   | 拒绝时结果                              |
| ------------------- | -------------------------- | --------------------------------------- |
| `startCoupling()`   | tryGetGroup 失败           | Error(ContextRejection)                 |
| `startCoupling()`   | gantry.isNotSynchronized() | Error(GantryRejection::NotSynchronized) |
| `startDecoupling()` | tryGetGroup 失败           | Error(ContextRejection)                 |
| `startDecoupling()` | gantry.isNotSynchronized() | Error(GantryRejection::NotSynchronized) |

> **已移除** `startDecoupling()` 的 `!isCoupled()` 守卫——领域层 GantryGroup 已幂等静默处理。

## 七、分层职责总结

| 层级                   | 角色       | 重复调用处理                                                 |
| ---------------------- | ---------- | ------------------------------------------------------------ |
| **GantryGroup**        | 领域逻辑层 | 幂等静默处理：已联动再次联动 → 返回 None 不生成命令；已解耦再次解耦 → 同 |
| **GantryOrchestrator** | 编排层     | 不重复幂等判断，只拦截会产生外部副作用的危险状态（NotSynchronized）；负责检测 PLC 异常反馈（UnexpectedFeedback） |
| **EnableUseCase**      | 用例层     | 自身幂等：已使能再次使能 → 返回成功                          |
