## 分析报告：`SystemContext::m_isGantryCoupled` 问题诊断与解决方案

---

### 一、问题的存在性确认

**是的，存在你描述的问题。** `SystemContext::m_isGantryCoupled` 确实剥夺（或者说架空）了 `GantryGroup` 的联动状态决策权。

---

### 二、问题解剖

#### 2.1 两套状态系统并存

| 组件            | 状态变量                               | 驱动来源                                 | 语义                           |
| --------------- | -------------------------------------- | ---------------------------------------- | ------------------------------ |
| `SystemContext` | `m_isGantryCoupled` (bool)             | 手动 `setCoupledState()`                 | 决定 `tryGetAxis` 的轴访问控制 |
| `GantryGroup`   | `m_state` (GantryCouplingState 四态机) | PLC 反馈 `applyFeedback(GantryFeedback)` | 真实的物理联动状态             |

#### 2.2 关键断点：`tryGetAxis` 查询的是错误的状态源

```cpp
// SystemContext.h 第 33 行
bool tryGetAxis(AxisId id, Axis*& outAxis, ContextRejection& reason) {
    if (m_isGantryCoupled) {          // ← 使用 SystemContext 自己的 bool
        if (id == AxisId::X1 || id == AxisId::X2) { ... }
    } else {
        if (id == AxisId::X) { ... }
    }
    ...
}
```

应该查询的是：
```cpp
if (m_gantry->isCoupled()) {          // ← 应该使用 GantryGroup 的状态机
```

#### 2.3 两个状态从不自动同步

看 `GantryOrchestrator::tick()` 完成联动流程后的收尾：

```cpp
case Step::WaitingCoupled:
    if (gantry.isCoupled()) {        // GantryGroup 已经通过 applyFeedback 进入 Coupled
        m_step = Step::Done;
    }
    // ⚠️ 从未调用 group->setCoupledState(true) 来同步 m_isGantryCoupled！
```

反过来看 `requestCouple(false)` 解耦成功后：
```cpp
// GantryOrchestrator::tick() WaitingDecoupled:
// GantryGroup 进入 Decoupled，但 m_isGantryCoupled 仍然是 true！
```

**结论：** `GantryGroup` 的内部状态随 PLC 反馈正确流转，但 `SystemContext::m_isGantryCoupled` 停在了初始值（true）或手动测试设置的值，导致 `tryGetAxis` 的访问控制与物理现实脱节。

#### 2.4 测试代码大量绕过状态机直接操纵 `setCoupledState`

所有测试（10+ 处调用）目前通过 `setCoupledState()` 直接设置访问控制，完全绕过了 `GantryGroup` 的状态机和 `GantryOrchestrator` 的编排流程：

| 测试文件                         | 调用次数 | 当前行为                              |
| -------------------------------- | -------- | ------------------------------------- |
| `test_system_context.cpp`        | 3        | 测试 `tryGetAxis` 的联动/解耦访问控制 |
| `test_enable_usecase.cpp`        | 3        | 模拟联动/解耦状态                     |
| `test_jog_usecase.cpp`           | 1        | 模拟解耦                              |
| `test_move_absolute_usecase.cpp` | 2        | 模拟联动/解耦                         |
| `test_move_relative_usecase.cpp` | 2        | 模拟联动/解耦                         |
| `test_stop_usecase.cpp`          | 2        | 模拟联动/解耦                         |

这些测试手动"注入"状态，而不是让 `GantryOrchestrator` + PLC 反馈自然产生状态变更。

### 三、为什么会出现这个设计问题（历史演进推测）

1. **早期版本**：只有 `m_isGantryCoupled` bool，用于轴访问控制
2. **后期引入**：`GantryGroup` + `GantryCouplingState` 四态机 + `GantryOrchestrator` 为完整编配
3. **断点**：`tryGetAxis` 从未被重构为查询 `GantryGroup::isCoupled()`，形成了两套并行的状态系统

### 四、解决方案清单

#### 4.1 核心原则

**唯一状态源原则**：`GantryGroup::m_state`（`GantryCouplingState`）是龙门联动状态的唯一真实来源。`SystemContext` 不应维护独立的 `m_isGantryCoupled`。

#### 4.2 SystemContext 修改

**删除：**
- `bool m_isGantryCoupled` 成员变量
- `bool isGantryCoupled() const` 方法
- `void setCoupledState(bool coupled)` 方法

**重构 `tryGetAxis`：**
- 联动状态查询源从 `m_isGantryCoupled` → `m_gantry->isCoupled()`
- `tryGetAxis` 的联动/解耦访问控制逻辑不变，仅数据源变更

#### 4.3 测试修改策略

测试分两类处理：

**A 类测试（测试 `tryGetAxis` 访问控制本身）**——如 `test_system_context.cpp`：
- 改为通过 `GantryGroup::applyFeedback()` 设置真实状态
- 例如：联动时 → `gantry.applyFeedback({.isCoupled = true, .errorCode = 0})`
- 例如：解耦时 → `gantry.applyFeedback({.isCoupled = false, .errorCode = 0})`

**B 类测试（UseCase 层行为测试）**——如 `test_enable_usecase.cpp` 等：
- 改为通过 `GantryGroup::applyFeedback()` 来设定前置条件
- 不再使用 `ctx->setCoupledState(true/false)`

**C 类测试（SystemManager 状态隔离测试）**——如 `test_system_manager.cpp`：
- 改为通过 `ctx->gantry().applyFeedback()` 来操纵各组独立状态
- 保持多组隔离的行为验证不变

#### 4.4 GantryOrchestrator 不做修改

当前的 `GantryOrchestrator` 已经正确地将所有流程状态变更通过 `gantry.applyFeedback()` 驱动 `GantryCouplingState`。不需要修改。

### 五、多组 + 多轴状态下的联动状态声明与流程控制

#### 5.1 每组的联动状态声明

```
每个 SystemContext（组）拥有独立的 GantryGroup 实例：
- GroupA → GantryGroup_A → 维护 GroupA 的 X1/X2 联动状态
- GroupB → GantryGroup_B → 维护 GroupB 的 X1/X2 联动状态
```

**状态查询接口**（唯一的、不重复的）：
| 查询路径                                | 含义                                  |
| --------------------------------------- | ------------------------------------- |
| `ctx->gantry().isCoupled()`             | 该组当前是否处于物理联动状态          |
| `ctx->gantry().isCouplingRequested()`   | 该组联动请求是否已发送但 PLC 尚未确认 |
| `ctx->gantry().isDecouplingRequested()` | 该组解耦请求是否已发送但 PLC 尚未确认 |
| `ctx->gantry().hasError()`              | 该组龙门是否有错误                    |
| `ctx->tryGetAxis(AxisId::X, ...)`       | 联动时可访问逻辑轴 X，解耦时不可访问  |
| `ctx->tryGetAxis(AxisId::X1, ...)`      | 联动时不可访问物理轴 X1，解耦时可访问 |
| `ctx->tryGetAxis(AxisId::X2, ...)`      | 联动时不可访问物理轴 X2，解耦时可访问 |

#### 5.2 联动流程（标准路径）

```
1. 外部触发 GantryOrchestrator::startCoupling()
2. tick() 逐帧推进：
   EnsuringEnabled  →  通过 EnableUseCase 使能逻辑轴 X（PLC 内部同时使能 X1/X2 电机）
   WaitingEnabled   →  等待 X 轴反馈为 Idle
   Coupling         →  调用 gantry.requestCouple(true)，生成 GantryCommand
                    →  通过 ISystemDriver::sendGantry() 下发到 PLC
   WaitingCoupled   →  等待 PLC 反馈 GantryFeedback{ isCoupled=true }
                    →  gantry.applyFeedback() 驱动状态机进入 Coupled
   Done             →  联动完成
3. 此后：SystemContext::tryGetAxis 查询 gantry.isCoupled() → true
   → 物理轴 X1/X2 被锁定，仅逻辑轴 X 可操作
```

#### 5.3 解耦流程（标准路径）

```
1. 外部触发 GantryOrchestrator::startDecoupling()
2. tick() 逐帧推进：
   Decoupling       →  调用 gantry.requestCouple(false)，生成 GantryCommand
                    →  通过 ISystemDriver::sendGantry() 下发到 PLC
   WaitingDecoupled →  等待 PLC 反馈 GantryFeedback{ isCoupled=false }
                    →  gantry.applyFeedback() 驱动状态机进入 Decoupled
   Done             →  解耦完成
3. 此后：SystemContext::tryGetAxis 查询 gantry.isCoupled() → false
   → 逻辑轴 X 被锁定，物理轴 X1/X2 可独立操作
```

#### 5.4 多组并行的独立性

```
GroupA 联动 ≠ GroupB 联动
- GroupA 可处于 Coupled，GroupB 同时处于 Decoupled
- 每个组独立通过自己的 GantryOrchestrator 和 GantryGroup 管理
- 互不干扰，因为 GantryGroup 实例完全隔离
```

### 六、总结

| 项目         | 当前状态                                      | 目标状态                                              |
| ------------ | --------------------------------------------- | ----------------------------------------------------- |
| 联动状态源   | `SystemContext::m_isGantryCoupled`            | `GantryGroup::m_state` (唯一来源)                     |
| 状态同步     | 手动 `setCoupledState()`，与 GantryGroup 脱钩 | 仅通过 `GantryGroup::applyFeedback()` 从 PLC 反馈驱动 |
| 测试注入方式 | `ctx->setCoupledState(true/false)`            | `ctx->gantry().applyFeedback({...})`                  |
| 访问控制查询 | `m_isGantryCoupled`                           | `m_gantry->isCoupled()`                               |

核心修改范围：
1. **删除** `SystemContext` 的 `m_isGantryCoupled`、`isGantryCoupled()`、`setCoupledState()`
2. **重构** `tryGetAxis` 查询源为 `m_gantry->isCoupled()`
3. **更新** 所有测试从 `setCoupledState()` 改为 `gantry().applyFeedback()`

这是最小化、最聚焦的修改——不动 GantryGroup、GantryCouplingState、GantryOrchestrator 和 UseCase 层的任何逻辑，仅消除重复的状态源。
