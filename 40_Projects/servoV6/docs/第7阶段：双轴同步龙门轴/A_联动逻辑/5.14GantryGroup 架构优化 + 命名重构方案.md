# GantryGroup 架构优化 + 命名重构方案

---

## 一、问题诊断

### 1.1 `Axis& x1/x2` 依赖分析

当前 `GantryGroup::requestCouple()` 对 `Axis&` 的唯一实际使用：

```cpp
// 安全屏障：任一轴处于 Error 状态，拒绝联动
if (m_x1.state() == AxisState::Error || m_x2.state() == AxisState::Error) {
    return GantryRejection::AxisStateError;
}
```

**这个检查的冗余性**：

| 检查项目          | 谁做               | 结果                                 |
| ----------------- | ------------------ | ------------------------------------ |
| X1 是否使能       | PLC（errorCode=2） | 联动失败，反馈 `Gantry_Error_Code=2` |
| X2 是否使能       | PLC（errorCode=3） | 联动失败，反馈 `Gantry_Error_Code=3` |
| X1 是否静止       | PLC（errorCode=4） | 联动失败，反馈 `Gantry_Error_Code=4` |
| X2 是否静止       | PLC（errorCode=5） | 联动失败，反馈 `Gantry_Error_Code=5` |
| 是否超差          | PLC（errorCode=1） | 联动失败，反馈 `Gantry_Error_Code=1` |
| **轴 Error 状态** | **上位机 Domain**  | **提前拒绝，不发送命令**             |

**结论**：上位机做的唯一额外检查是「轴 Error 状态提前拦截」，但：
- PLC 的 `X1NotEnabled` / `X2NotEnabled` 已经覆盖「电机异常」场景
- 轴 Error 状态本质上是上位机侧的聚合态，PLC 不感知
- 如果 X1/X2 处于 Error，PLC 收到联动命令后也会因为 `X1NotEnabled` 等拒绝，形成冗余校验
- **删除此检查不会损失安全性**，PLC 是最终安全裁决者

移除后 `GantryGroup` 不再需要持有任何 `Axis` 引用，成为纯粹的 **耦合状态机 + PLC 反馈消费者**。

### 1.2 命名问题

| 当前名称                | 问题                                              | 推荐名称                     | 理由                     |
| ----------------------- | ------------------------------------------------- | ---------------------------- | ------------------------ |
| `GantryGroup`           | 结构词（Group），无法表达职责                     | `GantryCouplingController`   | 精确表达「联动耦合控制」 |
| `GantryMotorController` | Controller 语义泛化，无法区分运动控制 vs 上电控制 | `GantryServoPowerController` | 精确表达「伺服上下电」   |

## 二、修改步骤（共 8 步）

### Step 1：移除 `GantryCouplingController` 对 `Axis&` 的依赖

**操作**：编辑 `domain/gantry/GantryGroup.h`

**修改内容**：

```cpp
// === 删除 ===
#include "entity/Axis.h"

// === 构造函数变更 ===
// 旧：
GantryGroup(Axis& x1, Axis& x2) : m_x1(x1), m_x2(x2) {}
// 新：
GantryGroup() = default;

// === 删除成员 ===
// 旧：
Axis& m_x1;
Axis& m_x2;

// === requestCouple 中删除 Axis 状态检查 ===
// 删除以下 5 行：
if (m_x1.state() == AxisState::Error || m_x2.state() == AxisState::Error) {
    return GantryRejection::AxisStateError;
}
```

**理由**：PLC 通过 `errorCode` 反馈覆盖所有物理安全校验，上位机无需重复。

---

### Step 2：删除 `GantryRejection::AxisStateError`

**操作**：编辑 `domain/gantry/GantryRejection.h`

```cpp
// 删除此行：
AxisStateError = 101,
```

**理由**：该拒绝原因的唯一触发点（`GantryGroup::requestCouple` 中检查 `AxisState::Error`）已移除。`NotSynchronized = 102` 应顺延为 `101`，或保留编号空洞（不影响 PLC 通信，PLC 错误码 0-5 不受影响）。

**推荐**：保留编号空洞 `101`，后续可扩展其他领域拦截。

**操作**：编辑

### Step 7：更新基础设施层

**`ISystemDriver.h`**：
```cpp
// 旧
virtual void sendGantry(const GantryCommand& cmd) = 0;
virtual void sendGantryMotor(const MotorCommand& cmd) = 0;
// 新
virtual void sendGantryCoupling(const GantryCouplingCommand& cmd) = 0;
virtual void sendServoPower(const ServoPowerCommand& cmd) = 0;
```

**`FakeAxisDriver.h`**：
```cpp
void sendGantryCoupling(const GantryCouplingCommand& cmd) override { ... }
void sendServoPower(const ServoPowerCommand& cmd) override {
    plc.onCommand(AxisId::X1, EnableCommand{cmd.enable});
    plc.onCommand(AxisId::X2, EnableCommand{cmd.enable});
}
```

## 四、架构收益

```
修复前：
  GantryGroup ──[持有]──→ Axis& x1/x2 ──[可能反向依赖]──→ Gantry 状态
  ↑ 聚合根之间直接引用，隐患

修复后：
  GantryCouplingController ──[仅依赖]──→ GantryCouplingState（值对象）
  GantryServoPowerController ──[仅依赖]──→ GantryFeedback（DTO）
  ↑ 两个控制器完全解耦，各自独立

PLC 是唯一安全裁决者：
  上位机下发意图 → PLC 校验 → PLC 反馈结果 → 上位机同步状态
```

## 五、建议实施顺序

1. **先做 Step 1+2**（移除 `Axis&` 依赖 + 删除 `AxisStateError`）—— 这是架构修正，与命名无关
2. **再做 Step 3+4**（重命名两个类）—— 纯重命名，不改逻辑
3. **再做 Step 5~8**（更新所有引用 + 测试）—— 跟随重命名
4. **最后更新 `tests/CMakeLists.txt`**—— 注册新文件名
