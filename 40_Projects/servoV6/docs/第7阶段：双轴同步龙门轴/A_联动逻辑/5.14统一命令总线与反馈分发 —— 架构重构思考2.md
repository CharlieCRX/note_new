# 统一命令总线与反馈分发 —— 架构重构思考

------

# 一、背景

随着 servoV6 从：

```text
单轴控制
```

逐渐演化为：

```text
多轴 + 龙门联动 + 电机使能 + 安全控制
```

系统中的“控制意图（Command）”数量开始快速增长。

当前已经存在：

- `AxisCommand`
- `GantryCouplingCommand`
- `ServoPowerCommand`

未来还可能增加：

- `EmergencyStopCommand`
- `AlarmResetCommand`
- `SafetyCommand`
- `IOCommand`

在这个阶段，原有的：

```cpp
IAxisDriver
```

结构已经开始暴露架构问题。

------

# 二、当前结构

------

## 2.1 当前接口结构

```cpp
class IAxisDriver {
public:
    virtual void send(AxisId id,
                      const AxisCommand& cmd) = 0;
};
```

后来为了支持龙门控制，又新增：

```cpp
class ISystemDriver : public IAxisDriver {
public:
    virtual void sendGantry(
        const GantryCouplingCommand& cmd) = 0;
};
```

未来如果继续扩展，还会出现：

```cpp
sendServoPower(...)
sendAlarm(...)
sendSafety(...)
sendIO(...)
```

------

## 2.2 当前结构的问题

当前 Driver 接口：

```text
按“聚合对象”扩展
```

而不是：

```text
按“控制意图”抽象
```

也就是说：

| 当前结构      | 实际应该关注                  |
| ------------- | ----------------------------- |
| Axis 发什么   | 系统要执行什么命令            |
| Gantry 发什么 | Move / Stop / Enable / Couple |
| 哪个聚合产生  | 控制意图本身                  |

------

# 三、当前结构存在的问题

------

# 3.1 Interface Patchwork（接口补丁化）

当前结构已经出现：

```cpp
sendAxis(...)
sendGantry(...)
```

这种：

```text
“按功能不断追加接口”
```

的趋势。

如果继续扩展：

```cpp
class ISystemDriver {
    sendAxis(...)
    sendGantry(...)
    sendServo(...)
    sendSafety(...)
    sendAlarm(...)
};
```

最终：

```text
Driver Interface ≈ 系统功能菜单
```

这是典型的：

```text
Fat Interface（胖接口）
```

问题。

------

# 3.2 Infrastructure 感知 Domain 聚合

当前：

```cpp
sendGantry(...)
```

意味着：

```text
Infrastructure 知道：
系统存在 Gantry 聚合
```

但实际上：

```text
PLC 并不关心聚合
```

PLC 真正关心的是：

- Enable
- Move
- Stop
- Couple
- ResetAlarm

这些：

```text
控制意图（Intent）
```

而不是：

```text
命令来自哪个聚合
```

------

# 3.3 命令体系开始碎片化

当前：

```cpp
AxisCommand
GantryCouplingCommand
ServoPowerCommand
```

未来：

```cpp
EmergencyStopCommand
SafetyCommand
AlarmResetCommand
```

如果每种命令：

```text
对应一个 sendXxx()
```

那么 Driver 接口将持续膨胀。

最终：

```text
系统越来越难维护
```

------

# 四、问题本质

当前系统已经开始进入：

# “命令驱动架构（Intent-Driven Architecture）”

也就是说：

```text
Domain 不再直接操作 PLC

而是：
产生控制意图（Command）
```

因此：

```text
Command
```

已经成为：

```text
一级架构元素
```

而不是：

```text
Axis 的附属结构
```

------

# 五、重构目标

本次重构目标：

------

## 1. 建立统一命令边界

形成：

```text
Domain Intent
    ↓
Unified Command Boundary
    ↓
Infrastructure
```

------

## 2. Driver 不再感知聚合来源

Infrastructure 不应该知道：

```text
命令来自 Axis
还是 Gantry
```

它只应该知道：

```text
系统需要发送什么控制命令
```

------

## 3. 保持 Driver 接口稳定

新增命令时：

```text
不再修改 Driver 接口
```

避免：

```text
Interface Patchwork
```

------

# 六、推荐方案：统一命令总线

------

# 6.1 定义统一命令边界

```cpp
using SystemCommand = std::variant<
    AxisCommand,
    GantryCouplingCommand,
    ServoPowerCommand
>;
```

------

# 6.2 为什么采用 variant

这里必须明确：

采用：

```cpp
std::variant
```

并不是因为：

```text
更符合经典 OCP
```

因为：

```text
variant 是封闭集合
```

新增命令：

```text
必须修改 variant 定义
```

严格来说：

```text
它不是开放式插件结构
```

------

## 工业控制为什么适合 variant

工业控制系统通常具有：

------

### 1. 命令集合有限

例如：

- Move
- Stop
- Jog
- Enable
- Couple

这些命令：

```text
长期稳定
```

------

### 2. 更强调确定性

工业控制更关注：

- 编译期检查
- 无 RTTI
- 无 dynamic_cast
- 无运行期类型失败
- 更可预测的行为

------

### 3. 希望新增命令时强制检查所有处理逻辑

例如新增：

```cpp
EmergencyStopCommand
```

编译器会强制：

```text
所有 visitor 必须处理它
```

这对于工业控制：

```text
非常重要
```

------

# 七、统一 Driver 接口

------

## 当前结构

```cpp
sendAxis(...)
sendGantry(...)
```

------

## 推荐结构

```cpp
class ISystemDriver {
public:
    virtual ~ISystemDriver() = default;

    virtual void send(
        const SystemCommand& cmd) = 0;
};
```

------

# 八、Driver 实现方式

```cpp
void PlcDriver::send(
    const SystemCommand& cmd)
{
    std::visit([this](auto&& c) {
        handle(c);
    }, cmd);
}
```

------

然后：

```cpp
handle(const AxisCommand&)
handle(const GantryCouplingCommand&)
handle(const ServoPowerCommand&)
```

负责：

```text
Command → PLC寄存器
```

转换。

------

# 九、该结构带来的收益

------

# 9.1 Driver 不再感知聚合来源

Driver：

```text
不再关心：
命令来自 Axis 还是 Gantry
```

只关心：

```text
系统需要执行什么控制意图
```

------

# 9.2 Driver 接口保持稳定

未来新增：

```cpp
EmergencyStopCommand
```

不需要：

```text
新增 sendEmergencyStop()
```

只需要：

```cpp
SystemCommand += EmergencyStopCommand
```

即可。

------

# 9.3 命令体系统一

形成：

```text
Domain
    ↓
SystemCommand
    ↓
ISystemDriver
    ↓
PLC
```

统一命令流。

------

# 十、反馈方向分析

------

# 10.1 当前反馈结构

当前：

```cpp
GantryFeedback feedback;

power.applyFeedback(feedback);
coupling.applyFeedback(feedback);
```

------

# 10.2 当前结构的优点

------

## DTO 已独立

```cpp
GantryFeedback
```

不是某个聚合内部结构。

这是正确的。

------

## 多消费者共享同一快照

多个 Controller：

```text
共享同一帧 PLC 数据
```

这是正确的数据一致性策略。

------

## 依赖方向正确

```text
PLC → Domain
```

属于：

```text
外层驱动内层
```

符合 Clean Architecture。

------

# 10.3 当前反馈结构的问题

------

## 分发责任散落

轮询线程需要知道：

```text
哪些对象需要 feedback
```

未来：

- Safety
- Alarm
- Monitor

增加后：

```text
轮询线程会逐渐变成 Feedback Coordinator
```

------

## 缺少统一反馈入口

当前：

```text
反馈是点对点调用
```

没有：

```text
FeedbackDispatcher
```

概念。

------

## 调用顺序隐式

例如：

```cpp
power.applyFeedback();
coupling.applyFeedback();
```

顺序未定义。

未来可能产生：

```text
隐藏状态依赖
```

问题。

------

# 十一、为什么当前不建议重构反馈系统

当前消费者数量：

```text
非常少（2~3 个）
```

如果现在引入：

- FeedbackBus
- Dispatcher
- Observer

会导致：

```text
复杂度大于收益
```

因此：

------

## 当前推荐策略

```text
优先统一命令边界
```

反馈系统：

```text
保持简单
```

------

# 十二、IAxisDriver 是否保留

------

# 当前问题

```cpp
IAxisDriver
```

与：

```cpp
ISystemDriver
```

已经开始语义重叠。

因为：

```cpp
send(AxisId, AxisCommand)
```

本身已经属于：

```text
SystemCommand
```

的一部分。

------

# 推荐方案

------

## 推荐一（更干净）

直接移除：

```cpp
IAxisDriver
```

统一：

```cpp
ISystemDriver::send(SystemCommand)
```

------

# 十三、最终推荐架构

------

# Domain

产生：

```text
AxisCommand
GantryCouplingCommand
ServoPowerCommand
```

------

# Unified Boundary

```cpp
SystemCommand
```

------

# Infrastructure

```cpp
ISystemDriver
    send(SystemCommand)
```

------

# Adapter

```cpp
std::visit(...)
```

------

# PLC

执行：

```text
寄存器写入
```

------

# 十四、最终结论

当前问题本质不是：

```text
多了一个 sendGantry()
```

而是：

# 系统缺少统一命令边界

真正需要建立的是：

```text
Domain Intent
    ↓
Unified Command Boundary
    ↓
Infrastructure Physicalization
```

而不是：

```text
按聚合不断扩展 Driver 接口
```

------

# 十五、核心原则

最终：

```text
Infrastructure 不应该感知：
命令来自哪个聚合
```

它只应该知道：

```text
系统需要执行什么控制意图
```

这是本次重构最核心的架构原则。
