# 合并 `GantryMotorController` 到 `GantryGroup` 分析文档

---

## 一、合并的论点

### 1.1 支持合并的理由

| 理由                 | 说明                                                         |
| -------------------- | ------------------------------------------------------------ |
| **业务语义内聚**     | 龙门「使能-联动-解耦-掉电」是一条完整的生命周期链。一个 PLC 返回三个寄存器（轴X状态显示、轴X联动状态、Gantry_Error_Code），被拆成两个类接收反馈是反直觉的 |
| **命令通道统一**     | 当前已有 `GantryCommand{ bool enableCoupling }`，再加 `MotorCommand{ bool enable }`，两者通过同一个 `ISystemDriver` 下发。合并后 `GantryCommand` 可扩展为 `{ bool enable; bool enableCoupling }`，或保留为两个可选的 pending intent，避免调用方需要依次弹出两个不同类的命令 |
| **反馈接口统一**     | `GantryFeedback` 目前只含 `isCoupled` + `errorCode`，缺少「使能状态」信息。若新增独立类，PLC 轮询线程需要分别调用 `motor.applyMotorFeedback()` 和 `gantry.applyFeedback()`，分两次跨两个对象，增加时序不一致的风险 |
| **既存耦合已经存在** | `GantryOrchestrator` 同时依赖 `GantryMotorController` 和 `GantryGroup`，实际上它只需要龙门这个单一聚合根 |

### 1.2 反对合并的理由

| 理由                        | 说明                                                         |
| --------------------------- | ------------------------------------------------------------ |
| **TDD 测试覆盖重做**        | `GantryGroup` 已有完整的 `test_gantry_group.cpp`（~ 行）和 `test_gantry_coupling_state.cpp`（~ 行）。合并将需要重写大部分测试 |
| **单一职责原则（SRP）边界** | 「控制电机使能」和「控制联动状态机」是不同的寄存器操作，有各自的幂等逻辑、各自的初始同步点（`NotSynchronized` vs `m_synchronized`） |
| **复用之虞**                | 若将电机控制固定在 `GantryGroup` 内部，场景 2「使能解耦」（只使能电机、不联动）时调用方仍需通过 `GantryGroup` 间接操作，语义上「我不需要龙门联动，却必须和龙门聚合根打交道」，有微妙的命名异味 |

## 二、架构判定：**反对合并，保留独立类**

基于 Clean Architecture 和 Clean Code 的评估：

### 2.1 单一职责原则（SRP）：两个独立的「变更原因」

```
GantryMotorController 的变更原因:
  └─ PLC 寄存器「使能轴X电机」的协议变化
     （例如：将来增加伺服准备好标志位、电流反馈等）

GantryGroup 的变更原因:
  └─ 联动状态机逻辑变化（NotSynchronized / Coupled / Decoupled / ...）
  └─ Gantry_Error_Code 错误码映射表扩展
  └─ 联动/解耦命令的幂等语义调整
```

两者各自只有一个变更原因，合并后变成一个类有两个变更原因，违反 SRP。

### 2.2 接口隔离原则（ISP）：合并后调用方被迫依赖不需要的东西

| 调用方                                | 需要的接口                            | 合并后的代价                                                 |
| ------------------------------------- | ------------------------------------- | ------------------------------------------------------------ |
| `GantryOrchestrator::EnsuringEnabled` | `requestEnable(bool)` + `isEnabled()` | 同时暴露 `requestCouple(bool)` / `isCoupled()` 等——无需但可误用 |
| 场景 2 外部调用                       | `requestEnable(true)`                 | 被迫持有 `GantryGroup` 引用（而 `GantryGroup` 构造需要 `Axis& x1, Axis& x2`），语义含混 |
| `SystemContext::tryGetAxis`           | `isNotSynchronized()` / `isCoupled()` | 同时暴露 `requestEnable(bool)` — 轴访问逻辑不应触碰电机控制  |

### 2.3 依赖倒置原则（DIP）：当前方向正确

```
高层：  GantryOrchestrator
         ├──→ GantryMotorController  (控制电机)
         └──→ GantryGroup            (控制联动状态机)

低层：  ISystemDriver (send 仅传输命令 bytes，不关心语义)
```

两个类各自是独立的意图生成者（intent producer），分别生成「使能命令」和「联动命令」。合并后只有一个意图生成者，但 `GantryOrchestrator` 在 `EnsuringEnabled` 步需要使能却在 `Coupling` 步才需要联动——合并后需要分步弹出两种命令，逻辑复杂度不变甚至增加。

### 2.4 当前 `GantryFeedback` 不全的问题

用户指出 `GantryFeedback` 缺少 `enable` 字段。**这是需要修改之处，但不是合并的理由。** 解决方案是扩展 `GantryFeedback`，让 PLC 轮询线程**一次性读取全部三个寄存器，分别推送给两个对象**，或对 Architecture 做个轻微调整（见第 3 节）。

## 三、推荐方案：合并反馈，不合并类

### 3.1 保留两个独立领域类

```
domain/gantry/GantryMotorController.h   ← 新增
domain/gantry/GantryGroup.h             ← 保持现有（小幅扩展）
domain/gantry/GantryCouplingState.h     ← 保持现有
domain/gantry/GantryRejection.h         ← 扩展
domain/gantry/GantryFeedback.h          ← 新增独立 DTO 文件（从 GantryGroup.h 抽离）
```

### 3.2 统一反馈 DTO

将 `GantryFeedback` 从 `GantryGroup.h` 中抽离到独立文件，并扩展 `enable` 字段：

```cpp
// domain/gantry/GantryFeedback.h   ← 新文件
#pragma once

struct GantryFeedback {
    bool enable;        // PLC 寄存器「轴X状态显示」1=电机使能, 0=掉电
    bool isCoupled;     // PLC 寄存器「轴X联动状态」
    int  errorCode;     // PLC 寄存器「Gantry_Error_Code」
};
```

**设计意图**：PLC 轮询线程一次读取三个寄存器，构造单个 `GantryFeedback` 结构体，**同时**推送给两个对象：

```cpp
// PLC 轮询线程/驱动层
void pollAndApplyFeedback() {
    GantryFeedback fb = readGantryRegistersFromPLC();
    // ① 电机控制器（追踪使能状态）
    gantryMotor.applyFeedback(fb);
    // ② 龙门聚合根（驱动联动状态机）
    gantryGroup.applyFeedback(fb);
    // ③ 同时驱动 Axis::X 实体的 AxisState（它需要从 fb.enable 推导）
    axisX.applyMotorFeedback(fb.enable);
}
```

这避免了两次读寄存器的时序问题，保持了类的独立性。

### 3.3 `SystemContext` 暴露两个对象

```cpp
class SystemContext {
public:
    // 现有
    GantryGroup& gantry()      { return *m_gantry; }

    // 新增：独立于联动状态，任何时候均可访问
    GantryMotorController& gantryMotor() { return *m_gantryMotor; }

    // tryGetAxis 不变（X/X1/X2 暴露逻辑仍由 GantryGroup 裁决）

private:
    std::unique_ptr<GantryMotorController> m_gantryMotor;  // 新增
    std::unique_ptr<GantryGroup> m_gantry;                 // 现有
    // ...
};
```

## 六、结论

**保留独立类 `GantryMotorController`，与 `GantryGroup` 并列存在于 `domain/gantry/` 目录下。** 理由：

1. **Clean Architecture 的 SRP**：电机使能和联动状态机是两个独立寄存器、独立协议，各自有独立的幂等语义和同步标记。合并后违反单一职责原则。
2. **Clean Code 的「类应该小巧」**：`GantryGroup` 已经包含状态查询、错误查询、意图生成、意图弹出、反馈接收、错误翻译 6 个职责小组。再加电机控制将使其变成上帝类。
3. **测试重做代价不匹配收益**：合并带来的唯一真实收益是「少写一个 applyFeedback 调用」，而代价是重写已有的 TDD 测试和破坏既有的职责边界。
4. **统一反馈但分散处理**是常见的 Port-Adapter 模式：同一个 DTO 推送至多个领域对象，各取所需。

唯一的收敛点是：**将 `GantryFeedback` 抽离为独立 DTO**，包含 `enable`/`isCoupled`/`errorCode` 三个字段，由 PLC 轮询线程**一次读取、分别推送**。
