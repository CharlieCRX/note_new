

# 问题描述：被“写死”的单轴日志标识

## 问题背景

现在随便一个轴的运动日志都类似：

```
[11:00:01.615][INFO][APP][JogOrch][G1][Y][T-240] START Jog direction=Forward(+)
[11:00:01.623][TRACE][HAL][Driver][G1][Y][T-240] Sending to PLC: EnableCommand(active=true)
[11:00:01.773][DEBUG][APP][JogOrch][G1][Y][T-240] Step: EnsuringEnabled -> IssuingJog
[11:00:01.785][TRACE][HAL][Driver][G1][Y][T-240] Sending to PLC: JogCommand(dir=Forward, active=true)
[11:00:01.785][DEBUG][APP][JogOrch][G1][Y][T-240] Step: IssuingJog -> Jogging
[11:00:02.014][TRACE][HAL][System][N/A][N/A][N/A] Tick: X1.pos=0.000000 X2.pos=0.000000 X1.isEnabled=1 X2.isEnabled=1 Mode=Decoupled
[11:00:02.317][INFO][UI][AxisVM][G1][Y][T-9610] User RELEASED Jog Positive (+)
[11:00:02.317][INFO][APP][JogOrch][G1][Y][T-9610] Stop Jog Requested by UI
[11:00:02.323][TRACE][HAL][Driver][G1][Y][T-240] Sending to PLC: JogCommand(dir=Forward, active=false)
[11:00:02.323][DEBUG][APP][JogOrch][G1][Y][T-240] Step: IssuingStop -> WaitingForIdle
[11:00:02.334][DEBUG][APP][JogOrch][G1][Y][T-240] Step: WaitingForIdle -> EnsuringDisabled
[11:00:02.344][TRACE][HAL][Driver][G1][Y][T-240] Sending to PLC: EnableCommand(active=false)
[11:00:02.354][SUMMARY][APP][JogOrch][G1][Y][T-240] Jog Operation -> SUCCESS (Safely Stopped)
```

可以看出单轴日志里有 `[G1][Y][T-240]` 这样的上下文追踪。

在系统从**单轴/单组 (Single-Axis)** 向**多轴/多组 (Multi-Axis & Gantry)** 演进的过程中，早期为了快速跑通流程而“硬编码（Hardcode）”的上下文标识，现在成为了日志追踪的毒药。

具体来说，当前的“病灶”在于：

> 在系统边界（UI 调用的入口处，即 `AxisViewModelCore`）写死了上下文，而下游所有的应用层（Orchestrator）和基础设施层（Driver）都依赖这个源头的上下文。

**最终目标**：我们需要让 `AxisViewModelCore` 以及它底层的编排器，在被实例化时（即在 `main.cpp` 中组装时），明确知道自己的 **组别 (GroupId)** 和 **轴名称 (AxisId)**。每次触发动作时，`TraceScope` 必须动态读取这两个属性，从而生成类似 `[G1][Z]` 或 `[G2][R]` 的正确日志。

好消息是，由于之前设计了非常优秀的 `TraceScope`（基于线程局部栈的上下文守卫），我们**不需要**去逐个修改 `JogOrchestrator` 或底层的类。我们只需要**在源头（`AxisViewModelCore`）把正确的 `GroupId` 和 `AxisId` 动态推入栈中**，底层的日志就会自动继承正确的标识！
