------

### 🏛️ 一、 架构原则：整洁架构下的日志法则

在 Clean Architecture 中加入日志，最容易犯的错误就是“让核心领域层（Domain）强依赖于某个具体的日志库（比如 log4cxx 或 spdlog）”。

**核心原则：**

1. **依赖倒置（DIP）：** 核心层只依赖日志的“接口”或“抽象宏”，绝对不依赖具体实现。
2. **上下文隔离（Contextual Logging）：** 看到一条日志，必须能在一秒内知道它来自哪一层、哪个类。
3. **分级克制：** 不是什么都要 log。Domain 层应该是相对安静的，Application 层记录流程，Infrastructure 层记录海量数据。

------

### 📊 二、 日志的分层与分级分类法 (Taxonomy)

为了让日志实现“结构化”，我们需要对日志的**级别（Level）**和**归属层（Layer）**进行极其严格的语义定义。

#### 1. 日志级别的业务语义

- **🔴 ERROR (错误):** 只有当系统出现无法自动恢复、必须阻断流程的异常时才触发。（例如：`AxisState::Error` 被触发，硬件限位被撞击）。
- **🟠 WARN (警告):** 发生了不符合预期的事，但系统做了拦截或自愈，没崩溃。（例如：UI 尝试在 `Disabled` 状态下发送点动指令，被 `RejectionReason` 拦截）。
- **🟢 INFO (信息):** 粗粒度的系统生命周期改变和核心业务节点。（例如：Orchestrator 启动定位任务、ViewModel 收到用户点击）。
- **🔵 DEBUG (调试):** 细粒度的状态机流转。（例如：Orchestrator 从 `EnsuringEnabled` 流转到了 `IssuingMove`）。
- **⚪ TRACE (追踪):** 极高频的底层数据流。（例如：FakePLC 每一个 tick 产生的物理坐标计算、Driver 发送的每一个字节）。

#### 2. 各个架构层的日志职责

| **架构层 (Layer)** | **日志频率** | **主要记录什么？**                                 | **示例格式**                                                 |
| ------------------ | ------------ | -------------------------------------------------- | ------------------------------------------------------------ |
| **Presentation**   | 低           | 用户的输入意图、UI状态转换                         | `[INFO][UI:AxisVM] User pressed Jog Positive`                |
| **Application**    | 中           | 状态机流转、业务用例的成功/被拒                    | `[DEBUG][App:JogOrch] Transition: Idle -> Jogging` `[WARN][App:MoveUc] Rejected move to 1500.0: TargetOutOfLimit` |
| **Domain**         | 极低         | 核心实体极少打印日志，主要靠返回 `RejectionReason` | `[ERROR][Dom:Axis] Hard limit triggered at pos 1000.0!`      |
| **Infrastructure** | 极高         | 硬件 IO 细节、Tick 心跳、驱动报文                  | `[TRACE][Infra:PLC] Tick 10ms. AbsPos: 123.45`               |

------

### 🧱 三、 落地设计模式：Facade（外观模式）

针对 C++ 项目，如果把 `ILogger` 对象通过构造函数注入到每一个 `UseCase` 和 `Orchestrator` 中，会导致依赖注入（DI）极其臃肿。

**业界标准做法：建立一个轻量级的跨层 Facade（外观）。**

建议在项目根目录新建一个 `common/` 或 `infrastructure/logging/` 目录，设计如下：

#### 1. 抽象接口层 (定义规范)

设计一组宏或静态方法。这些宏在编译时会被替换为真正的日志调用。

C++

```
// 示例：通用的宏设计，强制要求传入 Tag（标签）以实现结构化
#define LOG_INFO(TAG, ...)  Logger::info(TAG, __VA_ARGS__)
#define LOG_WARN(TAG, ...)  Logger::warn(TAG, __VA_ARGS__)
#define LOG_ERROR(TAG, ...) Logger::error(TAG, __VA_ARGS__)
#define LOG_TRACE(TAG, ...) Logger::trace(TAG, __VA_ARGS__)
```

#### 2. 结构化 Tag 约束

在你的代码中，规定每个文件顶部定义自己的 `TAG`：

C++

```
// 在 AutoAbsMoveOrchestrator.cpp 中
static constexpr const char* TAG = "AutoAbsOrch";

void AutoAbsMoveOrchestrator::start(double target) {
    LOG_INFO(TAG, "Starting absolute move to target: {:.2f}", target);
    // ...
}
```

C++

```
// 在 FakePLC.cpp 中
static constexpr const char* TAG = "FakePLC";

void FakePLC::tick(int ms) {
    LOG_TRACE(TAG, "Tick {}ms, Current Pos: {:.3f}", ms, m_feedback.absPos);
    // ...
}
```

#### 3. 底层实现 (Infrastructure)

底层的 `Logger` 类可以封装一个业界成熟的 C++ 日志库，强烈推荐 **`spdlog`**。它极其快速（Header-only 选项），支持异步，且支持 `{}` 风格的 Python 式格式化，非常适合现代 C++ (C++20)。

------

### 🚀 四、 行动建议（下一步怎么做）

如果你决定引入日志，为了不破坏当前美好的全绿测试状态，我建议分三步走：

1. **基础设施搭建：** 引入 `spdlog`（或者手写一个简单的 `std::cout` 包装器作为雏形），建立 `Logger` 门面接口。
2. **织入 Application 层：** 在 `Orchestrator` 的状态流转 `switch-case` 中，以及 `UseCase` 的被拒拦截处，加入 `LOG_DEBUG` 和 `LOG_WARN`。
3. **织入 ViewModel 层：** 在 `AxisViewModelCore` 接收到指令的地方加入 `LOG_INFO`。

通过这种方式，下次当你运行 TDD 测试时，终端不仅会显示绿灯，还会打印出极其清晰的系统运行轨迹，例如：

Plaintext

```
[2026-04-23 11:31:00] [INFO ] [UI:AxisVM] moveAbsolute requested to 100.0
[2026-04-23 11:31:00] [DEBUG] [App:AbsOrch] Step changed: Initial -> EnsuringEnabled
[2026-04-23 11:31:00] [TRACE] [Infra:Driver] Sending EnableCommand(true)
[2026-04-23 11:31:00] [DEBUG] [App:AbsOrch] Step changed: EnsuringEnabled -> IssuingMove
...
```

你是希望我们先手写一个基于标准输出（`std::cout`）的轻量级 Logger Facade 来快速验证概念，还是直接引入工业级标准库（如 `spdlog`）进行一步到位的集成？
