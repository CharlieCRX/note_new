这是一个非常棒的梳理！你已经完全掌握了 PLC 软元件地址（如 `M`、`D` 寄存器）到 **Modbus 逻辑地址**和功能码（Function Code, FC）的映射规律。

在这个映射体系中，规则非常清晰：

1. **M 寄存器（位/BOOL）**：对应 Modbus 的**线圈（Coil）**。在 Modbus Poll 中，读取使用 `FC 01`，写入通常后台调用 `FC 05`（单线圈写入）。地址即为 M 后面的数字。
2. **D 寄存器（字/INT/REAL）**：对应 Modbus 的**保持寄存器（Holding Register）**。读取使用 `FC 03`，写入调用 `FC 06/16`。地址即为 D 后面的数字。
* **INT** 占用 1 个寄存器。
* **REAL**（32位浮点数）占用连续的 2 个寄存器，且需要按你提到的小端序交换（Byte Swap / CDAB / BADC 等，具体视 PLC 厂家而定）进行解码。

### 给下一步的架构建议：

现在你已经非常清晰地把这些映射梳理出来了，结合我们之前的 TDD 讨论，这就是 `RegisterAddress.h` 里 `namespace plc::reg::y_axis` 的绝佳素材！

你会发现你的 PLC 工程师在地址分配上有一个很有意思的特点：

* **控制命令**（M42, M43, D24, D26 等）的地址偏**低**且不规律。
* **状态反馈**（M101-M115, D101-D156）的地址段比较**集中**。

**这对底层代码意味着什么？**

1. **写命令（控制）**：像触发 M42 或写 D24 时，通过 `PlcRegisterMap` 生成离散的单个 `RegisterWrite` 即可。
2. **读反馈（状态）**：你可以定义一个 `RegisterBlock Y_STATUS_BLOCK = { 101, 60 }`。在 `pollFeedback()` 时，一把抓取从 `D101` 到 `D160`（涵盖了状态、报警、所有 REAL 位置反馈），然后丢给解码器，直接将这一整块 byte array 解构为状态对象。这样通讯效率会非常高！
