## 问题整理与解决思路分析

### 一、当前错误码体系的三层结构

| 层级                 | 错误类型                       | 定义位置                           | 产生者                                                       |
| -------------------- | ------------------------------ | ---------------------------------- | ------------------------------------------------------------ |
| **Axis 领域层**      | `RejectionReason` (9个枚举值)  | `domain/entity/Axis.h`             | `Axis::enable()`、`Axis::moveAbsolute()` 等                  |
| **SystemContext 层** | `ContextRejection` (7个枚举值) | `domain/entity/ContextRejection.h` | `SystemManager::tryGetGroup()`、`SystemContext::tryGetAxis()` |
| **Gantry 联动层**    | `GantryRejection`              | `domain/gantry/GantryRejection.h`  | `GantryGroup` 联动操作                                       |

当前已有聚合方案 `UseCaseError = std::variant<monostate, ContextRejection, RejectionReason, GantryRejection>`，`EnableUseCase::execute()` 内部采用"漏斗式"逐层捕获并及早返回。

**真正的痛点**：UseCase/Orchestrator → ViewModel → UI 链条断裂。`AxisViewModelCore` 完全没有错误处理接口——没有返回 `UseCaseError` 的方法，`std::variant` 对 UI 层（QML）是"裸 variant"，需要大量 `std::visit` 分派。

### 二、推荐方案：ViewModel 作为防腐层（Anti-Corruption Layer）

核心原则：**Domain 层的错误枚举保持纯领域语义，UI 层永不直接消费领域枚举。ViewModel 负责将 `UseCaseError` 翻译为 UI 友好的结构化消息。**

```
┌──────────────────────────────────────────────────────┐
│  UI (QML)                                            │
│  - 绑定 errorMessage: string                         │
│  - 绑定 errorCode: string (用于图标/颜色/行为分发)     │
│  - 绑定 errorCategory: string (区分弹窗/内联/静默)    │
└──────────────────────┬───────────────────────────────┘
                       │ signal/property
┌──────────────────────▼───────────────────────────────┐
│  ViewModel (AxisViewModelCore / QtAxisViewModel)      │
│                                                       │
│  translate(UseCaseError) → ViewModelError {          │
│      string code;        // "AXIS_AT_POSITIVE_LIMIT" │
│      string userMessage; // "轴已到达正向限位"        │
│      string debugMessage;// "AxisId=X, pos=100.0,    │
│                          //  limit=100.0"            │
│      ErrorCategory cat;  // Inline / Modal / Silent  │
│  }                                                    │
│                                                       │
│  内部使用 std::visit 做一次性类型分发                   │
└──────────────────────┬───────────────────────────────┘
                       │ UseCaseError (variant)
┌──────────────────────▼───────────────────────────────┐
│  UseCase / Orchestrator                               │
│  - 继续返回 UseCaseError，不做 UI 翻译                │
└──────────────────────────────────────────────────────┘
```

### 三、具体实施步骤

1. **定义 `ViewModelError` 结构体**（`presentation/viewmodel/ViewModelError.h`）：包含 `code`（机器可读）、`userMessage`（用户可读）、`debugMessage`（日志调试）、`category`（Inline/Modal/Silent 三种）。

2. **在 ViewModel 中集中做 variant 翻译**：单一 `translate(const UseCaseError&)` 函数，用 `std::visit` 一次性覆盖所有可能的枚举值，映射为 `ViewModelError`。编译器保证覆盖所有分支。

3. **为每个 UseCase 调用附带错误状态**：`AxisViewModelCore` 调用 UseCase 后：
```cpp
UseCaseError err = m_enableUc.execute(...);
if (!std::holds_alternative<std::monostate>(err)) {
    m_lastError = translate(err);
    emit errorChanged();
}
```
`m_lastError` 通过 `Q_PROPERTY` 暴露给 QML，QML 直接绑定 `userMessage` 和 `errorCode`。

4. **Orchestrator 延迟错误的收集**：在 `tick()` 中轮询各 Orchestrator 的 `hasError()` / `lastError()`，统一翻译后通知 UI。

### 四、为什么这个方案是"优雅"的

| 备选方案                  | 缺点                                             |
| ------------------------- | ------------------------------------------------ |
| 把所有错误揉成一个大 enum | 丢失错误来源信息，各层耦合加剧                   |
| 每层各自暴露枚举给 UI     | UI 需 switch 三种 enum，与领域细节强耦合         |
| std::error_code 生态      | 对 QML 不友好，需注册 error_category，过度工程化 |

**推荐方案的优势**：
- 领域层保持解耦（各层枚举独立定义）
- UI 层只看到字符串（绝不泄露领域类型到 QML）
- 翻译集中管理（方便国际化、方便调整用户提示措辞）
- 类型安全（`std::visit` 由编译器保证覆盖所有分支）
- 一次性成本（`ViewModelError` 结构体 + 一个 `translate` 函数）

### 五、核心思想

**让每一层只说自己的语言，ViewModel 充当唯一的翻译官。** 优雅不在于消除复杂性，而在于将复杂性关在唯一一个可控的地方。
