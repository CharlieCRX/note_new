# 📘 servoV6 架构设计说明书

------

# 一、项目背景与问题

当前系统面临的核心问题：

```text
1. PLC寄存器驱动开发 → 业务逻辑与协议强耦合
2. 功能增加后 → 代码难以维护（if/else爆炸）
3. 无法测试 → 依赖真实设备，调试成本高
4. 扩展困难 → 更换PLC/协议几乎需要重写
5. UI与控制逻辑混杂 → 修改界面影响底层逻辑
```

👉 本质问题：

```text
“系统缺乏清晰的分层与抽象”
```

------

# 二、设计目标（给领导看的重点）

我们这次架构升级的目标不是“重构代码”，而是：

```text
1. 提升系统稳定性（工业级）
2. 降低后续开发成本
3. 支持多设备扩展（PLC / ARM / 其他协议）
4. 提高开发效率（减少联调时间）
5. 支持自动化测试（减少现场问题）
```

------

# 三、总体架构设计

采用：

```text
Clean Architecture（分层架构） :contentReference[oaicite:0]{index=0}
```

------

## 🏗 架构分层

```text
UI（界面层）
↓
Presentation（适配层）
↓
Application（业务流程层）
↓
Domain（核心业务层）
↓
Domain Service（领域服务层）
↓
Infrastructure（设备通信层）
↓
HAL（硬件抽象层）
```

------

# 四、为什么要这样分层（核心汇报内容）

下面这部分是你汇报的重点👇

------

# 1️⃣ UI层（界面层）

## 作用：

```text
用户操作 + 数据展示
```

## 为什么要独立？

```text
避免UI逻辑污染控制逻辑
```

## 如果不分：

```text
修改界面 → 可能影响设备控制（高风险）
```

------

## ✅ 优势

```text
✔ UI可以独立迭代（适配安卓/PC）
✔ 支持未来更换界面技术
```

------

# 2️⃣ Presentation层（ViewModel）

## 作用：

```text
UI 与业务之间的“翻译层”
```

------

## 为什么必须有？

```text
UI数据格式 ≠ 业务数据格式
```

例如：

```text
QString → std::string
按钮点击 → 业务命令
```

------

## ✅ 优势

```text
✔ UI修改不影响业务
✔ 业务逻辑不依赖Qt
```

------

# 3️⃣ Application层（UseCase）

## 作用：

```text
定义“系统能做什么”
```

例如：

```text
点动
定位
报警复位
```

------

## 为什么需要这一层？

👉 关键点：

```text
把“业务流程”集中管理
```

------

## 举例：

```text
点动流程：
1. 检查轴状态
2. 生成命令
3. 发送到设备
```

------

## ✅ 优势

```text
✔ 业务逻辑清晰可控
✔ 避免逻辑分散在UI/底层
✔ 易测试（不依赖硬件）
```

------

# 4️⃣ Domain层（核心业务层）

## 作用：

```text
描述“系统是什么”
```

------

## 包含：

```text
Axis（轴）
Motor（电机）
Alarm（报警）
```

------

## 为什么这是核心？

👉 因为：

```text
业务逻辑应该独立于硬件
```

------

## 举例：

```text
“轴在运动” ≠ “寄存器=1”
```

------

## ✅ 优势

```text
✔ 业务逻辑清晰（不依赖PLC）
✔ 可复用（未来换协议不影响）
✔ 可单元测试
```

------

# 5️⃣ Domain Service（领域服务层）

------

## 作用：

```text
处理“复杂业务逻辑”
```

------

## 举例：

```text
X1X2同步
超差判断
运动规划
```

------

## 为什么要单独拆？

👉 如果不拆：

```text
❌ Domain变复杂
❌ UseCase变臃肿
```

------

## ✅ 优势

```text
✔ 复杂逻辑集中管理
✔ 易维护
✔ 易扩展
```

------

# 6️⃣ Infrastructure（通信层）

## 作用：

```text
处理Modbus / TCP / 寄存器
```

------

## 为什么必须隔离？

👉 因为：

```text
协议会变，但业务不会变
```

------

## 举例：

```text
寄存器3004 → AxisPosition
```

------

## ✅ 优势

```text
✔ 更换PLC不影响业务逻辑
✔ 通信问题不会污染业务代码
```

------

# 7️⃣ HAL（硬件抽象层）

## ⭐ 这是“工程能力加分项”

------

## 作用：

```text
屏蔽底层设备差异
```

------

## 为什么要加？

👉 未来可能：

```text
PLC → ARM驱动
Modbus → EtherCAT
```

------

## ✅ 优势

```text
✔ 支持多设备
✔ 上层代码无需修改
✔ 提升系统复用性
```

------

# 五、系统核心优势（领导最关心）

------

## ✅ 1. 开发效率提升

```text
无需每次连接PLC即可开发
（支持模拟环境）
```

👉 减少联调时间 30%+

------

## ✅ 2. 系统稳定性提升

```text
业务逻辑与通信隔离
```

👉 通信异常不会破坏核心逻辑

------

## ✅ 3. 可测试性（非常重要）

```text
核心逻辑可单元测试
```

👉 提前发现问题，而不是现场发现

------

## ✅ 4. 扩展能力

```text
支持：
多轴
多设备
多协议
```

👉 后续需求成本降低

------

## ✅ 5. 长期维护成本降低

```text
结构清晰 → 新人容易接手
```

------

# 六、对比传统方案

------

## ❌ 传统方式：

```text
UI直接写寄存器
逻辑混在一起
```

问题：

```text
改一个功能 → 全局风险
```

------

## ✅ 当前方案：

```text
分层 + 解耦
```

结果：

```text
修改UI ≠ 修改控制逻辑
更换PLC ≠ 重写系统
```
