在“绝对位置坐标管理”中，根据你提供的设计说明书（节 9.1, 9.3, 9.4）以及工业控制的通用安全标准，我们需要通过 TDD 锁定的核心业务约束可以分为以下四个维度：

### 1. 现实数据保真约束 (Data Fidelity)

这是系统的“眼睛”，确保软件模型与物理编码器数据严格同步。

- **同步原子性**：`Axis` 必须能实时反映反馈包（`AxisFeedback`）中的绝对位置数值（D124） [cite: 2026-04-03]。
- **上电即有效性**：绝对位置不依赖于任何“回原点”流程，只要系统进入 `Unknown` 以外的状态，绝对位置必须是一个有效的物理值 [cite: 2026-04-03]。

------

### 2. 执行期安全屏障 (Safety Gatekeeping)

根据说明书 9.3 节，绝对位置清零是最高等级的“受控操作”，因为它会改变整个系统的物理基准 [cite: 2026-04-03]。

- **静止准入约束**：**严禁**在轴处于运动状态（`Jogging`, `MovingAbsolute`, `MovingRelative`）时接受“绝对位置清零”指令 [cite: 2026-04-03]。
- **状态排他性**：在“绝对位置清零”意图挂起期间，必须屏蔽任何新的运动指令，直到清零完成 [cite: 2026-04-03]。
- **错误状态锁定**：虽然 `Error` 状态属于停机，但通常业务约束要求必须先执行 `faultReset` 之后，才允许修改物理基准（除非该报警本身就是由于基准丢失引起的） [cite: 2026-04-03]。

------

### 3. 意图生命周期约束 (Intent Lifecycle)

确保指令从发出到 PLC 执行再到确认完成的闭环。

- **意图唯一性**：由于使用了 `std::variant` 命令系统，发起“绝对位置清零”必须物理性地覆盖并销毁当前槽位中的任何运动指令 [cite: 2026-04-03]。
- **反馈确认闭环**：`ZeroAbsoluteCommand` 意图的消失，不能仅仅因为指令发出了，而必须等到反馈中的 `absPos` 确实变为 $0$。
- **断电不保持约束**：根据 Excel 寄存器定义，该位置为“不保持”，因此 TDD 需验证系统重置或进入 `Disabled` 后，相关清零意图应被重置，防止上电误触发 [cite: 2026-04-03]。

------

### 4. 坐标系联动约束 (Coordinate Consistency)

根据 9.1 节的计算公式：

$$Relative Position = Absolute Position - Relative Zero Offset$$

绝对位置的变动会直接产生数学联动效应 [cite: 2026-04-03]。

- **基准偏移联动**：TDD 必须约束，当执行“绝对位置清零”后，虽然物理位置变了，但由于 `Relative Zero Offset` 尚未修改，此时的 `Relative Position` 必须反映出这个巨大的跳变，而不是静止不变 [cite: 2026-04-03]。
- **重置提醒约束**：业务上需约束，一旦绝对位置基准发生变化，系统必须产生一个“需要重新校准相对原点”的标志位或状态提醒 [cite: 2026-04-03]。
