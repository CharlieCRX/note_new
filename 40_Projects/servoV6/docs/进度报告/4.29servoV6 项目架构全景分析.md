# servoV6 项目架构全景分析

---

## 一、项目总览

**servoV6** 是一个 **伺服电机控制系统** 软件项目，基于 **C++20 + Qt6/QML** 技术栈，采用 **清洁架构（Clean Architecture）+ 领域驱动设计（DDD）+ 实体-用例-端口-适配器** 分层模型。

核心业务：**多轴电机运动控制**（Y/Z/R/X/X1/X2 六轴），支持**龙门双轴（Gantry）联动/解耦**模式，提供点动（Jog）、绝对定位（MoveAbsolute）、相对定位（MoveRelative）、停止（Stop）、使能/掉电（Enable）等功能。

## 三、分层架构详解（依赖方向：外层 → 内层）

```
┌──────────────────────────────────────────────────┐
│              Presentation（表示层）                │
│   QML Views ← QtAxisViewModel ← AxisViewModelCore │
└───────────────────────┬──────────────────────────┘
                        │ 依赖
┌───────────────────────▼──────────────────────────┐
│              Application（应用层）                 │
│   Orchestrator → UseCase → IAxisDriver(端口)       │
│                    → AxisRepository                │
└───────────────────────┬──────────────────────────┘
                        │ 依赖
┌───────────────────────▼──────────────────────────┐
│              Domain（领域层）                      │
│   Axis(聚合根) · AxisId · SystemContext            │
│   AxisCommand(variant) · AxisFeedback · 枚举状态   │
└──────────────────────▲──────────────────────────┘
                        │ 实现端口（依赖反转）
┌───────────────────────┼──────────────────────────┐
│              Infrastructure（基础设施层）          │
│   FakeAxisDriver(实现IAxisDriver)                  │
│   FakePLC(物理引擎) · Logger · TraceScope          │
└──────────────────────────────────────────────────┘
```

**核心依赖规则**：
- **Domain** 不依赖任何外层 → 零外部依赖，纯 C++ 标准库
- **Application** 依赖 Domain → 使用 IAxisDriver 抽象端口（依赖反转）
- **Infrastructure** 实现 Application 的端口 → 依赖 Domain 的类型定义
- **Presentation** 依赖 Application → 通过 ViewModel Core 间接操控

## 四、逐层详细分析

### 4.1 Domain 层 —— 内层核心，零依赖

| 文件                  | 作用                                                         | 关键设计                                                     |
| --------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`Axis.h`**          | **聚合根实体**。封装轴的完整状态机与业务规则。包含 10 种命令类型（`AxisCommand = std::variant<...>`）、6 种状态枚举（`AxisState`）、拒绝原因枚举（`RejectionReason`）。提供 `jog()`、`moveAbsolute()`、`moveRelative()`、`stop()`、`enable()` 等纯业务方法，**内部执行限位检查、状态准入、幂等性保护**等全部领域规则。 | `m_pending_intent` 意图槽位设计：同一时刻只允许一个运动意图，通过 `std::variant<std::monostate, ...>` 实现。`applyFeedback()` 方法在收到 PLC 反馈后自动消费已完成的意图（闭环收敛判定）。 |
| **`AxisId.h`**        | **值对象**。定义 6 个轴标识符枚举：`Y`、`Z`、`R`、`X`（逻辑龙门轴）、`X1`（物理龙门轴1）、`X2`（物理龙门轴2）。 | 用于多轴寻址，贯穿整个系统。                                 |
| **`SystemContext.h`** | **分组容器（"平行宇宙"）**。持有 `unordered_map<AxisId, Axis>` 注册表 + `IAxisDriver*` 驱动绑定 + `m_isGantryCoupled` 龙门全局状态。关键约束：**不同 SystemContext 之间绝对隔离**（各自持有独立的 Axis 实例和 PLC 驱动），Axis 实体不感知自己属于哪个分组。 | 支持未来多工位/多分组架构。`registerAllStandardAxes()` 注册全部 6 轴。 |

**领域层设计亮点**：
- **意图-反馈闭环**：Axis 不直接控制硬件，只产生"意图"（`m_pending_intent`），由外部反馈驱动意图消费。
- **纯业务规则**：所有限位检查、状态准入、数值预检均在 Domain 层完成，不依赖任何框架。

### 4.2 Application 层 —— 用例编排与策略

#### 4.2.1 端口定义

| 文件                | 作用                                                         |
| ------------------- | ------------------------------------------------------------ |
| **`IAxisDriver.h`** | **依赖反转边界**。纯虚接口 `send(AxisId, const AxisCommand&)`，将领域意图物理化。Domain 层不知道谁实现了它。 |

#### 4.2.2 仓储

| 文件                   | 作用                                                         |
| ---------------------- | ------------------------------------------------------------ |
| **`AxisRepository.h`** | 轴的运行时注册表，`unordered_map<AxisId, Axis>`，提供 `registerAxis()` / `getAxis()`。 |

#### 4.2.3 用例（UseCase）—— 单一语义动作

| 文件                        | 职责                 | 流程                                                         |
| --------------------------- | -------------------- | ------------------------------------------------------------ |
| **`EnableUseCase.h`**       | 使能/掉电            | `axis.enable(active)` → 成功则 `driver.send()`               |
| **`JogAxisUseCase.h`**      | 点动（含启动和停止） | `axis.jog(dir)` → `driver.send()` / `axis.stopJog(dir)` → `driver.send()` |
| **`MoveAbsoluteUseCase.h`** | 绝对定位             | `axis.moveAbsolute(target)` → `driver.send()`                |
| **`MoveRelativeUseCase.h`** | 相对定位             | `axis.moveRelative(distance)` → `driver.send()`              |
| **`StopAxisUseCase.h`**     | 紧急停止             | `axis.stop()` → `driver.send()`                              |
| **`AxisSyncService.h`**     | PLC 反馈同步         | `axis.applyFeedback(feedback)`                               |

**所有用例遵循相同模式**：① 从 Repository 获取 Axis 引用 → ② 调用领域方法产生意图 → ③ 如果成功，将意图通过 `IAxisDriver` 端口下发 → ④ 如果失败，透传 `RejectionReason`。

#### 4.2.4 策略编排器（Policy Orchestrator）—— 状态机

| 文件                            | 状态机步骤                                                   | 职责                                                         |
| ------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`JogOrchestrator.h`**         | `Idle → EnsuringEnabled → IssuingJog → Jogging → IssuingStop → WaitingForIdle → EnsuringDisabled → Done/Error` | 将点动操作建模为完整生命周期：自动使能→点动→(用户释放)→停止→自动掉电。包含防误杀校验（同时校验 AxisId 和 Direction）。 |
| **`AutoAbsMoveOrchestrator.h`** | `Initial → EnsuringEnabled → IssuingMove → WaitingMotionStart → WaitingMotionFinish → Done/Error` | 绝对定位编排：使能→发指令→检测运动开始→等待物理到位（双重验证：意图清除 + 位置容差）。 |
| **`AutoRelMoveOrchestrator.h`** | 同上（使用 MoveRelativeUseCase）                             | 相对定位编排：额外计算 `m_startPos + m_distance` 作为物理终点验证。 |

**编排器设计亮点**：
- 每个 `update()` 方法在物理时钟 `tick()` 中每 10ms 调用一次，驱动状态机步进。
- 全局最高优先级拦截 `AxisState::Error`。
- 运动完成后自动掉电（安全策略）。

### 4.3 Infrastructure 层 —— 适配器与横切关注点

#### 4.3.1 硬件抽象实现

| 文件                   | 作用                                                         | 关键设计                                                     |
| ---------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`FakeAxisDriver.h`** | **实现 `IAxisDriver` 端口**。将命令路由到 `FakePLC.onCommand(AxisId, cmd)`，并附加日志。 | 极薄适配器（4 行核心代码）。                                 |
| **`FakePLC.h`**        | **虚拟物理引擎**。维护 6 轴独立寄存器组（`AxisStateInternal`），实现 `onCommand()` 命令分发（`std::visit`）、`tick(ms)` 物理时钟（运动学计算 + 限位检测 + 状态转换）、`getFeedback()` 传感器数据回流。 | 每轴完全独立的内部状态（位置/速度/定时器/限位）。运动学使用 P 控制收敛算法。启用延迟 150ms 模拟真实上电。 |

#### 4.3.2 日志系统（可观测性）

| 文件               | 作用                                                         |
| ------------------ | ------------------------------------------------------------ |
| **`LogContext.h`** | 定义 `LogLevel`(6级)、`LogLayer`(UI/APP/DOM/HAL)、`LogContext` 结构体。 |
| **`Logger.h`**     | **全局异步日志系统**。使用生产者-消费者模式 + 双缓冲队列（`m_queue.swap(localQueue)` 瞬间释放锁，O(1) 复杂度）。支持同时输出到控制台和文件（按时间戳命名）。 |
| **`TraceScope.h`** | **调用链追踪**。基于 `thread_local std::vector<LogContext>` 的 RAII 栈，自动压入/弹出上下文（Group/Axis/TraceId）。 |

#### 4.3.3 工具

| 文件                     | 作用                                                         |
| ------------------------ | ------------------------------------------------------------ |
| **`CommandFormatter.h`** | 使用 `std::visit` + `if constexpr` 将所有 10 种命令类型格式化为可读字符串，供日志使用。 |

### 4.4 Presentation 层 —— ViewModel + QML

#### 4.4.1 ViewModel 双层设计

| 文件                           | 作用                                                         | 关键设计                                                     |
| ------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`AxisViewModelCore.h/.cpp`** | **纯 C++ 核心 ViewModel**（零 Qt 依赖，可单独单元测试）。状态投影（`state()`/`absPos()`/`relPos()`...）+ 控制输入（`jogPositivePressed()` 等，通过 TraceScope 创建操作上下文）+ `tick()` 驱动所有编排器状态机。 | 每个操作前创建 `TraceScope` 和 `generateTraceId()`，实现全链路追踪。 |

## 五、架构思想总结

### 5.1 核心设计模式

| 模式                | 应用位置                                                     | 说明                                                         |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **清洁架构**        | 整体分层                                                     | Domain（内层）→ Application → Infrastructure/Presentation（外层），依赖单向内指 |
| **依赖反转（DIP）** | `IAxisDriver` 端口                                           | Application 定义抽象，Infrastructure 实现，Domain 完全不知情 |
| **聚合根（DDD）**   | `Axis` 类                                                    | 轴的唯一访问入口，封装全部业务规则                           |
| **端口-适配器**     | `IAxisDriver` → `FakeAxisDriver`                             | 测试时可替换为 Mock 或真实硬件驱动                           |
| **策略模式**        | `JogOrchestrator` / `AutoAbsMoveOrchestrator` / `AutoRelMoveOrchestrator` | 每种运动模式有独立的状态机策略                               |
| **命令模式**        | `AxisCommand = std::variant<...>`                            | 10 种命令类型统一通过 `m_pending_intent` 槽位传递            |
| **MVVM**            | `AxisViewModelCore` → `QtAxisViewModel` → QML                | ViewModel 双向解耦：纯 C++ 核心 + Qt 适配器                  |
| **RAII**            | `TraceScope`                                                 | 自动管理调用链上下文的生命周期                               |
| **生产者-消费者**   | `Logger` 异步写入                                            | 双缓冲队列避免日志 I/O 阻塞主线程                            |

### 5.2 关键架构决策

1. **意图-反馈闭环**：Axis 不直接控制硬件，只产生"意图"。外部 PLC 反馈驱动意图消费，实现物理级闭环验证。
2. **单意图槽位**：`m_pending_intent` 同一时刻只允许一个命令，天然防止指令冲突。
3. **平行宇宙隔离**：`SystemContext` 支持多分组完全隔离的轴实例和驱动，为多工位并行控制奠定基础。
4. **ViewModel 双层解耦**：`AxisViewModelCore`（纯 C++，可单元测试）+ `QtAxisViewModel`（Qt 绑定），避免核心逻辑污染 Qt 依赖。
5. **所有循环依赖被打破**：Domain → 零外部依赖；Application → 仅依赖 Domain；Infrastructure → 实现 Application 端口。
