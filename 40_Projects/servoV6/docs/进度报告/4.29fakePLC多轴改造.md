### 🏆 一、 FakePLC (基础设施层) 改造进度总结

你已经成功将一个“单轴玩具”升级成了一个“轻量级多轴物理仿真引擎”，核心里程碑如下：

1. **实现了物理内存隔离 (Memory Mapping)**
   - **之前**：PLC 内部只有一组全局寄存器（单轴）。
   - **现在**：引入了 `std::unordered_map<AxisId, AxisStateInternal>`，为 `Y/Z/R/X1/X2` 每个物理轴分配了绝对独立的内存空间、状态机和限位配置。彻底杜绝了数据交叉污染。
2. **打通了精准指令路由 (Instruction Routing)**
   - **Driver 层**：`FakeAxisDriver` 成功降级为纯粹的“路由器”，它拦截带有 `AxisId` 的指令，并精准投递。
   - **PLC 层**：利用现代 C++ 的 `std::visit` 优雅解包多态指令，取代了臃肿的 `if-else`。
3. **构建了独立物理仿真时钟 (Simulation Cycle)**
   - `tick(ms)` 函数现在会遍历所有实例化的轴。即使 Y 轴正在高速点动、Z 轴撞击了软限位、R 轴正在绝对定位，它们在同一个 PLC 扫描周期内也能互不干扰地独立演算物理反馈。
4. **高标准的红黑盒测试网 (TDD Validation)**
   - 编写了极其严苛的系统集成测试，涵盖了“多轴使能时序隔离”、“多轴运动隔离”、“限位处逃逸死锁验证”以及“急停抢占”等工业级边缘场景，测试全线绿灯。

### 🗺️ 二、 规划：下一步的开发路线图

目前，你的“Domain (业务模型) -> Application (动作编排) -> Infrastructure (硬件仿真)”这条主干道已经完美支持多轴。接下来，我们站在了一个分岔路口，有两个高优先级的方向：

#### 路线 A：纵向打通贯穿（打通 Presentation / UI 层）

**目标：让现在的多轴控制能力在前端 UI 上真正跑起来。**

- **核心任务**：改造 `QtAxisViewModel`（或者引入 `SystemViewModel`）。
- **具体逻辑**：
  - ViewModel 需要维护一个“当前选中的 AxisId”状态。
  - 当用户在 UI（QML）的下拉框切换轴（比如从 Y 切换到 Z）时，ViewModel 内部的指令下发目标也随之切换。
  - UI 上的 Telemetry（位置、速度反馈）也需要根据当前选中的 `AxisId`，去订阅对应轴的状态流。
- **收益**：能够立刻在界面上看到多轴切换和控制的成果，获得直观的成就感。

#### 路线 B：横向扩展边界（引入 Group / SystemContext 概念）

**目标：完成我们在架构文档中定义的“多套系统（多 PLC）并行”的终极目标。**

- **核心任务**：建立 `SystemContext` 实体。
- **具体逻辑**：
  - 目前我们只有一个全局的 `AxisRepository` 和一个 `FakePLC`。我们需要将它们打包进一个 `SystemContext` (或者 `Group`) 中。
  - 实例化多个 Group（例如 `Group1` 绑定 `FakePLC_A`，`Group2` 绑定 `FakePLC_B`）。
  - 确保 Group 之间的时钟和指令完全物理隔离。
- **收益**：彻底闭环底层架构的抽象，为后续“X1/X2 多轴同步联动”和“多机台切换”打下完美基础。
