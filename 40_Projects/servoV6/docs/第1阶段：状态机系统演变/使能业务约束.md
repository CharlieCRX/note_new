### 1. 意图定义：`EnableCommand`

首先，我们将使能与掉电统一为一个带参数的指令，并放入 `std::variant`。

    bool active; // true: 使能(上电), false: 掉电

### 2. 语义约束逻辑设计

在工业控制中，使能操作具有极高的**安全优先级**。我们需要定义以下三条核心约束：

#### A. 准入屏蔽（Admission Shielding）

并不是任何时候都能使能或掉电。

| **发起动作**     | **当前轴状态** | **语义约束（规则）**                                         | **拒绝原因 (RejectionReason)** |
| ---------------- | -------------- | ------------------------------------------------------------ | ------------------------------ |
| **使能 (true)**  | `Error`        | **严禁使能**。必须先执行 `Reset`（清除故障）才能尝试上电。   | `InvalidState`                 |
| **掉电 (false)** | `Moving`       | **安全拦截**。电机正在高速旋转时直接切断动力会导致飞车或机械损伤。必须先 `Stop`。 | `AlreadyMoving`                |
| **使能 (true)**  | `Idle`         | **幂等处理**。如果已经上电了，再次发送上电指令不应产生新意图。 | `None` (静默通过)              |
| **掉电 (false)** | `Disabled`     | **幂等处理**。已经掉电了，无需重复掉电。                     | `None` (静默通过)              |

#### B. 互斥覆盖（Mutual Exclusion）

- 如果当前有一个“上电中”的意图，此时用户点下“强制掉电”，新的 `EnableCommand(false)` 必须**立即覆盖**旧的意图。
- `StopCommand` 具有最高优先级，但它不应自动触发掉电。

#### C. 生命周期闭环（Closure Conditions）

意图何时从“待办事项”中消失？

- **使能闭环**：当 `applyFeedback` 传回的状态变为 `Idle`、`Jogging` 或 `Moving` 时，证明硬件已响应上电，`EnableCommand(true)` 消费完成。
- **掉电闭环**：当反馈状态变为 `Disabled` 时，`EnableCommand(false)` 消费完成。

### 3. 代码实现参考

```C++
bool Axis::enable(bool active) {
    // 1. 安全拦截：运行中禁止掉电
    if (!active && (m_state == AxisState::Jogging || m_state == AxisState::MovingAbsolute)) {
        m_last_rejection = RejectionReason::AlreadyMoving;
        return false;
    }

    // 2. 状态检查：故障时禁止上电
    if (active && m_state == AxisState::Error) {
        m_last_rejection = RejectionReason::InvalidState;
        return false;
    }

    // 3. 幂等性优化
    if (active && (m_state != AxisState::Disabled && m_state != AxisState::Unknown)) return true;
    if (!active && m_state == AxisState::Disabled) return true;

    // 4. 产生意图
    m_pending_intent = EnableCommand{ active };
    m_last_rejection = RejectionReason::None;
    return true;
}
```

### 4. `applyFeedback` 中的闭环逻辑更新

我们需要在反馈回流时，根据“现实”来清理这个意图。

C++

```C++
void Axis::applyFeedback(const AxisFeedback &feedback) {
    // ... 其他更新逻辑 ...

    if (auto* cmd = std::get_if<EnableCommand>(&m_pending_intent)) {
        if (cmd->active) {
            // 如果意图是上电，当状态不再是 Disabled/Unknown 时，视为成功
            if (m_state != AxisState::Disabled && m_state != AxisState::Unknown) {
                m_pending_intent = std::monostate{};
            }
        } else {
            // 如果意图是下电，当状态变为 Disabled 时，视为成功
            if (m_state == AxisState::Disabled) {
                m_pending_intent = std::monostate{};
            }
        }
    }
}
```

### 🧠 架构师的思考：为什么要区分“停止”与“掉电”？

在你的设计中，`stop()` 是为了停止**运动**，而 `enable(false)` 是为了切断**动力**。

- **场景 1**：按“停止”按钮，轴停下但电机依然锁死（保持 Idle 态），此时你可以立即再次点动。
- **场景 2**：点“急停”或“断电”，轴停下且电机松开（变为 Disabled 态），此时你需要重新“使能”才能再次点动。
