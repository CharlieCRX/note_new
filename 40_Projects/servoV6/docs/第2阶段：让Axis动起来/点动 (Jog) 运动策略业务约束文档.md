# 点动 (Jog) 运动策略业务约束文档

## 1. 概述

本策略定义了系统执行“点动 (Jog)”操作时的状态机、流转规则、约束条件以及异常处理机制。点动操作的核心特征是：用户（或上层系统）发出持续的运动请求（按下按钮），系统开始运动；当请求撤销（松开按钮）时，系统减速停止。

## 2. 参与实体与指责划分

- **Orchestrator（点动编排器）：** 负责跨越时间的流程控制，处理上电、发指令、等待停止、掉电的完整生命周期。
- **Use Case（用例）：** 执行单一动作（如：使能、发起点动意图、发起停止意图）。
- **Entity（Axis 实体）：** 维护轴的真实状态，执行核心业务规则（软硬限位拦截、状态机准入校验），并记录拒绝原因。

## 3. 编排器 (Orchestrator) 状态机定义

点动编排器维护自身的生命周期状态 (`Step`)，以向调用方提供当前进度。

| **状态 (Step)**      | **描述**                                      |
| -------------------- | --------------------------------------------- |
| **Idle**             | 待命状态，无任何活动。                        |
| **EnsuringEnabled**  | 正在请求上电，等待底层反馈使能成功。          |
| **IssuingJog**       | 正在下发点动指令给底层系统。                  |
| **Jogging**          | 底层已接受指令，轴正在持续点动运行中。        |
| **IssuingStop**      | 正在下发停止指令给底层系统。                  |
| **WaitingForIdle**   | 等待轴减速停止，直到状态恢复为空闲 (`Idle`)。 |
| **EnsuringDisabled** | 正在请求掉电，等待底层反馈使能关闭。          |
| **Done**             | 完整流程正常结束（已安全停机并掉电）。        |
| **Error**            | 流程异常终止，需查询具体拒绝原因。            |

## 4. 核心业务动作与约束

### 4.1 启动点动 (Start Jog)

- **触发条件：** 外部请求 `startJog(Direction)`。
- **前置状态：** 无严格限制，但通常应在 `Idle`、`Done` 或 `Error`（复位后）状态下触发。
- **行为：**
  1. 记录目标方向 (`Direction`)。
  2. 重置内部控制标志（防重入/幂等标志）。
  3. 状态流转至 `EnsuringEnabled`。

### 4.2 停止点动 (Stop Jog)

- **触发条件：** 外部请求 `stopJog(Direction)`。
- **约束：**
  - **方向匹配约束：** 请求停止的方向必须与当前正在执行或请求的方向一致。如果不一致，忽略该请求（防止误杀）。
  - **状态准入约束：** 仅当编排器处于 `EnsuringEnabled`、`IssuingJog` 或 `Jogging` 状态时，才允许响应停止请求。
- **行为：** 状态流转至 `IssuingStop`。

## 5. 状态流转规则与内部校验 (Update 周期)

编排器在每个执行周期 (Tick/Update) 根据当前的 `Step` 和 `Axis` 实体的反馈状态进行流转。

### 5.1 全局异常拦截 (最高优先级)

- **规则：** 无论编排器处于何种状态，只要发现 `Axis` 实体的状态为 `Error`，立即将编排器状态置为 `Error`，并中断当前流程。

### 5.2 正常流转逻辑

- **EnsuringEnabled -> IssuingJog**
  - **条件：** `Axis` 实体状态变为 `Idle`。
- **IssuingJog -> Jogging**
  - **条件：** 成功下发点动意图（无拒绝原因）。
  - **异常处理：** 如果下发被 `Axis` 实体拒绝（例如：超限位、状态非法），编排器记录 `RejectionReason`，立即请求底层掉电，状态流转至 `Error`。
- **Jogging -> IssuingStop**
  - **触发条件 1（正常）：** 外部调用 `stopJog()` 改变了编排器状态。
  - **触发条件 2（异常跌落保护）：** 外部未调用停止，但发现 `Axis` 实体状态意外变回 `Idle`（通常是因为底层触发限位强行清除了运动意图）。此时编排器主动接管，流转至停止流程。
- **IssuingStop -> WaitingForIdle**
  - **行为：** 确保且仅确保一次向底层下发停止意图。
- **WaitingForIdle -> EnsuringDisabled**
  - **条件：** `Axis` 实体状态变为 `Idle`（确认已完全停稳）。
- **EnsuringDisabled -> Done**
  - **条件：** `Axis` 实体状态变为 `Disabled`。

## 6. 拒绝原因 (Rejection Reason) 穿透

当编排器进入 `Error` 状态时，调用方可查询具体的失败原因，以便向上层（UI/网络）展示具体的错误提示。可能的拒绝原因包括（继承自实体层）：

- `InvalidState`: 轴状态非法，无法执行当前操作。
- `AlreadyMoving`: 轴已在运动中，无法执行新的指令。
- `AtPositiveLimit`: 当前已处于正向限位点，禁止正向点动。
- `AtNegativeLimit`: 当前已处于负向限位点，禁止负向点动。
