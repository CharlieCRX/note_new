在 `servoV6` 这种涉及机械安全和实时交互的项目中，引入 `RejectionReason`（拒绝原因）绝不是画蛇添足，而是系统从“黑盒逻辑”向**“高透明度工业套件”**跨越的核心。

以下是从架构、运维和测试三个维度对引入 `RejectionReason` 必要性及价值的深度分析：

------

## 🛠️ 一、 解决“沉默失败”的必要性

在工业现场，操作员点击按钮后“机器没动”是一个极其危险且令人沮丧的场景。

1. **消除反馈歧义**：

   单纯的 `bool` 只能告诉你“不行”，但不能告诉你“为什么不行”。

   - **场景**：轴处于 `Error` 状态，操作员尝试 `Jog`。
   - **无原因**：界面只提示“操作失败”，用户可能会反复点击，甚至怀疑是网络通讯断了。
   - **有原因**：界面明确提示“轴处于故障态，请先复位”，引导用户进行正确操作。

2. **锁定“瞬间真相”（瞬时状态捕获）**：

   PLC 的状态回流（Feedback）是毫秒级的。

   - **竞态风险**：假设轴在移动中触发了限位，`Axis::jog()` 返回了 `false`。但由于通讯延迟，UI 在 50ms 后查询状态时，轴可能已经因为惯性停稳回到了 `Idle`。
   - **价值**：如果在 `jog()` 失败的**那一瞬间**由领域层（Domain）记录下 `RejectionReason::LimitActive`，就能保证给用户的报错信息是绝对真实且不可篡改的“案发现场”。

------

## 🏗️ 二、 架构设计的解耦价值

引入 `RejectionReason` 枚举体现了 **Clean Architecture（简洁架构）** 的核心思想：

1. **逻辑与表现的分离**：

   - **领域层（Domain）**：只负责**判定**（定性）。它只产生类似 `AxisNotIdle` 的逻辑枚举，不关心怎么显示。
   - **表现层（UI/ViewModel）**：只负责**翻译**（展示）。它将枚举翻译成“轴正忙，请稍候”。
   - **优势**：如果你以后要支持多语言（中/英/德），你只需要修改 UI 层的翻译表，而不需要动轴控制的核心代码。

2. **降低 UseCase 层的复杂度**：

   如果没有这个机制，`JogAxisUseCase` 可能需要写一堆 `if/else` 来猜测为什么 `Axis` 拒绝了它。

   - **有了它**：UseCase 只管传话。`Axis` 说什么，它就记什么。这符合“UseCase 只做流程，不写逻辑”的铁律。

------

## 🧪 三、 测试体系（TDD）的质变

在 Phase 1 的测试中，我们只能验证“行为是否被拦截”。引入原因后，我们可以验证“拦截逻辑是否正确”。

- **深层验证能力**：

  ```C++
  // 以前：只能测出“没成功”
  EXPECT_FALSE(axis.moveAbsolute(1001.0)); 
  
  // 以后：能测出“因为超限而没成功”
  EXPECT_FALSE(axis.moveAbsolute(1001.0));
  EXPECT_EQ(axis.lastRejection(), RejectionReason::TargetOutOfSoftLimit); 
  ```

  这能有效防止开发过程中出现的**“误杀”**（即因为 A 逻辑错误导致 B 功能意外失效）。

------

## 📋 四、 业务价值总结

| **维度**       | **引入 RejectionReason 的具体价值**                          |
| -------------- | ------------------------------------------------------------ |
| **调试效率**   | 日志中不再是 `Error`，而是 `Axis_Not_Enabled`，远程诊断效率提升 80%。 |
| **系统安全性** | 明确区分“软件拦截”与“硬件报警”，防止误操作带来的机械损坏。   |
| **用户体验**   | 交互从“猜测”变为“引导”，极大降低了操作员的上手门槛。         |
| **代码质量**   | 强制开发者在编写逻辑时考虑所有的“失败分支”，代码鲁棒性更强。 |
