在工业控制软件开发中，`FakePLC`（虚拟 PLC）不是一个简单的 Mock 对象，而是一个**“运行在内存中的物理仿真引擎”**。它是 Phase 3 的核心任务，决定了你的系统能否在没有硬件的情况下真正“活”起来。

以下是关于 `FakePLC` 的价值定义及业务约束文档。

---

## 一、 `FakePLC` 的价值与存在意义

### 1. 软件开发与硬件交付的解耦
* **并行开发**：你不需要等待真实的电机或 PLC 通讯链路调通，即可在本地笔记本上完成 UI、UseCase 和多轴逻辑的全部开发。
* **零成本环境**：通过实例化的方式，你可以瞬间在内存中模拟出 Y、Z、R 三个轴，而无需购买三套伺服驱动器。

### 2. 极端工况与安全验证
* **风险零成本**：在真实设备上测试“高速撞限位”或“急停”可能导致机械损坏。在 `FakePLC` 中，你可以随意注入错误状态（Error Injection），验证软件的健壮性。
* **时序敏感性测试**：你可以模拟不同的网络延迟和硬件响应时间（如使能延迟 100ms），观察软件在非理想工况下的表现。

### 3. 构建闭环测试环境 (CI/CD)
* **自动化验证**：有了 `FakePLC`，你的集成测试可以验证从“点击按钮”到“坐标变化”再到“意图清除”的完整生命周期，而不仅仅是测试一个函数。

---

## 二、 `FakePLC` 业务约束文档 (Specification)

### 1. 状态机行为约束 (State Machine)
`FakePLC` 必须通过内部状态机模拟硬件的异步迁移过程，严禁瞬时切换：

* **使能动作**：收到 `Enable(true)` 后，PLC 应进入“启动中”状态，并在约 100~200ms 后将状态设为 `Idle`。
* **运动执行**：
    * 收到 `Jog` 指令，状态切换为 `Jogging`，直至收到 `active = false` 的停止信号。
    * 收到 `Move` 指令，状态切换为运动态，当物理坐标进入容差（$\epsilon = 0.01$）且停止移动后，状态切回 `Idle`。
* **故障响应**：支持外部手动触发 `Error` 状态，此时应无视除了 `Stop` 之外的所有指令。

### 2. 物理仿真约束 (Physics Simulation)
* **坐标更新**：必须模拟离散的扫描周期（如每 16ms 或 20ms 一次 tick）。
* **运动连续性**：在运动状态下，`absPos` 应根据预设速度逐步逼近 `target`，而非直接跳变。
* **限位联动**：
    * 当 `absPos` 超过 `posLimitValue` 时，PLC 内部的 `posLimit` 位必须自动置 1。
    * 一旦物理限位触发，PLC 必须模拟硬件级别的强制制动（即刻停止位移更新）。

### 3. 反馈数据约束 (Feedback Output)
`FakePLC` 必须周期性产出标准的 `AxisFeedback` 结构体，作为整个系统的“唯一现实来源”：

| 字段       | 约束说明                                             |
| :--------- | :--------------------------------------------------- |
| `state`    | 映射 PLC 当前的内部状态（Disabled, Idle, Moving 等） |
| `absPos`   | 模拟编码器的当前绝对数值                             |
| `relPos`   | 动态计算：`absPos - relZeroAbsPos`                   |
| `posLimit` | 物理正限位传感器的电平信号镜像                       |

### 4. 指令优先级约束
* **Stop 穿透权**：无论 PLC 处于何种忙碌状态，收到 `StopCommand` 必须在下一次 tick 中立即停止物理计算并准备切回静止态。
* **幂等指令处理**：如果收到与当前物理状态一致的指令（如对已上电的轴发 `Enable(true)`），PLC 应当维持现状，不产生额外的处理开销。

## 三、 架构位置与演进关系



1.  **Phase 3 (现在)**：你的 `FakeAxisDriver` 会直接持有一个 `FakePLC` 的引用，将命令直接喂给模拟器。
2.  **Phase 5 (未来)**：当你开发 `ModbusAxisDriver` 时，你只需要把 `FakePLC` 换成真实的 PLC 硬件。由于 `Axis` 领域层只认 `AxisFeedback` 结构体，它根本不会察觉到底层是“真的”还是“假的”。
