完全理解。剥离掉 UI 和表现层后，我们的视线将 100% 聚焦于**领域模型、业务编排和硬件物理边界**。这正是构建一个极为健壮的工业控制系统底座的关键所在。

以下是为你重新修剪、提纯后的纯后端视点架构文档，它将作为我们下一步硬核代码开发的直接指导方针：

# 📘 多系统分组与龙门控制 (SystemContext / Group) 核心架构设计与业务约束文档

## 🎯 一、 核心概念与系统拓扑 (Core Definition & Topology)

**分组（Group / SystemContext）** 是为了在单一软件进程中，完美复用控制逻辑并绝对隔离物理状态而抽象出的“最高级子系统容器”。它代表了一套物理上完全独立运作的机械设备（例如机台 A 或控制柜 B）。

**控制模型拓扑结构：**
*   **SystemContext (系统聚合根)**：持有全局状态（急停、龙门联动模式）、唯一的 `AxisRepository` 和绑定的 `FakePLC`。
*   **逻辑/单轴对象 (Y, Z, R, 逻辑 X)**：业务层面的标准控制对象，对外呈现统一的运动学接口。
*   **物理基元对象 (X1, X2)**：仅在龙门解耦的特殊维护模式下，允许被业务层直接寻址操作。

---

## 🛑 二、 业务语义约束 (Business Semantic Constraints)

这是系统底层的“宪法”，任何代码调用都不得违背：

### 1. 组间绝对物理隔离 (Absolute Group Isolation)
*   **约束：** 不同的 Group 在内存中是“平行宇宙”。
*   **语义：** 切换控制流仅仅是将指令发送到了不同的 `SystemContext` 实例。Group A 发生了致命的机械超差报警，绝对不会阻塞或影响 Group B 的 `tick()` 物理时钟演算。

### 2. 轴实体的同级挂载原则 (Flat Registration)
*   **约束：** 在同一个 `SystemContext` 内部的 `AxisRepository` 中，单轴（Y/Z/R）、逻辑龙门轴（X）和物理龙门轴（X1/X2）在注册阶段**地位绝对平等**。
*   **语义：** 轴容器（Repository）是一个纯粹的“对象字典”，不负责鉴别轴的特殊身份，不动态销毁或创建轴。身份的互斥由编排层的 Policy 来负责。

### 3. 龙门状态的系统级归属 (System-Level Gantry State)
*   **约束：** “联动 (Coupled)” 与 “解耦/分动 (Decoupled)” 不属于某一个具体的轴，而是属于 `SystemContext` 的全局状态（对应 PLC 的 M125）。
*   **语义：** 业务层如果要进行龙门鉴权，必须向 `SystemContext` 索要当前状态，禁止在 `Axis` 实体内部私藏联动标志位。

### 4. 龙门操作的强互斥防呆原则 (Gantry Mutual Exclusion)
这是防止机械结构在软件层面被撕裂的核心指令拦截网：
*   **当 SystemContext 处于【联动 (Coupled)】模式时：**
    *   系统业务层（Application）**只接受**针对逻辑轴 `X` 以及 `Y, Z, R` 的操作指令。
    *   **拦截网：** 任何试图对 `X1` 或 `X2` 发送独立运动的指令，必须在 Policy 层被直接击毙，返回非法操作错误。
*   **当 SystemContext 处于【解耦 (Decoupled)】模式时：**
    *   系统业务层（Application）**只接受**针对物理基元轴 `X1, X2` 以及 `Y, Z, R` 的操作指令。
    *   **拦截网：** 逻辑轴 `X` 的概念暂时失效，任何针对 `X` 的指令必须被拦截。

---

## 🏛️ 三、 后端分层架构映射 (Backend Layered Mapping)

围绕上述约束，DDD 三个核心后端的职责边界如下：

### 🟣 1. 领域层 (Domain)
*   **升级聚合根：** 创建 `SystemContext` 实体。它作为工厂和管理者，持有该子系统的 `AxisRepository` 和 `Driver`。
*   **状态收口：** `SystemContext` 内部维护 `isGantryCoupled` 状态，并对外提供鉴权只读接口。

### 🔵 2. 应用编排层 (Application)
*   **依赖进化：** 之前的 UseCase (如 `JogAxisUseCase`) 需要注入 `SystemContext`，而不再是零散的 Repository。
*   **植入拦截器：** 编写 `GantrySafetyPolicy`。在 UseCase 的 `execute` 方法最顶端，调用该 Policy 进行操作互斥审查（校验上述第 4 条约束）。

### 🟠 3. 基础设施层 (Infrastructure - FakePLC)
作为最终的物理真相（Hardware Truth），它负责防范不可抗力的物理损坏：
*   **物理心跳并行化：** 必须支持多个 `FakePLC` 实例拥有独立的 `tick()` 调用栈。
*   **解耦物理监控（防撕裂底线）：** 在 `FakePLC` 内部，若当前为分动模式，`tick()` 必须实时计算 `abs(X1.pos - X2.pos)`。一旦差值超过 `D1030`（超差阈值），立刻将相关轴状态强制切为 `Error` 并触发急停。

## 🚀 四、 后续开发切入点建议 (Next Steps)

剥离了 UI 的干扰，我们的开发路径变得极其纯粹且聚焦，推荐按照以下 TDD 节奏推进：

1.  **Step 1：构建 `SystemContext` 容器**
    *   定义类结构，将 `AxisRepository` 和 `FakePLC` 装入其中。
    *   编写测试：确保可以实例化两套并行的 `SystemContext`，且它们的使能、运动互相毫无干扰。
2.  **Step 2：夯实 `FakePLC` 的龙门硬件逻辑**
    *   在 `FakePLC` 中引入联动标志位。
    *   编写测试：红/绿测试模拟分动状态下的“单轴超差撕裂触发急停”逻辑。
3.  **Step 3：织入 `GantrySafetyPolicy`**
    *   编写应用层的拦截器。
    *   编写测试：在联动模式下，试图调用 `MoveAbsoluteUseCase` 操作 X1，断言其抛出异常或返回 Policy Rejection。

**现阶段（开发 Group 逻辑）不应该完全屏蔽龙门结构，而是应该包含龙门结构的“骨架与状态占位”，但暂时剥离其“复杂的物理和防呆行为”。**

用建筑学的话来说：这阶段我们把 X1、X2 的“房间”在 Group 这栋楼里建好，并把“联动开关”装在 Group 的大厅里，但暂时不给它们通电，也不去写房间里复杂的安保规则。

具体来说，在当前的“分组 (SystemContext) 开发阶段”，龙门结构的开发划分如下：

### 🟢 必须包含的部分（龙门结构的“骨架”）

在开发 `SystemContext` 时，你**必须**把以下与龙门相关的基础结构写进去，因为它们是 Group 这个容器的天然组成部分：

1. **实体全量注册 (Axis Registration)：**

   在初始化 `SystemContext` 内部的 `AxisRepository` 时，你需要把 `AxisId::X`（逻辑轴）、`AxisId::X1` 和 `AxisId::X2`（物理轴）全部 `registerAxis()` 进去。让它们作为平等的“一等公民”存在于容器中。

2. **全局状态锚点 (System State)：**

   龙门的联动状态是属于 Group 的。所以在 `SystemContext` 类中，你需要声明这个状态属性：

   ```C++
   class SystemContext {
   private:
       bool m_isGantryCoupled = true; // 默认处于联动模式
   public:
       bool isGantryCoupled() const { return m_isGantryCoupled; }
       void setGantryCoupled(bool coupled) { m_isGantryCoupled = coupled; }
       // ... 其他 repository 和 FakePLC 的成员
   };
   ```

**为什么要现在做？** 因为如果不把这三个轴放进去、不把状态变量写在 `SystemContext` 里，你的 Group 容器就是一个“残缺”的隐患容器，后续加龙门状态时还得回头改 Group 的数据结构。

------

### 🔴 必须屏蔽的部分（龙门结构的“行为与规则”）

虽然骨架建好了，但在这一阶段，你**绝对不要**去碰以下深水区的逻辑，把它们严格留给下一个阶段（Step 2 & 3）：

1. **FakePLC 的防撕裂运算 (Infrastructure 层)：**

   现在 `FakePLC::tick()` 里不需要去算 `abs(X1 - X2)` 是否超过超差阈值，也不用写触发急停的代码。现在即使 X1 走到天涯海角，X2 停在原地，代码也不要管。

2. **安全拦截策略 (Application 层)：**

   暂时不要去写 `GantrySafetyPolicy`，也不要在你的 `JogAxisUseCase` 或 `MoveAbsoluteUseCase` 里去判断“如果联动就报错”。现阶段，如果测试代码去操作 X1，让它动就完事了。

**为什么要现在屏蔽？** 因为这是 TDD（测试驱动开发）的铁律：**“一次只跨越一个维度的复杂度”**。如果把组的物理隔离和龙门防撕裂揉在一起写，一旦测试亮红灯，你根本不知道是内存隔离没做好，还是龙门逻辑写错了。

------

### 💡 架构师的阶段验收标准 (Definition of Done)

当你完成了“分组 (SystemContext)”这个阶段的开发时，你的系统应该能跑通这样一个简单的 TDD 测试用例：

- **断言 1：** 实例化 `SystemContext groupA` 和 `SystemContext groupB`，它们各自有自己的 `Y, Z, R, X, X1, X2`。
- **断言 2：** `groupA.setGantryCoupled(false)`，随后 `groupA.isGantryCoupled()` 为 `false`，但 `groupB.isGantryCoupled()` 依然保持默认的 `true`。
- **断言 3：** 无论操作 `groupA` 的 `X1` 还是 `Y`，它们都能独立运动，且绝不影响 `groupB`。

只要这个测试绿了，说明你的“多系统容器”已经完美建好。接下来，你就可以心无旁骛地在这个无懈可击的容器里，专门针对 X1/X2 编写那些硬核的“联动同步”与“超差防撕裂”规则了。

你看这样“骨架先行，行为后补”的切分，是不是让你的开发思路瞬间轻松了很多？
