# 📘 ServoV6 架构重构方案：错误语义解耦与分层反馈机制

## 1. 现状问题定义 (Problem Statement)

在当前的实现中，`RejectionReason` 作为一个单一枚举，同时承担了两个完全不同层级的职责：

1. **准入层 (Access Control)**：由 `SystemContext` 判定。例如：由于龙门联动锁定，禁止访问 X1/X2 物理轴。
2. **执行层 (Execution Logic)**：由 `Axis` 实体判定。例如：轴未使能、目标超限、硬件报警。

**核心矛盾**：

- **语义重叠**：`InvalidState` 在 `SystemContext` 中表示“访问权限非法”，在 `Axis` 中表示“运行状态非法”。
- **反馈丢失**：UI 无法区分是“系统规则拦截”还是“硬件状态拦截”，导致用户无法获得精准的交互引导（例如：是该去解耦龙门，还是该去使能轴？）。

------

## 2. 架构约束方案 (Proposed Constraints)

为了符合“单一职责原则”并消除耦合，建议将错误信息拆分为 **两级拦截机制**。

### A. 定义分层枚举

将原本臃肿的 `RejectionReason` 拆解为两个独立的语义域：

- **`AccessRejection` (准入层拒绝)**：
  - `None`: 允许访问。
  - `GantryLocked`: 龙门耦合中，物理轴被锁定。
  - `LogicAxisDisabled`: 龙门未耦合，逻辑轴不可用。
  - `AxisNotFound`: 轴 ID 在当前上下文中不存在。
- **`AxisRejection` (执行层拒绝)**：
  - `None`: 正常执行。
  - `NotEnabled`: 轴未使能。
  - `SoftLimitViolation`: 目标超出软限位。
  - `HardwareAlarm`: 驱动器报警。

### B. 统一返回结构 (Result Pattern)

为了维持你偏好的 **Try-Get** 模式，引入一个轻量级的包装结构：

```C++
struct CommandResult {
    bool accepted = true;
    AccessRejection accessError = AccessRejection::None;
    AxisRejection executionError = AxisRejection::None;

    // 辅助判定接口
    bool isSuccess() const { return accepted && executionError == AxisRejection::None; }
};
```

------

## 3. 逻辑修改方案 (Modification Plan)

### 阶段一：SystemContext 的门卫职责升级

修改 `tryGetAxis` 接口，使其只负责 `AccessRejection` 的判定，不再混用轴内部的错误状态。

- **逻辑变化**：如果龙门状态冲突，返回 `AccessRejection::GantryLocked`。
- **解耦效果**：`SystemContext` 此时只关心“权限”，完全不感知 `Axis` 内部的 `AxisState`。

### 阶段二：UseCase 层的消费模式调整

在 `MoveAbsoluteUseCase` 或 `EnableUseCase` 中，按照由外向内的顺序进行判定：

1. **第一步**：调用 `SystemContext::tryGetAxis`。若失败，直接向 UI 反馈准入错误（提示用户检查龙门状态）。
2. **第二步**：调用 `Axis::move(...)`。若失败，获取 `AxisRejection`（提示用户检查轴状态或限位）。

------

## 4. TDD 验证策略 (Test-Driven Validation)

重构过程中，需确保以下测试用例通过，以验证解耦的有效性：

1. **准入测试 (Access Test)**：
   - **Context**: 龙门处于 `Coupled` 状态。
   - **Action**: 尝试获取物理轴 `X1`。
   - **Expect**: 返回 `false`，且错误码明确为 `AccessRejection::GantryLocked`。
2. **执行测试 (Execution Test)**：
   - **Context**: 龙门处于 `Decoupled` 状态，`X1` 获取成功。
   - **Action**: 对未使能的 `X1` 发起运动。
   - **Expect**: `tryGetAxis` 返回 `true`，但 `Axis::move` 返回 `false`，错误码为 `AxisRejection::NotEnabled`。

------

## 5. 预期收益 (Expected Benefits)

- **UI 交互精准化**：界面可以根据 `accessError` 自动切换龙门控制视图，根据 `executionError` 闪烁报警灯。
- **代码自文档化**：开发者通过返回值就能立刻判断故障发生的层级。
- **扩展性**：未来若增加“安全门校验”或“权限系统”，只需扩展 `AccessRejection`，无需触动底层 `Axis` 逻辑。
