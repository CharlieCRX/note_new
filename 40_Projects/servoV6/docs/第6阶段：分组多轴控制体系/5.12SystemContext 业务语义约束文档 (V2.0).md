这份文档（V2.0 版本）在 V1.0 的基础上，深度整合了 `GantryGroup` 领域组件，确立了 `SystemContext` 作为聚合根如何协调“静态轴资源”与“动态龙门行为”的逻辑。

------

# 📘 SystemContext 业务语义约束文档 (V2.0)

## 1. 定位与角色 (High-Level Role)

`SystemContext` 是物理系统的**聚合根（Aggregate Root）**。它不仅是轴实体的容器，更是该系统内**业务规则的终点**。

- **资源持有者**：持有并管理该组唯一的轴池（Axes）和龙门组件（GantryGroup）。
- **准入执行者**：基于龙门状态，强制执行轴访问的合法性校验。
- **状态协调者**：在 `GantryGroup` 的状态机与 `SystemContext` 的访问开关之间同步状态。

------

## 2. 内部结构组合 (Composition & Structure)

每个 `SystemContext` 实例由以下三部分强耦合组成：

1. **轴池 (Axis Repository - Internal)**：
   - 固定包含：`Y`, `Z`, `R` (常规轴)；`X` (逻辑龙门轴)；`X1`, `X2` (物理基元轴)。
2. **龙门大脑 (GantryGroup)**：
   - 作为领域服务组件，持有 `X1` 和 `X2` 的引用。
   - 负责龙门状态机切换（联动/解耦、对齐校验）。
3. **驱动通道 (IAxisDriver)**：
   - 该组绑定的唯一物理通信接口。

------

## 3. 核心业务约束 (Core Business Rules)

### A. 轴访问权限的“二元互斥”逻辑

`SystemContext` 维护一个内部标志 `m_isGantryCoupled`，该标志直接决定 `getAxis(id)` 的行为：

| **当前模式**         | **getAxis(AxisId::X)** | **getAxis(AxisId::X1/X2)** | **业务含义**                                    |
| -------------------- | ---------------------- | -------------------------- | ----------------------------------------------- |
| **联动 (Coupled)**   | **允许**               | **拒绝 (抛出异常)**        | 物理轴被锁定，由逻辑轴 X 统一指挥（PLC 映射）。 |
| **解耦 (Decoupled)** | **拒绝 (抛出异常)**    | **允许**                   | 逻辑轴 X 失去物理意义，允许独立操作物理电机。   |

> [!IMPORTANT]
>
> **约束理由**：防止应用层在联动时错误地给单电机发指令导致机械损伤，或在解耦时给已不存在的逻辑轴发指令导致逻辑混乱。

### B. 龙门状态转换的闭环逻辑

龙门状态的变更遵循“**意图触发 -> 硬件执行 -> 反馈闭环**”的原则：

1. **意图发起**：由 `GantryGroup` 评估 `X1/X2` 状态，生成 `GantryCommand`。
2. **权限挂起**：在转换期间（如解除联动中），`SystemContext` 可选择锁定 X、X1、X2 的所有访问权限。
3. **状态同步**：当驱动层反馈 `GantryFeedback::isCoupled` 发生变化后，`GantryGroup` 更新状态机，并同步调用 `SystemContext::setGantryCoupled()`。

### C. 解耦模式下的动作降级 (Degraded Mode)

当通过 `getAxis()` 访问 `X1` 或 `X2` 时，虽然返回了 `Axis` 实体，但 Application 层必须遵守以下降级规则：

- **合法动作**：点动 (Jog)、绝对位置移动 (MoveAbsolute)。
- **非法动作**：相对位置移动 (MoveRelative) —— 为了防止在龙门未对齐情况下进行累积误差运动。

------

## 4. 交互流控制 (Interaction Flow)

### 场景一：常规轴运动 (如 Y 轴)

1. `UseCase` 调用 `context.getAxis(AxisId::Y)`。
2. `Axis Y` 产生 `MoveCommand`。
3. `UseCase` 通过 `context.driver()->send(AxisId::Y, cmd)` 发出。

### 场景二：龙门解耦操作

1. `GantryUseCase` 调用 `context.gantry().requestDecouple()`。
2. `GantryGroup` 检查 X1/X2 是否静止，产生 `GantryCommand { coupled: false }`。
3. `UseCase` 拿到意图并发送给 PLC。
4. PLC 执行硬件脱开。
5. `SystemContext` 收到反馈，通过 `context.gantry().applyFeedback()` 驱动状态机。
6. `GantryGroup` 调用 `context.setGantryCoupled(false)`，**权限门卫即刻切换**。

------

## 5. 多组隔离约束 (Group Isolation)

- 每个 `SystemContext` 是一个独立的实例。
- **禁止跨组引用**：`SystemContext A` 里的轴绝不能出现在 `SystemContext B` 的逻辑中。
- **驱动绑定**：每个实例在初始化时必须绑定其对应的 `IAxisDriver`（如 IP 地址 A 对应组 A）。

------

**确认点：**

1. **反馈闭环**：你是否同意由 `GantryGroup` 处理完反馈后再回调同步 `SystemContext` 的权限位？（这样能保证权限切换与物理状态绝对同步）。
2. **初始化逻辑**：是否在 `SystemContext` 构造时就自动完成所有轴和 `GantryGroup` 的实例化？（这样能保证聚合根一出生就是完整的）。

## 6. 驱动层抽象 (Driver Abstraction)

- **聚合驱动 (Unified Driver)**：`SystemContext` 必须持有且仅持有一个实现 `ISystemDriver` 接口的对象。
- **职责分离 (ISP)**：虽然物理实现是同一个对象，但在业务逻辑中：
  - **轴意图**：通过 `IAxisControl::send(AxisId, Command)` 消化。
  - **龙门意图**：通过 `IGantryControl::sendGantry(GantryCommand)` 消化。
- **分组隔离**：驱动实例必须与 `SystemContext` 实例 1:1 绑定。**严禁**组 A 的 `SystemContext` 调用组 B 的驱动实例。
