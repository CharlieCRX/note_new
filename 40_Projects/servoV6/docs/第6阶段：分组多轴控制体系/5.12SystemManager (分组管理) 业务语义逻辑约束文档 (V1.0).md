**分组（SystemManager）**逻辑，是解决“规模化”和“物理拓扑映射”的关键。在进入代码实现之前，我们需要先确定分组管理器的语义约束。

------

# 📘 SystemManager (分组管理) 业务语义逻辑约束文档 (V1.0)

## 1. 定位与职责 (Definition & Responsibility)

`SystemManager` 是整个应用层的**中心枢纽（Hub）**。它不负责具体的轴控制算法，其核心职责在于：

- **实例生命周期管理**：负责创建、存储和销毁 `SystemContext` 实例。
- **拓扑映射（Routing）**：将逻辑上的“组名”（如 "Station_Left"）映射到具体的物理资源实体。
- **物理隔离维护**：确保不同分组之间的内存空间、命令队列和状态完全解耦，实现“平行宇宙”效应。

------

## 2. 核心逻辑约束 (Core Constraints)

### A. 身份唯一性约束 (Identity Uniqueness)

- 每个 `SystemContext` 实例在 `SystemManager` 中必须拥有**全局唯一的字符串标识（GroupId/Name）**。
- 不允许在同一 Manager 下注册两个相同 ID 的分组。

### B. 资源独占性约束 (Resource Exclusivity)

- **轴不跨组**：一个 `Axis` 实例只能属于一个 `SystemContext`。严禁将组 A 的 Axis 引用插入到组 B 的容器中。
- **驱动 1:1 绑定**：每个 `SystemContext` 实例必须绑定一个独立的 `ISystemDriver`。即使两个机台逻辑完全一致，也必须通过不同的驱动实例（或具有不同寻址信息的同一驱动）进行物理隔离。

### C. 访问路由语义 (Routing Semantics)

- **禁止直接持有 Context**：除初始化逻辑外，应用层的 UseCase 或 ViewModel 不应长期持有 `SystemContext` 的引用。
- **按需获取**：操作发起时，应通过 `SystemManager.getGroup(name)` 获取引用，并在操作结束后释放对 Context 的直接引用。这保证了在动态删减分组时不会出现悬空指针。

------

## 3. 分组隔离性（Parallel Universe Principle）

这是系统最核心的工业语义：

1. **报警隔离**：组 A 的轴报警（Error），绝不应导致组 B 的轴进入锁定状态。
2. **龙门状态隔离**：组 A 的龙门解耦（Decoupled），组 B 的龙门可以依然处于联动（Coupled）状态。
3. **驱动隔离**：组 A 发送的指令失败，不应阻塞组 B 的指令发送通道。

------

## 4. 标准交互流程 (Interaction Flow)

当 UI 发起一个“使能”请求时，逻辑流向如下：

1. **定位阶段**：UI 传递 `(groupName: "Station_1", axisId: X)`。
2. **路由阶段**：`SystemManager` 查找并返回 `Station_1` 对应的 `SystemContext` 实例。
3. **鉴权阶段**：`SystemContext` 检查当前是否允许操作 `Axis X`（龙门约束）。
4. **执行阶段**：`UseCase` 拿到经过授权的 `Axis` 对象，产生命令并通过 `SystemContext` 绑定的驱动发出。

------

## 5. 初始化与配置逻辑 (Bootstrapping)

在系统启动阶段，`SystemManager` 需按照配置表执行以下动作：

1. `createGroup("Group_A")` -> 实例化 `SystemContext`。
2. `assignDriver("Group_A", driverInstance)` -> 绑定物理通信链路（如指定 IP 的 PLC）。
3. `configureAxes("Group_A", ...)` -> 可选：配置轴的软限位、默认速度等参数。
