这份文档旨在确立 `SystemContext` 在系统架构中的“宪法”地位。它不仅是一个容器，更是所有业务规则的**强制执行点**。

------

# 📘 SystemContext 业务语义约束文档 (V1.0)

## 1. 定位与边界 (Definition & Boundary)

`SystemContext` 是一个**独立物理系统（Group）的聚合根**。它代表了一套完整的、逻辑闭环的控制单元（例如：机台 A）。

- **唯一性**：一个 `SystemContext` 实例内部包含且仅包含一套特定的轴实体（Y, Z, R, X, X1, X2）。
- **物理隔离**：不同 `SystemContext` 实例之间互为“平行宇宙”，内存状态、报警状态、驱动连接完全隔离，互不干扰。

------

## 2. 核心成员构成 (Component Inventory)

每个 `SystemContext` 内部强制维护以下资源：

1. **轴池 (Axis Pool)**：
   - **常规轴**：`Axis Y`, `Axis Z`, `Axis R`。
   - **逻辑轴**：`Axis X`（代表龙门整体）。
   - **物理基元**：`Axis X1`, `Axis X2`（代表龙门背后的独立电机）。
2. **全局状态**：`m_isGantryCoupled` (bool)，标识龙门当前处于“联动”还是“解耦”模式。
3. **驱动抽象**：持有 `IAxisDriver` 指针，指向该系统绑定的物理（或模拟）通信通道。

------

## 3. 核心业务语义约束 (Business Rules)

### A. 访问受控逻辑 (The "Gatekeeper" Rule)

`SystemContext` 必须根据 `m_isGantryCoupled` 状态，对轴的访问权限进行硬性拦截：

| **龙门状态**         | **允许访问的轴 ID** | **禁止访问的轴 ID** | **拦截理由**                                        |
| -------------------- | ------------------- | ------------------- | --------------------------------------------------- |
| **联动 (Coupled)**   | Y, Z, R, **X**      | **X1, X2**          | 联动时，物理轴由 PLC 内部接管，应用层禁止直接干预。 |
| **解耦 (Decoupled)** | Y, Z, R, **X1, X2** | **X**               | 解耦后，逻辑轴 X 失去物理意义，禁止调用。           |

> [!CAUTION]
>
> **违规处理**：若 Application 层尝试通过 `getAxis()` 获取当前状态下禁用的轴，`SystemContext` 必须抛出异常或返回空结果，拒绝后续任何指令生成。

### B. 联动逻辑语义

1. **指令下沉**：当系统处于联动模式时，对 `Axis X` 发送的使能（Enable）、移动（Move）、停止（Stop）指令，通过驱动层发送至 PLC 的 `AxisId::X` 寄存器。
2. **物理同步**：PLC 负责在硬件层级实现 X 到 X1/X2 的同步映射。`SystemContext` 内部不进行 X1/X2 的软件同步，仅维持 `Axis X` 的单轴逻辑状态。

### C. 解耦运行约束 (Maintenance Mode)

当处于解耦模式且访问 `X1` 或 `X2` 时，系统进入“受限维护模式”：

1. **动作受限**：仅允许执行点动（Jog）和绝对位置移动（MoveAbsolute）。
2. **相对位移禁用**：禁止执行相对移动（MoveRelative），防止在未校准状态下造成机械损伤（此约束由 Application Policy 配合执行）。

------

## 4. 指令执行流 (Command Workflow)

`SystemContext` 不负责“执行”动作，它只负责“路由”**和**“授权”。

1. **请求阶段**：应用层 UseCase 请求 `context.getAxis(id)`。
2. **授权阶段**：`SystemContext` 校验当前龙门状态是否允许操作该 ID。
3. **生成阶段**：对应的 `Axis` 实体产生 `AxisCommand`（意图）。
4. **分发阶段**：UseCase 从 `context.driver()` 获取驱动引用，将 ID 和 Command 发送出去。

------

## 5. 状态转换约束 (Gantry Transition)

龙门状态的切换（联动 $\leftrightarrow$ 解耦）是高风险操作：

- **发起者**：仅允许由 `GantryGroup` 领域服务发起状态变更。
- **变更前提**：必须确保 `X1` 和 `X2` 处于静止（Idle）且无报警状态。
- **副作用**：状态切换完成后，`SystemContext` 必须立即更新 `m_isGantryCoupled` 标志，从而瞬间改变 `getAxis()` 的拦截行为。
