# 📄 MiniMotion-Demo 项目架构文档

#### 1. 目录结构规划

标准的 Qt/C++ 项目结构，强制分离 UI 和业务逻辑：

```bash
MiniMotion-Demo/
├── CMakeLists.txt             # 构建脚本
├── src/
│   ├── main.cpp               # 程序入口，负责注册 QML 类型和实例化 C++ 对象
│   ├── backend/               # C++ 后端逻辑区 (TDD 重点测试区域)
│   │   ├── MotionController.h # ViewModel 层：连接 QML 和 Modbus 的桥梁
│   │   ├── MotionController.cpp
│   │   ├── ModbusService.h    # 硬件层：专门处理 QModbusTcpClient 通信
│   │   └── ModbusService.cpp
│   └── tests/                 # 单元测试区
│       └── test_motion_logic.cpp # 独立编译的测试用例
└── qml/                       # 前端 UI 区
│   ├── main.qml               # 主窗口，负责整体布局
│   └── components/            # 自定义的可复用组件
│       └── SafeSlider.qml     # 带软限位的可视化滑块组件 (你现在的开发重点)
└── tests/                     # 单元测试工程
    ├── CMakeLists.txt         # 编译测试用例的脚本
    ├── test_model/            # 针对硬件解析的测试
    └── test_viewmodel/        # 针对业务逻辑的测试 (重点)
```

#### 2. 分层架构职责 (MVC/MVVM 变体)

- **UI 层 (qml/)**：
  - **原则**：绝对不写复杂的业务判断。QML 只负责两件事：**展示状态**和**接收用户输入**。
  - **实现**：通过读取 C++ 暴露的属性（`currentPos`, `minLimit`, `maxLimit`）来渲染滑块位置和报警颜色。
- **ViewModel 层 (`MotionController`)**：
  - **原则**：继承自 `QObject`，充当“胶水代码”。它是 QML 唯一能直接对话的 C++ 类。
  - **实现**：使用 `Q_PROPERTY` 宏将数据暴露给 QML。在这里实现 `bool validateTarget(double target)` 函数，判断用户输入的目标位置是否在软限位内。这部分代码完全不需要依赖真正的 PLC，非常适合写 TDD 单元测试。
- **硬件通信层 (`ModbusService`)**：
  - **原则**：只负责和真实的（或模拟的）PLC 打交道。
  - **实现**：维护 Modbus TCP 连接，定时轮询寄存器（当前绝对位置、软限位位置），并将读取到的原始字节转换为 `double` 数值，然后通过 Qt 的信号槽（Signals & Slots）机制发送给 `MotionController`。
