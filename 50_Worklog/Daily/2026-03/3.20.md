# 从 `C_COLLECT_START`看业务

今天刚接到一个任务，就是需要将 3MHz 的波形尽量好的还原出来：

> 基于 FPGA + PCIe 的信号采集系统：
>
> - 采样率：20MHz
> - 原支持频率：2kHz ~ 2MHz
> - 新需求：支持 3MHz 信号观测
>
> 当前系统流程：
>
> 采样 → DMA → find_pulse → FFT → 幅值
>
> 问题：
> 3MHz 下每周期只有约6.7个点，导致：
>
> - 波形起点不稳定（相位漂移）
> - FFT结果不稳定
>
> 当前结论：
> 问题不在 FFT，而在 find_pulse（起点检测）

这部分的逻辑涉及到了`parse_and_handle_message`中的`C_COLLECT_START`流程：

```C
  switch (cmd->valueint) {
    case C_COLLECT_START:
      iRet = handle_collect_start(json, response_buf, response_buf_size);
      if(!iRet){
        set_targetIP(g_cliaddr.sin_addr.s_addr);
        sem_post(&g_app_cfgctrl->start_collect_sem);
      }
      break;
```

然后我们顺着代码来梳理一遍这个逻辑。

## 1. `handle_collect_start`逻辑

- 获取采集持续时间`g_app_cfgctrl->sampleMs`

- 获取预期脉冲的电压（用于后续寻找脉冲）:`g_app_cfgctrl->pulseMV`脉冲峰峰值电压

- 如果存在`gains`字段，则进行每一个通道的增益控制。
  - 更新`g_app_cfgctrl->cfg.gain[i]`：更新用户配置中每个通道的增益
  - 调用`gain_control`，在正式开始“抓取”数据前，按照用户的要求调整好每个通道的放大倍数（dB）
- 解析可选的 `pulseFreq`和 `pulseWidth`
- 计算数据量并申请内存。
  - 调用 `get_datalen` 计算出这次任务一共会产生多少字节的数据。
  - 调用 `req_dataSize` 告诉 PCIe 驱动：“准备好这么大的缓冲区，马上要开闸放水了”。
- 反馈与状态显示
  - 将配置结果回传给上位机，并让面板上的运行灯（Run LED）进入“超快闪”模式，表示设备已准备就绪，进入临战状态。

## 2.`gain_control`逻辑

这个函数的功能就是将业务层的“增益值”（以 dB 为单位），转换为 FPGA 硬件芯片能认出来的“数字编码” （CODE ），并安全地写进硬件寄存器里去控制模拟前端的放大器。

### 2.1 做了什么

针对特定的通道，去读取用户配置（`cfg`）中的目标增益（比如你想让信号放大 20dB）。

然后，它根据当前板卡的硬件型号、以及该通道在运行时校准出的“误差补偿值”，计算出一个精确的硬件控制码，最后把这个控制码写到 FPGA 对应的增益寄存器（`FPGA_REG_GAIN_A` 等）中。

### 2.2 怎么做的

1. 通道参数校验与读取目标增益
2. 识别硬件的板卡类型
3. 获取补偿值：找出这个通道在这个增益档位下的误差补偿值
4. 核心算法计算硬件 CODE
5. 安全写入硬件

## 3.`get_datalen` 是怎么算的？

这个公式非常经典，它将“时间”转化成了“数据量”。

$$\text{Bytes} = \frac{\text{sampleMs}}{1000} \times \text{Channels} \times \text{SampleRate} \times \frac{\text{BitWidth}}{8}$$

- **`sampleMs * 1.0 / 1000`**：将毫秒转为**秒**。
- **`* MAX_CHN_CNT`**：乘以通道数（8 通道）。因为 8 个通道是**同步采集**的，每一时刻产生 8 个数据点。
- **`* g_app_cfgctrl->cfg.sampleRate`**：乘以采样率（每秒采多少次，比如 20,000,000 次。在本系统中等于`DEFAULT_SAMPLERATE` 20MHz）。
- **`* (g_app_cfgctrl->cfg.bitWidth / 8)`**：将位数转为**字节数**。因为 16 位精度等于 2 个字节（Byte），所以每个采样点占 2 字节空间。（这里参数初始化为为`DEFAULT_BITWIDTH`，是 16 位）

**业务关联**：如果你设置采集 1000ms（1秒），采样率 20MHz，8通道，16位。计算结果就是：$1 \times 8 \times 20,000,000 \times 2 = 320,000,000$ 字节（约 305MB）。

## 4.综合后的执行流程

总结下`handle_collect_start`的逻辑：

1. **指令到达**：上位机要求采集 $X$ 毫秒数据。
2. **计算字节**：调用 `get_datalen` 算出基础字节数 $size$。
3. **计算物理空间**：调用 `req_dataSize`，在 $size$ 基础上增加 **100ms 的冗余空间**，并除以 **4080**（每个块的净容量），得到最终需要向硬件申请的 **DMA 块总数**。
4. **硬件联动**：通过 `gain_control` 配置 8 个通道的放大倍数，确保 ADC 能够以最佳幅值量化信号。
5. **启动准备**：设置 `g_need_block_cnt`，随后触发信号量 `start_collect_sem`，通知专门负责 PCIe 搬运的线程（`rcvPcie_proc`）准备接收这批数据。
