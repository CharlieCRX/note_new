## 软限位解决方案

昨天经过与开发人员的交流，确认了软限位的相关逻辑。

之前触碰到软限位后，无法移动（位置和点动）。现在放开了点动逻辑。

在询问：

> 为何触发了软限位后，不能使用绝对定位来撤离到软限位以内

的时候，开发给出的解释为：

- 触发软限位视为故障场景：在绝对位置移动之前，需要判断目标位置是否超出软限位，如果超出则不允许移动。

所以仅在点动模式的时候，才能到达软限位，所以也只能通过反向点动脱离软限位。而不能继续使用绝对位置移动。

## 软限位的读取

现在可以读取软限位的相关数据，方便在绝对位置移动之前限制其移动到非法区域。

### 现在的困境

最重要的就是对于AI工作流的理解和实践不清晰。

我现在关于 AI 的使用方式，就简单局限在了：

> 将问题描述给 AI Web对话框，然后等待解决方案（需求方案分析整理、代码实现以及修改）

如果出现了对话框的条数过多，我会给当前这个对话框一个提示，告诉他总结一份当前的工作进度和对话汇总，方便我将其复制到另一个 AI 对话框中保留其记忆。

但是这样做的最大问题就是：

- 问题太过零散：针对一个大型项目，我既有架构的问题需要向 AI 请教；也需要合理化的需求进行讨论；需要根据需求进行 TDD 开发框架的开发；针对出现的 BUG 和报错，需要继续询问和 debug；对于后续出现的代码逻辑混乱，需要向 AI 请教架构微调和重构的问题...

一个项目，需要开大量的对话窗口。

并且一个窗口问的差不多了满了以后，即将面临卡顿风险。此时我需要手动提示它进行总计然后复制到新的对话框进行新的对话，导致之前的对话可能失真缺失。

------

# 第一层：项目知识仓库（必须做）

在项目目录里建立：

```text
project/
  docs/
    01_requirements.md
    02_architecture.md
    03_domain_model.md
    04_tdd_strategy.md
    05_modbus_protocol.md
  ai/
    prompts/
    summaries/
```

规则：

- 每次重要讨论 → 更新 docs 文件
- 不再依赖对话记忆
- AI 只作为分析工具

这样：

AI 是“思考器”
你的仓库是“记忆体”

------

# 第二层：单项目单 AI 协作入口

你希望：

> 一个项目只问一个 AI

可以这样实现：

建立一个 CLI：

```bash
ai ask "review domain layer"
```

内部逻辑：

1. 自动读取 `docs/*.md`
2. 自动读取相关 src 目录
3. 拼接 prompt
4. 调用 API
5. 输出结果
6. 保存对话到 `ai/logs/`

你获得：

- 单入口
- 自动上下文
- 本地持久化
- 无卡顿
