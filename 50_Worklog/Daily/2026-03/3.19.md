#  RTFSC 五步法

## ① 找入口（main）

已经给了 `main.c`，我们一起看关键路径：

```C
int main()
{
  comm_init();
  create_workers();
  dealCmdLoop(NULL);
}
```

- `comm_init()`
  - 初始化结构体`T_app_cfgctrl`指针变量`*g_app_cfgctrl`
  - 执行`adc_training`，如果训练失败直接退出
  - 初始化开始采集信号`start_collect_sem`和数据准备完毕信号`data_ready_sem`为0
  - 读取配置文件`CFG_FILE`到指针`g_app_cfgctrl`的成员变量`user_cfg_t`中
  - 初始化 pcie
  - `init_dataSend`：创建一个后台线程，等待数据就绪信号，然后通过 TCP 网络将数据发送至远端服务器 AMS
- `dealCmdLoop(NULL)`
  - 监听`g_lsn_port`端口数据
  - 阻塞线程并监听，直到 UDP 网络包发送过来
  - 解析 JSON 格式的指令并处理
  - 将处理结果（JSON 格式的回复）原路返回给发送指令的人

### 整个系统的闭环逻辑

如果我们把前后三段代码串起来，你会发现这是一个非常完整的**嵌入式控制系统**逻辑：

1. **准备阶段**：初始化信号量（红绿灯）、创建发送数据的线程（快递员）、创建接收指令的线程（指挥部）。
2. **等待阶段**：
   - 数据发送线程在 `sem_wait(data_ready_sem)` 处睡觉。
   - 采集线程在 `sem_wait(start_collect_sem)` 处睡觉。
   - **指挥部线程**在 `recvfrom` 处等待指令。是等命令再采集。
3. **触发阶段**：
   - 你从电脑发了一个 JSON 消息：`{"cmd": 1, "msg": "start"}` 给设备。
   - **指挥部**收到消息，解析出 `C_COLLECT_START`。
   - **指挥部**执行 `sem_post(start_collect_sem)`。
4. **联动阶段**：
   - **采集线程**苏醒，开始采数据。
   - 采集完数据后，采集线程执行 `sem_post(data_ready_sem)`。
   - **发送线程**苏醒，把数据通过 TCP 发送到你指定的 IP 端口。

## ② 找“全局大脑”（核心结构体）

在 `commdef.h` 里有个关键东西：

👉 `T_app_cfgctrl` 

这个结构体你要当成：

> 🧠 **整个系统的“大脑 + 状态中心”**

比如：

```
volatile uint8_t *dmaMemUserAddr;
volatile uint8_t *pcieBarUserAddr[PCIE_BARCNT];
```

👉 你要意识到一件事：

这是**直接操作硬件内存（DMA + PCIe）**。

这些 `volatile uint8_t*` 指针，是要直接操作 FPGA 硬件映射的内存。

PCIe 设备 FPGA 通过 **MMIO（内存映射输入输出）** 方式与 CPU 交互。设备内部的寄存器和内存缓冲区被映射到 CPU 的物理地址空间，CPU 可直接使用访存指令读写这些地址，实现对设备的控制。

`volatile` 是一个类型限定符，告知编译器该变量的值可能**在程序控制流之外被意外修改**，因此编译器**不得对该变量的访问进行优化**，每次使用都必须直接从内存地址读取或写入。

# 项目理解

## main在干嘛

接收命令并启动程序。

- 初始化采集和发送信号和线程
- 等待网络命令

## `T_app_cfgctrl` 是什么

程序的灵魂吧。所有的功能都要看此结构的成员变量走。

## 画一个最简单流程

数据是怎么过来并处理的。

例如接收到网络采集命令：

```
{
	"cmd":1
}
```

看下如何接收命令并进行处理的：

1. **UDP 线程**：收到 JSON。

2. **`handle_collect_start`（本函数）**：

   - 设置增益（`gain_control`）。

   - **挖坑准备存数据（`req_dataSize`）**。

   - 写好回执单（`rsp_str`）。

   - 疯狂闪灯。

3. **返回到 `parse_and_handle_message`**：

   - 执行 `set_targetIP`（记住发件人）。

   - 执行 **`sem_post(&g_app_cfgctrl->start_collect_sem)`**（**真正按下采集开关！**）。

然后我们看下，采集是怎么做的。

1. 线程`rcvPcie_proc`接收到信号`start_collect_sem`后继续运行
2. 提示`sample_req`要采集`g_need_block_cnt`次数据
   - 每一次采集后的数据存放的地址、长度等标志位信息，将会存放在 BAR 的第二个地址空间的特定位置中（`pcieBarUserAddr[1]`）
   - 每次采集命令下发后，会主动循环`200,0000`次来阻塞进程，只为等待 FPGA 给出采集成功的标志`DMA_READ_DONE`为1
3. `g_need_block_cnt`次采集成功后，将调用`copy2useraddr_and_findpulse`来对采集好的数据进行处理
