## 1. 背景

目前 `cm_server` 的开发采用「PC 编译 + 多块开发板联调」的方式。

每次修改代码并重新编译后，都需要将最新生成的 `cm_server` 部署到多块开发板（如 `10.1.2.151`、`10.1.2.152`、`10.1.2.154`）进行验证。

当前部署流程完全依赖人工操作，需要分别登录每一块开发板，并执行以下步骤：

1. SSH 登录开发板。
2. 查询 `cm_server` 相关进程。
3. 手动记录并结束相关进程。
4. 确认进程已经退出。
5. 使用 `scp` 或其他方式将最新的 `cm_server` 上传至 `/www/` 目录。
6. 修改程序执行权限（如需要）。
7. 重新启动 `/www/cm_server.sh`。
8. 检查服务是否启动成功。
9. 对其它开发板重复上述流程。

随着开发频率增加，上述流程每天可能需要重复执行几十次，属于典型的重复性工作。

------

# 2. 当前存在的问题

目前人工部署主要存在以下问题：

## （1）效率低

每次部署均需要：

- 登录多块开发板；
- 多次输入密码；
- 重复执行相同命令。

一次完整部署通常需要数分钟，严重影响开发效率。

------

## （2）容易遗漏步骤

例如：

- 忘记停止旧进程；
- 上传到了错误的开发板；
- 忘记重新启动服务；
- 某一块开发板未更新。

这些问题会导致：

- 不同开发板运行不同版本的软件；
- 调试结果不一致；
- 增加问题定位难度。

------

## （3）维护成本高

随着开发板数量增加，例如：

- 151
- 152
- 154
- 后续新增设备

人工部署的工作量会线性增长。

------

## （4）影响调试效率

对于 `cm_server` 这类后端服务，每次修改代码后都需要重新部署。

如果部署过程耗时较长，会降低：

- Bug 修复效率；
- 接口验证效率；
- 联调效率。

------

# 3. 当前限制

原计划采用 SSH 公钥认证实现免密登录。

但经过排查发现：

开发板 RootFS 中大量目录（包括 `/`、`/root`、`/etc`）的 Owner 为 UID=1000，而不是 UID=0(root)。

由于 OpenSSH 开启了：

```
StrictModes yes
```

因此会拒绝公钥认证：

```
Authentication refused:
bad ownership or modes for directory /root
```

该问题属于 RootFS 制作流程问题，短期内无法通过脚本解决。

因此当前自动部署方案采用：

- `sshpass`
- SSH 密码认证

实现无人值守部署。

------

# 4. 自动部署脚本设计目标

部署脚本应完成以下工作：

1. 自动连接所有目标开发板；
2. 自动停止当前运行的 `cm_server`；
3. 自动上传最新编译生成的 `cm_server`；
4. 自动修改执行权限；
5. 自动启动 `cm_server.sh`；
6. 自动检查服务是否启动成功；
7. 输出每块开发板部署结果；
8. 即使某一块开发板部署失败，也不会影响其它开发板继续部署。

整个部署过程仅需执行一次命令即可完成。

------

# 5. 部署流程

```
PC 编译完成
        │
        ▼
执行 deploy.sh
        │
        ├────────────► 10.1.2.151
        │                 │
        │                 ├─ 停止 cm_server
        │                 ├─ 上传新程序
        │                 ├─ chmod +x
        │                 ├─ 启动 cm_server.sh
        │                 └─ 检查运行状态
        │
        ├────────────► 10.1.2.152
        │                 │
        │                 ├─ 停止服务
        │                 ├─ 上传程序
        │                 ├─ 启动服务
        │                 └─ 检查状态
        │
        └────────────► 10.1.2.154
                          │
                          ├─ 停止服务
                          ├─ 上传程序
                          ├─ 启动服务
                          └─ 检查状态
```

------

# 6. 使用方式

编译完成后，仅需执行：

```
make deploy
```

或者：

```
make -j && make deploy
```

即可完成所有开发板的软件更新。

------

# 7. 预期收益

部署脚本引入后，可获得以下收益：

- 将多块开发板部署统一为一次命令执行，提高开发效率。
- 避免重复的人工操作，减少误操作和遗漏。
- 保证所有开发板运行相同版本的软件，提升联调一致性。
- 为后续持续集成（CI）、自动测试及批量升级提供统一的部署入口。
- 随着开发板数量增加，仅需维护一份部署脚本即可扩展到更多设备，降低维护成本。

------

# 8. 后续优化方向

待开发板 RootFS 文件所有权问题修复后，可进一步启用 SSH 公钥认证，实现真正的免密登录，并移除 `sshpass` 依赖。

此外，部署脚本还可进一步扩展为通用部署工具，支持不同应用（如 `cm_server`、`ruclock`、`collecter` 等）的统一部署，进一步提升项目自动化水平。
