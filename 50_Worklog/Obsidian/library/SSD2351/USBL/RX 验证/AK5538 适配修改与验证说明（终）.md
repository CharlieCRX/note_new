## 1. 文档目的

本文整理当前内核源码中，为使 PCUPID 平台上的 AK5538 能够被 ASoC 注册、生成采集设备并完成音频采集而引入的修改。

当前产品目标配置为：

| 项目           | 当前配置                            |
| ------------ | ------------------------------- |
| AK5538 功能    | ADC，仅采集                         |
| ALSA 设备      | `card 0, device 1`，即 `hw:0,1`   |
| PCM 节点       | `/dev/snd/pcmC0D1c`             |
| 采样率          | 192000 Hz                       |
| 通道数          | 2                               |
| PCM 格式       | `S32_LE`                        |
| 串行格式         | I2S，2 个 32-bit slot             |
| AK5538       | BCLK/LRCK provider（主设备）         |
| CPU BACH2 RX | BCLK/LRCK consumer（从设备）         |
| MCLK 来源      | FPGA，当前由外部程序手动配置                |
| MCLK         | 24.576 MHz                      |
| BCLK 理论值     | 12.288 MHz，即 `192 kHz × 2 × 32` |
| LRCK 理论值     | 192 kHz                         |
| 物理数据线        | AK5538 SDTO1 到 CPU I2S RX 数据输入  |

注意：本源码目录没有可用的 Git 提交基线，因此本文是根据当前源码、文件中的适配逻辑和板端调试过程整理的功能修改清单，不是由 `git diff` 自动生成的逐行补丁。

## 3. 各文件修改说明

### 3.1 `sound/soc/codecs/ak5538.c`

实现 AK5538 ASoC codec 驱动，主要内容如下：

1. 注册 I2C 驱动，支持以下 compatible：
   - `ak5538`
   - `asahi-kasei,ak5538`
2. 使用 regmap 管理 `0x00`～`0x05` 寄存器。
3. 注册名为 `ak5538-hifi` 的 capture DAI：
   - 1～8 通道能力；
   - 支持 `S16_LE`、`S24_LE`、`S32_LE`；
   - 采集方向，不提供 playback stream。
4. 增加 8 路模拟输入、ADC 和 `SDTO` 的 DAPM widget/route。
5. 在 `set_fmt()` 中支持：
   - AK5538 为 BCLK/LRCK provider：`SND_SOC_DAIFMT_CBP_CFP`；
   - I2S 和 Left Justified 数据格式。
6. 在 `set_sysclk()` 中要求 MCLK 为 codec 输入：
   - 方向必须是 `SND_SOC_CLOCK_IN`；
   - 频率不能为 0；
   - 只记录/校验外部 MCLK，不直接控制 FPGA。
7. 在 `hw_params()` 中按采样率设置 CKS、按位宽设置 DIF，并执行：
   - `RSTN=0`；
   - 写 CKS/DIF；
   - 确认 PCM 模式；
   - `RSTN=1`；
   - 等待 ADC 数据稳定。
8. 192 kHz 对应的配置为：
   - MCLK = 24.576 MHz；
   - MCLK/fs = 128；
   - CKS = 0；
   - `S32_LE` 对应 32-bit I2S DIF。
9. `set_tdm_slot()` 根据 `slots × slot_width` 设置 Normal/TDM128/TDM256/TDM512。当前 2 × 32 = 64 bit，因此 codec 使用 Normal I2S，而不是 TDM256。
10. 读取板级约束属性：
    - `ak5538,fixed-sample-rate`；
    - `ak5538,fixed-channels`。
11. 在 ALSA startup 阶段施加单值约束，避免用户态打开错误的采样率或通道数。
12. 对 192 kHz 增加 MCLK 一致性校验：声明的 MCLK 不是 24.576 MHz 时返回 `-EINVAL`。

当前 192 kHz、2 通道、32-bit 采集时，预期 regmap 值为：

```text
0: ff    # 8 个 ADC 通道上电
1: 01    # RSTN=1，Normal 模式
2: 07    # CKS=0，32-bit I2S，HPF enable
3: 00    # Normal I2S，非 TDM128/256/512
4: 00    # PCM、Sharp Roll-Off
5: 00
```

### 3.2 `sound/soc/codecs/ak5538.h`

补充 AK5538 的寄存器和位域定义，包括：

- `POWER_MGMT1`～`DSD`，地址 `0x00`～`0x05`；
- 8 路 ADC power bit；
- `RSTN` 和 MONO 模式；
- `CKS[3:0]`、`DIF[1:0]`、`HPFE`；
- Normal/TDM128/TDM256/TDM512；
- PCM/DSD 和数字滤波位。

这里最关键的是依据 AK5538 手册修正 CKS、DIF 和 TDM 位的位置，否则寄存器虽可写入，但输出时钟和串行数据格式会错误。

### 3.3 `sound/soc/codecs/Kconfig`、`Makefile` 和 `.config`

构建系统增加：

```text
CONFIG_SND_SOC_AK5538=y
```

`Kconfig` 中配置项依赖 I2C 并选择 `REGMAP_I2C`；`Makefile` 将其构建为 `snd-soc-ak5538.o`。

### 3.4 `sound/soc/sstar/pcupid/sstar_asoc_card.c`

机器驱动修改解决了“AK5538 已 probe，但 `/proc/asound/pcm` 只有 ACM8816 playback”的问题：

1. 根据 DTS 的 `codec-name` 或 `compatible` 识别 `ak5538-hifi`。
2. 为每条普通 DAI link 按相同索引选择不同 CPU DAI：
   - link 0：`bach1` + `acm8816-hifi`；
   - link 1：`bach2` + `ak5538-hifi`。
3. 将 ACM8816 link 标记为 `playback_only=1`。
4. 将 AK5538 link 标记为 `capture_only=1`。
5. AK5538 link 使用：

```text
I2S | NB_NF | CBP_CFP
```

其中 `CBP_CFP` 表示 codec 提供 BCLK 和 LRCK。

6. 增加 `late_probe()`：如果 vendor ASoC core 没有为第二条普通 runtime 创建 PCM，则在声卡注册完成前调用：
   - `snd_soc_runtime_set_dai_fmt()`；
   - `soc_new_pcm()`；
   - `snd_soc_pcm_dai_new()`。
7. 增加 link 和 runtime 日志，便于确认 PCM、playback/capture substream 以及两端 DAI capability。

该修改最终产生：

```text
00-01: aux-snd-control ak5538-hifi-1 :  : capture 1
/dev/snd/pcmC0D1c
```

### 3.5 `sound/soc/sstar/pcupid/sstar_bach.c`

CPU DAI 层增加了 AK5538 专用处理：

1. 打开 stream 时从 DTS 获得 `mclk0-freq`。
2. 对 AK5538 调用 `snd_soc_dai_set_sysclk()` 时使用 `SND_SOC_CLOCK_IN`，表示 MCLK 是 AK5538 的输入。
3. 为 AK5538 启用 `I2S_RX_A` 路径。
4. 从 `I2SA_RX0_CHANNEL` 读取 2/4/8 通道配置，并调用 codec 的 `set_tdm_slot()`。
5. 2 通道时向 codec 配置为 `2 slots × 32 bits`，对应 Normal I2S；4/8 通道时进入 TDM。
6. 根据声卡 DAI fmt 和 DTS，将 AK5538 接收方向设为 CPU slave、I2S、6-wire。

重要说明：当前源码的 `sstar_bach_hw_params()` 中仍存在一段早期 8 通道试验代码，会对 AK5538 强制设置：

```text
E_MHAL_AUDIO_MODE_TDM_SLAVE
E_MHAL_AUDIO_4WIRE_OFF
E_MHAL_AUDIO_CHN_8
```

并打印 `AK5538 RX config: CPU slave, 6-wire, TDM8`。这与当前板级 DTS 的 2 通道和 codec 的 Normal I2S 不完全一致。现有板测可以从第一对数据中获得 25 kHz 波形，但这仍是一个配置一致性风险。后续清理时应让 HAL 通道数使用 `params_channels(params)` 或 DTS 的 `i2s-rx0-channel`，并在 2 通道时明确选择普通 I2S，而不是继续强制 TDM8。

### 3.6 `sound/soc/sstar/pcupid/sstar_pcm.c`

PCM/DMA 层修改解决了“设备节点存在，但 `arecord` 返回 I/O error、文件为 0 字节”的问题：

1. 不在 component probe 时固定预分配大块 DMA 内存，改到 `hw_params()` 按实际 buffer 大小动态分配。
2. 从 BACH DTS DMA 节点恢复正确的 stream-to-DMA 映射：
   - BACH1：RDMA0/WDMA0；
   - BACH2：RDMA1/WDMA1。
3. 避免 `kzalloc` 后 BACH2 错误地继续使用默认 DMA0。
4. 将分配得到的地址和大小同步到：
   - vendor PCM config；
   - ALSA runtime；
   - substream DMA buffer。
5. 在 `hw_free()`/remove 时释放动态 DMA 缓冲区，避免反复录音造成内存泄漏。
6. 按通道数把 I2S RX A 接口 attach 到对应 AI DMA；当前双通道只 attach `I2S_RX_A_0_1` 到 WDMA1。
7. capture stop/close 时执行对应 detach、AI close 和 DMA 释放。
8. `request_irq()` 使用稳定的设备名作为 IRQ action name，修复空名称触发的 `__proc_create()` WARN/调用栈。
9. 增加 DMA 编号、缓冲区、AI attach 等关键日志。

预期日志应包含：

```text
sstar-bach 1f2a0400.bach1: PCM DMA mapping: rdma=0 wdma=0 irq=65
sstar-bach 1f2a0400.bach2: PCM DMA mapping: rdma=1 wdma=1 irq=65
...
AI attach succeeded for dma_id 1, channels 2
```

### 3.7 `arch/arm64/boot/dts/sstar/pcupid.dtsi`

SoC 公共 DTS 中完成以下连接：

1. 在 I2C1 上增加 AK5538：

```dts
ak5538: ak5538@10 {
    compatible = "ak5538";
    reg = <0x10>;
    #sound-dai-cells = <0>;
    status = "okay";
};
```

2. 主声卡包含两条 CPU/codec 配对：

```dts
sstar-audio-card,cpu {
    sound-dai = <&bach1>, <&bach2>;
};
sstar-audio-card,codec {
    sound-dai = <&acm8816>, <&ak5538>;
};
```

3. BACH2 使用独立 `bach_dma2`。
4. `bach_dma2` 配置为 RDMA1/WDMA1，避免与 ACM8816 的 BACH1 DMA0 冲突。
5. 公共 I2S RX 参数为 CPU slave、I2S、6-wire，并提供 2 通道默认值。

### 3.8 `arch/arm64/boot/dts/sstar/pcupid-ssm001c-s01a-voip.dts`

板级 DTS 将最终产品参数覆盖为：

```dts
&sound {
    i2s-rx0-tdm-mode = <2>;       /* CPU slave */
    i2s-rx0-tdm-fmt = <1>;        /* I2S */
    i2s-rx0-tdm-wiremode = <2>;   /* 6-wire */
    i2s-rx0-channel = <2>;
};

&bach2 {
    mclk0-freq = <24576000>;
};

&ak5538 {
    ak5538,fixed-sample-rate = <192000>;
    ak5538,fixed-channels = <2>;
};
```

`mclk0-freq` 是驱动对实际外部时钟的声明和校验值，并不会自动写 FPGA 寄存器。

## 4. FPGA 和硬件前置条件

在打开 AK5538 PCM 设备前，必须由外部控制程序将 FPGA 寄存器 `0x2[3:0]` 写为 `7`，使 AK5538 MCLK 引脚实际收到 24.576 MHz。

时钟职责如下：

```text
FPGA       --24.576 MHz MCLK--> AK5538
AK5538     --12.288 MHz BCLK--> CPU BACH2 RX
AK5538     --192 kHz LRCK-----> CPU BACH2 RX
AK5538     --SDTO1 data-------> CPU BACH2 RX
```

只修改 DTS 的 `mclk0-freq` 不会改变 FPGA 输出。若物理 MCLK 仍是 12.288 MHz，而驱动按 192 kHz/128fs 配置，AK5538 输出采样时钟必然不正确。

当前 2 通道模式只接收 SDTO1 上的第一对采样数据，即 AIN1/AIN2（原理图标注 AOUT0/AOUT1）。产品需要的 AOUT2/AOUT3/AOUT4 位于其他 AK5538 输入通道；由于 FPGA/CPU 接口当前只接收 1/2 通道，现有方案不能同时获取全部 5 路，需另行确认额外数据线、TDM 接收能力或 FPGA 聚合方案。

## 5. 问题演进与对应修复

| 现象 | 根因 | 对应修改 |
| --- | --- | --- |
| I2C probe 成功，但只有 `pcmC0D0p` | AK5538 link 没有生成 capture PCM | capture-only 标志、独立 BACH2 link、`late_probe()` 补建 PCM |
| card 注册时报 mixer control `-16`，随后 cleanup 崩溃 | 两条普通 link 复用同一 CPU DAI，widget/control 冲突 | link 0/1 分别使用 BACH1/BACH2 |
| `arecord` 打开时报 `Invalid SYSCLK: freq=0` | MCLK 声明缺失或方向错误 | BACH2 `mclk0-freq`、AK5538 使用 `SND_SOC_CLOCK_IN` |
| PCM 可打开但 read 返回 `Input/output error` | BACH2 仍使用 DMA0或 AI RX attach 不正确 | BACH2 RDMA1/WDMA1、保存 `dma_id`、I2S_RX_A attach |
| `__proc_create()` WARN | IRQ action name 为空 | `request_irq()` 改用 `dev_name()` |
| 48 kHz 测得 LRCK 8 kHz/BCLK 异常 | 外部 MCLK 与 AK5538 CKS/串行配置不匹配 | 明确 FPGA MCLK，按手册设置 CKS 和主从关系 |
| 需要采集 20～30 kHz 模拟信号 | 48 kHz Nyquist 频率仅 24 kHz，不适合 25/30 kHz | 改为 192 kHz，Nyquist 频率为 96 kHz |

## 6. 板端验证方法

### 6.1 验证驱动、PCM 和设备节点

```sh
dmesg | grep -iE 'ak5538|runtime\[|PCM DMA mapping|AI attach|MCLK|BCLK|LRCK'
cat /proc/asound/cards
cat /proc/asound/pcm
arecord -l
ls -l /dev/snd
```

关键结果应为：

```text
00-00: HiFi PCM acm8816-hifi-0 :  : playback 1
00-01: aux-snd-control ak5538-hifi-1 :  : capture 1
/dev/snd/pcmC0D0p
/dev/snd/pcmC0D1c
```

### 6.2 验证 ALSA 参数约束

```sh
arecord -D hw:0,1 --dump-hw-params \
    -t raw -f S32_LE -r 192000 -c 2 -d 1 /dev/null
```

应接受 192000 Hz、2 通道、S32_LE；使用 48000 Hz 或 8 通道打开时应被固定约束拒绝。

### 6.3 验证 AK5538 寄存器

```sh
cat /sys/kernel/debug/regmap/1-0010/registers
```

在 PCM 已执行 `hw_params` 后，对照第 3.1 节的预期值检查。

### 6.4 录制 WAV

```sh
arecord -D hw:0,1 \
    -t wav -f S32_LE -r 192000 -c 2 -d 5 \
    /mnt/ak5538-192k-stereo.wav
```

5 秒音频的 PCM 数据量理论值：

```text
192000 × 2 × 4 × 5 = 7,680,000 bytes
```

WAV 文件还会增加文件头。板上内存较小，曾出现 OOM 杀死 `sshd-session` 的情况，因此不要长期录制到 tmpfs 类型的 `/tmp`；优先使用 eMMC、NAND、NFS 或 U 盘等持久存储路径。

### 6.5 验证采样数据

向 AIN1/AIN2 输入已知幅度的 25 kHz 差分正弦信号，检查：

- 左右声道时域波形连续，无周期性丢样；
- FFT 主峰位于约 25 kHz；
- 不削顶，峰值保留合理余量；
- 5 秒采集持续产生数据，文件大小接近理论值；
- 示波器实测 MCLK≈24.576 MHz、BCLK≈12.288 MHz、LRCK≈192 kHz；
- dmesg 中没有 `xrun`、DMA、AI start 或 IRQ 错误。

当前板测截图已经显示双声道 WAV 中存在连续波形，频谱主峰约为 24997.79 Hz，说明 25 kHz 信号已经进入采集链路。仍应结合时钟实测、长时间无丢样测试和幅度/失真指标，才能判定完整采集质量。

## 7. 编译与烧录注意事项

目标 DTB 为：

```text
arch/arm64/boot/dts/sstar/pcupid-ssm001c-s01a-voip.dtb
```

通常不需要手动删除旧 DTB；只要 `.dts/.dtsi` 时间戳更新，Kbuild 会重新生成。若需要强制确认，可单独构建目标 DTB并检查时间戳，但不要只烧录内核而遗漏新 DTB。

AK5538 驱动、BACH/PCM 驱动和 DTS 都有修改，因此最终烧录包必须同时包含新 kernel image 和新 DTB。本文整理过程中没有执行编译。

## 8. 当前结论和遗留事项

当前已经确认完成的功能链路为：

```text
I2C probe
  -> AK5538 component/DAI 注册
  -> 第二条 capture-only DAI link
  -> pcmC0D1c 创建
  -> BACH2/WDMA1 分配和 attach
  -> FPGA 24.576 MHz MCLK
  -> AK5538 192 kHz BCLK/LRCK provider
  -> 双通道 S32_LE WAV
  -> 25 kHz 信号可见
```

尚需处理或确认：

1. 清理 `sstar_bach_hw_params()` 中强制 TDM8 的遗留逻辑，使 CPU HAL 与 2 通道 Normal I2S 完全一致。
2. 将 FPGA MCLK 寄存器配置纳入正式启动流程，避免依赖人工写寄存器。
3. 确认产品所需 5 路模拟信号的数字传输方案；当前双通道只能获得第一对通道。
4. 做长时间录制、丢帧、左右通道对应、幅频、THD+N 和满幅输入测试。
5. 若后续变更采样率，必须同时修改并校验 FPGA MCLK、AK5538 CKS、DTS MCLK 声明、ALSA rate 约束和 BCLK/LRCK 实测值。
