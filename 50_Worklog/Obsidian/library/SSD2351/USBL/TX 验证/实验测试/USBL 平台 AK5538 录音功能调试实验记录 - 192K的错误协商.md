## 三、观测现象

### 3.1 现象一：采样率协商异常

**日志输出**：

```text
Warning: rate is not accurate (requested = 128000Hz, got = 192000Hz)
         please, try the plug plugin 
Recording WAVE '/tmp/test.wav' : Signed 32 bit Little Endian, Rate 192000 Hz, Mono
```

**现象描述**：

| 项目 | 用户请求值 | 实际协商值 |
| :--- | :--- | :--- |
| 采样率 (`-r`) | 128000 Hz | **192000 Hz** |
| 硬件 LRCK（示波器测量） | — | **128000 Hz** |

**关键矛盾**：
- 用户请求 `-r 128000`
- ALSA 协商后实际使用 `192000`
- 硬件 LRCK 实测为 `128000`
- **CPU 端采样率与 AK5538 实际采样率不一致**

### 3.2 现象二：Kernel Oops

**日志输出**：

```text
Unable to handle kernel paging request at virtual address fffe0000ffff00c0
...
pc : snd_pcm_ioctl+0x78/0xf78
...
Unable to handle kernel paging request at virtual address 00000000ffff0020
pc : CamOsSpinLockIrqSave+0x18/0x5c
lr : HalAudDmaGetLevelCnt+0x3c/0xc0
```

**Call Trace**：

```text
sstar_snd_irq_handler
  → mhal_audio_ai_get_curr_datalen
    → HalAudDmaGetLevelCnt
      → CamOsSpinLockIrqSave
```

**崩溃地址分析**：

| 地址 | 含义 |
| :--- | :--- |
| `0xfffe0000ffff00c0` | 用户空间与内核空间之间的地址范围，未映射 |
| `0x00000000ffff0020` | 用户空间地址范围，未映射 |

**结论**：中断处理函数访问了未初始化的指针或已释放的内存。

### 3.3 现象三：DMA 句柄未初始化

日志显示：

```text
[ALSA ERROR][mhal_audio_ai_start]: enAiDma [0] bIsAttached FALSE
```

**说明**：AI DMA 在 `start` 时仍未完成 `attach`。


## 四、已完成的代码修改

为打通录音链路，已完成以下修改：

### 4.1 `acm8816.c` 修改

在 `acm8816_pcm_hw_params` 中，对 Capture 方向跳过 I2C 操作：

```c
if (substream->stream == SNDRV_PCM_STREAM_CAPTURE) {
    dev_info(&priv->i2c->dev, "Capture: skip I2C config, AK5538 already configured by script\n");
    return 0;
}
```

**效果**：✅ 消除了 `i2c-1 no ack` 错误。

### 4.2 `sstar_pcm.c` 修改

#### 4.2.1 移除 probe 预分配，改为动态分配

在 `sstar_pcm_probe` 中移除 `alloc_dmem` 调用，在 `sstar_pcm_hw_params` 中按需分配。

**效果**：✅ DMA 缓冲区大小与请求大小匹配，消除 `memset` 越界。

#### 4.2.2 添加 `mhal_audio_ai_attach`

在 `sstar_pcm_hw_params` 的 capture 分支中，在 `mhal_audio_ai_config` 之后、`mhal_audio_ai_open` 之前添加：

```c
MHAL_AI_Attach_t tAiAttach;
memset(&tAiAttach, 0, sizeof(tAiAttach));
tAiAttach.enAiDma = ss_pcm_stream->dma_id;
tAiAttach.nAiDmaCh = 0;
tAiAttach.enAiIf = E_MHAL_AI_IF_I2S_RX_A_0;
tAiAttach.nUseSrc = 0;
if (mhal_audio_ai_attach(&tAiAttach)) {
    dev_err(...);
    return -EFAULT;
}
```

**效果**：⚠️ 部分解决 `bIsAttached FALSE`，但中断中仍访问非法地址。

#### 4.2.3 添加 `sstar_pcm_hw_free`

在 `sstar_pcm_hw_free` 中释放 DMA 缓冲区，防止内存泄漏。

#### 4.2.4 注册 `hw_free` 回调

在 `sstar_pcmengine_register` 中添加 `.hw_free = sstar_pcm_hw_free`。


## 五、问题分析

### 5.1 采样率协商不匹配（现象一）

**根本原因**：

`arecord` 请求的 `128000 Hz` 不在 ALSA 核心计算出的可用速率列表中。

在 ASoC 框架中，`snd_pcm_hw_params` 阶段会调用 `snd_soc_runtime_calc_hw`，该函数遍历 CPU DAI 和 Codec DAI 支持的速率，计算交集。

当前 Codec DAI（`acm8816_dai`）的 `rates` 字段定义为：

```c
.rates = SNDRV_PCM_RATE_48000 | SNDRV_PCM_RATE_96000 | SNDRV_PCM_RATE_192000
```

**不包含 128000**。ALSA 核心在参数协商时，将 128000 向上取整到最近的合法值 192000，并以此值调用 `hw_params`。

**影响**：
- CPU I2S 控制器被配置为 192000 Hz
- AK5538 实际工作在 128000 Hz
- 两者采样率不匹配，录制的音频数据是错误的

### 5.2 Kernel Oops（现象二）

**根本原因**：

中断处理函数 `sstar_snd_irq_handler` 中调用了 `mhal_audio_ai_get_curr_datalen`，该函数内部访问了 HAL 层的一个指针。该指针指向的上下文对象未被正确初始化（可能是因为 `mhal_audio_ai_attach` 未完成或顺序不正确），导致访问了非法地址 `0xffff0000`。

**可能的根因**：

| 序号 | 可能原因 | 说明 |
| :--- | :--- | :--- |
| 1 | HAL 层内部上下文未初始化 | `mhal_audio_ai_attach` 未成功完成，或传入的 `dma_id` 无效 |
| 2 | 中断在 attach 之前被触发 | 中断使能过早，在 DMA 上下文建立前就触发了中断 |
| 3 | DMA 句柄指针错误 | `dma_id` 对应的内部对象已被释放或从未分配 |

## 七、下一步调试计划

### 7.1 解决采样率协商问题

**目标**：让 ALSA 接受 128000 Hz，使 CPU 端采样率与 AK5538 一致。

**方案**：
1. 修改 `acm8816_dai` 的 `rates` 字段，添加对 128000 的支持：
   ```c
   .rates = SNDRV_PCM_RATE_48000 | SNDRV_PCM_RATE_96000 | SNDRV_PCM_RATE_192000 | SNDRV_PCM_RATE_KNOT,
   .rate_min = 8000,
   .rate_max = 192000,
   ```

2. 或修改 `sstar_hardware`，明确支持 8000~192000 范围。

### 7.2 解决 Kernel Oops

**目标**：消除中断处理中的非法地址访问。

**排查方向**：
1. 确认 `mhal_audio_ai_attach` 是否真的成功
2. 检查 `mhal_audio_ai_start` 是否在 attach 完成之后才被调用
3. 检查中断是否在 DMA 上下文完全建立后才被使能
4. 检查 `dma_id` 在 HAL 层是否对应有效的对象

**建议**：
1. 在 `mhal_audio_ai_attach` 返回值后增加更详细的日志
2. 在 `mhal_audio_ai_get_curr_datalen` 前检查 `bIsAttached` 状态
3. 使用 `if (bIsAttached) { ... }` 保护中断处理中的访问
