# 1.什么叫逻辑管脚（Logical Pad）

例如

```
PAD_GPIOD_04
```

很多人第一次都会疑惑：

> 为什么不是 GPIO4？

因为 SoC 内部并不是每个引脚只能干一件事。

例如：

```
PAD_GPIOD_04
```

其实表示：

> **芯片内部的一个 PAD（引脚资源）。**

这个 PAD 可以配置很多功能。

例如：

```
PAD_GPIOD_04
```

可能可以配置成

```
GPIO

或者

I2S_BCLK

或者

PWM

或者

UART_TX

或者

SPI_CLK
```

也就是说：

```
PAD_GPIOD_04
```

只是：

> 这块焊盘（Pad）的名字。

它最后干什么，要由 PinMux 决定。

# 2.什么是 ARMTmux？

如果说《USBL水面设备管脚定义及器件地址说明》告诉你"最终要用哪些管脚"，那么《SSD2351X_HW Checklist V1.03_20240816.xlsx》的页面【ARMTmux 】告诉你的就是"如何让这些管脚变成对应的功能"。

| 文档                  | 回答的问题                                            |
| --------------------- | ----------------------------------------------------- |
| USBL 水面设备管脚定义 | PCB 上最终接的是哪几个引脚？                          |
| ARMTmux               | 这些 PAD 可以有哪些功能？需要设置哪个寄存器才能切换？ |

## 什么是 ARMTmux？

Mux 就是 **Multiplexer（复用器）**。

现代 SoC 不可能给每一个外设都准备一套独立引脚，所以一个 PAD 可以承担很多功能。

例如：

```
PAD_GPIOD_04
        │
        ▼
    PinMux(选择器)
 ┌─────┼──────────┐
 │     │          │
GPIO  UART_RX  I2S0_TX_BCK
```

CPU 会通过寄存器控制这个 PinMux。

例如：

```
reg_i2s0_4w_tx_mode = 1
```

那么：

```
PAD_GPIOD_04
        ↓
I2S0_TX_BCK
```

如果：

```
reg_uart5_mode = 1
```

那么同一个 PAD 就变成：

```
PAD_GPIOD_04
        ↓
UART5_RX
```

**所以 ARMTmux 本质上就是整个芯片的 PinMux 配置表。**

## `reg_i2s0_4w_tx_mode` 是什么

`reg_i2s0_4w_tx_mode` **本质上是一个寄存器（更准确地说是寄存器中的一个“位域”）**。

结合提供的 `SSD2351X_HW Checklist` 中的 `ARMTmux` 工作表，我们拆解它的具体含义和配置方法。

### 1. 它是什么？（寄存器地址与位域）

在 `ARMTmux` 工作表的第 4 行（寄存器定义行），明确写明了：

> **`reg[103cC5]#1 ~ #0`**

这揭示了它的物理本质：

- **`reg`**：表示这是一个芯片内部的硬件寄存器。
- **`103cC5`**：这是该寄存器的**物理地址**（基地址偏移）。
- **`#1 ~ #0`**：这表示该寄存器中的 **第 0 位和第 1 位**（即一个 2 位的二进制字段）。

所以，`reg_i2s0_4w_tx_mode` 并不是一个独立的、全新的寄存器，而是地址 `0x103cC5` 这个 32 位（或 16 位）寄存器中的**最低 2 个 bit（Bit0 和 Bit1）**。芯片通过这 2 个 bit 的组合（`00`、`01`、`10`、`11`）来控制该功能模块下具体引脚组的切换。

------

### 2. 表格中的“3”是怎么回事

**第一步：找到表格中的“模式编号”**

在 `ARMTmux` 工作表的第 3 行（功能行），`reg_i2s0_4w_tx_mode` 这一列的下方，写着一个数字 **`3`**。

这意味着：**对于“4线 I2S0 TX”这个功能，芯片预定义的硬件复用组编号是 `3`**。

**第二步：理解“配置为 3”**

- **为了选中表格中定义的“4线 I2S0 TX”功能，你必须将这 2 个 bit 整体赋值为 `3`（即二进制 `0b11`，Bit0=1 且 Bit1=1）**。

**具体配置步骤（以在 Uboot/Linux 驱动中操作为例）**：

1. **读取** 地址 `0x103cC5` 的当前值。
2. **修改** 该值的 Bit 0 和 Bit 1 为 `1` 和 `1`（即全部置 1，得到数值 `3`）。注意：需要保留该寄存器其他 bit 位的值不变（使用“读-改-写”操作）。
3. **写回** 地址 `0x103cC5`。

根据这份 `SSD2351X` 的 `ARMTmux` 表，我们可以得知

- 对于物理管脚 `PAD_GPIOD_04`，在 `reg_i2s0_4w_tx_mode` 这一列填写的功能名称是 **`I2S0_TX_BCK`**。
- 而要激活这一列的定义，**必须向 `reg[103cC5]#1~#0` 写入数值 `3`**

也就是说：

> 当 reg_i2s0_4w_tx_mode 设置成对应值时，这个 PAD 输出 I2S0_TX_BCK

### 3.结论

`reg_i2s0_4w_tx_mode` 是一个**寄存器位域（地址 0x103cC5 的 Bit0~1）**。
在该芯片中，将其**配置为数值 `3`（二进制 11）**，就能使被分配到该组的物理引脚（如 PAD_GPIOD_04）切换到 `I2S0_TX_BCK` 功能。

在设备树中看到的 `padmux` 数字（比如 `<6>` 或 `<7>`）必须严格参考 `ARMTmux` 表——**表格中这一列头顶上的那个数字（本例为 `3`），才是驱动最终要写入寄存器的值**。

# 3.当前的任务

现在正在验证 **SSD2351 的 I2S0 是否支持 6-wire TX/RX 同时工作**。建议把检查顺序固定为：

1. **ARMTmux**：确认所有 I2S0 相关 PAD 都已经切换到 I2S 功能，没有被 UART、SPI、LCD 等其它外设占用。
2. **Device Tree**：确认 pinctrl 节点引用了正确的 I2S0 PAD，并且配置了 6-wire 所需的引脚集合。
3. **sstar_bach 驱动**：确认 I2S 控制器寄存器被配置为 6-wire（TX 与 RX 同时启用，而不是仅 TX 或仅 RX）。
4. **示波器验证**：观察 BCLK、LRCK、TX_SDO、RX_SDI 是否都出现在对应的物理引脚上。

**可以把 ARMTmux 理解为"地图"，它告诉你每个 PAD 能去哪些地方；**

**而 Device Tree 则是在运行时选择其中的一条路。**

对于驱动调试来说，只有 PinMux 正确、控制器配置正确，这两个条件同时满足，I2S0 的信号才能真正从你文档中定义的那些物理管脚输出。
